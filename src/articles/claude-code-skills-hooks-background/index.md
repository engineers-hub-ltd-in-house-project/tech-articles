# Claude Code の Skills や Hooks が生まれた背景とは？

## はじめに

Claude Code を触っていると、Skills や Hooks という仕組みの存在に気づきます。ドキュメントを読めば使い方は分かるものの、わたしが気になったのは別のところでした。なぜこの形で存在しているのか、という点です。普通のツールを増やす選択肢もあったはずなのに、Anthropic はわざわざ Skill という「読めるドキュメント」の形と、Hook という「ツール呼び出しに割り込む」形を用意しました。その設計の理由を追いかけたくなったのが、この記事を書き始めた動機です。

手がかりになったのは、Anthropic が 2026 年 4 月に公開した "Seeing like an agent: how we design tools in Claude Code" というブログでした。読んだあと、頭のなかで散らかっていたものがいくつか繋がる感覚がありました。この記事はその感覚を、手を動かしながら自分で確かめた記録です。動作確認は Claude Code 2.1.110、Node 22.16.0 で行いました。作業用のディレクトリは `/tmp/skills-hooks-lab` に切って、そこで実験を閉じ込めています。

## 第1章　Claude は目と手を持たない

最初に共有しておきたい事実があります。Claude は、わたしたちがターミナルで bash を叩くように、直接コマンドを実行しているわけではありません。Claude が見ているのはツールの返り値だけです。人間でたとえるなら、目を閉じたまま助手に「ls して」と頼み、読み上げられた文字列だけを頼りに世界を組み立てている状態に近い。この制約を頭に入れておくと、このあとの話が一気にすっきりします。

小さな確認をしてみましょう。Claude Code には `-p` フラグで一発プロンプトを投げるモードがあって、`--verbose` と `--output-format stream-json` を足すと、内部で呼ばれたツールが JSON で流れてきます。

```bash
cd /tmp/skills-hooks-lab
claude -p "ls で作業ディレクトリのファイル一覧を出力してください" \
  --verbose --output-format stream-json
```

返ってきた出力のうち、Claude の動きが見える箇所だけを抜き出すと、こうなりました。

```json
{"type":"assistant","content":[{"type":"tool_use","name":"Bash",
  "input":{"command":"ls /tmp/skills-hooks-lab"}}]}
{"type":"user","content":[{"type":"tool_result",
  "content":"(Bash completed with no output)"}]}
{"type":"assistant","content":[{"type":"text",
  "text":"作業ディレクトリ /tmp/skills-hooks-lab は空です。"}]}
```

Claude は「ls で一覧を」という日本語を、Bash ツールへの `ls /tmp/skills-hooks-lab` という呼び出しへ翻訳しました。そしてその返り値である空文字列だけを見て、「空です」と答えています。このやり取りの全体が、Claude と外界との接点のすべてです。

ツールが渡されていなければ、Claude は何も見えないし、何もできません。ツールの設計がそのまま、Claude にできることの輪郭になります。

## 第2章　Anthropic が 3 回失敗した話

では、そのツールはどう作ればよいのか。Anthropic のブログには、AskUserQuestion という今では標準ツールになっているものが、過去に三度作り直された経緯が書かれています。同じ「ユーザーに選択肢を提示する」機能を、毎回違う形で実装しては捨てていった話です。この三つを並べると、ツール設計の難しさが立体的に見えてきます。

最初の試みは、既存ツールにパラメーターを足すことでした。たとえば何かを書き込むツールに、ついでに「ユーザーに尋ねる」オプションを付けてしまう。ツールの数が増えないので、モデルが迷う対象も増えない、という発想です。けれど実際にやってみると、一つのツールが二つの責務を抱えた瞬間から、モデルは呼び方に迷うようになったそうです。UNIX 哲学で ls が急にメール送信もし始めたら困る、というのと構造が似ています。境界が曖昧なツールは、モデルにとっても読みづらい相手になります。

二つ目の試みは、プロンプト側で自由形式のフォーマットを指定することでした。「こういう時は JSON で質問文と選択肢を返して」と事前に伝えておき、返ってきたテキストを正規表現でパースする、というやり方です。これも動くことは動きます。ただ、自由度が高いぶんモデルは時々フォーマットから外れます。わずか 1% の揺らぎでも、パーサーが落ちて動かないダイアログが出来上がる。サーバー側で LLM の出力を正規表現だけで掴もうとするとハマる、あのアンチパターンと同じ構造の問題が起きていました。

三つ目、これが正解として残った形でした。AskUserQuestion を独立したツールとして定義し、質問文や選択肢のスキーマをツールの入力として厳密に型付けする。モデルはツールを呼ぶ以上、そのスキーマに沿った JSON しか返せません。構造は自然に保証され、パーサーも不要になりました。加えて、ツールとして独立させたことで Claude 側から見ても「ユーザーに聞きたい」という意図が、そのままツール呼び出しとして表現できるようになりました。以前は、質問したい気持ちをわざわざ別ツールのオプションや自由テキストに包んで出す必要があったわけです。

この三つの失敗は、並べてみるとどれも人間のソフトウェア設計の失敗と同じ構造でした。責務を混ぜると呼ぶ側が迷う、自由形式でやり取りするとパーサーが壊れる、構造化されていない命令は守られにくい。Claude 相手だからといって特別なことは起きていません。わたしたちが普段 API を設計するときの感覚が、そのままツール設計にも効いてくるのだと感じました。

ここまでは設計上の話ですが、ブログの文章で一番刺さったのは次の一文でした。どれだけ設計が美しいツールでも、モデルが積極的に呼ばなければ意味がない。どれだけ整った箱を置いても、モデルが箱を見て「これを使いたい」と思わなければ箱は空のまま残ります。Skills も Hooks も、この「モデルが使いこなせる形」という基準から離れずに設計されています。

## 第3章　小さなツールを自分で作って確かめる

頭で理解するのと、実際に動かしてみるのは別物です。この章では、最小の Skill を自分の手で作ります。

### 3-1　最小の SKILL.md を作る

Claude Code の Skill は、`.claude/skills/<名前>/SKILL.md` というファイルを置くだけで認識されます。試しに、現在時刻を返すだけの hello skill を作ります。

````bash
mkdir -p /tmp/skills-hooks-lab/.claude/skills/hello
cat > /tmp/skills-hooks-lab/.claude/skills/hello/SKILL.md << 'EOF'
---
name: hello
description: テスト用スキル。現在時刻を返す
---

# hello skill

このスキルは Skills 機構の動作確認用です。
呼ばれたら次のコマンドを実行し、現在時刻を日本語で報告してください。

```bash
date +"%Y-%m-%d %H:%M:%S"
```
EOF
````

これだけです。この状態で Claude Code を起動すると、初期化ログに hello が並びます。実際に使わせてみましょう。

```bash
claude -p "hello skill を使って今の時刻を教えてください" \
  --verbose --output-format stream-json \
  --setting-sources "project,local"
```

トレースのなかで大事な部分を抜き出すと、こうなりました。

```json
{"type":"tool_use","name":"Skill","input":{"skill":"hello"}}
{"type":"tool_result","content":"Launching skill: hello"}
{"type":"text","text":"Base directory for this skill: .../hello\n\n# hello skill\n..."}
{"type":"tool_use","name":"Bash","input":{"command":"date +\"%Y-%m-%d %H:%M:%S\""}}
{"type":"tool_result","content":"2026-04-16 09:09:56"}
```

順を追うと、Claude はまず Skill ツールを呼びます。Skill ツール自体は何かを実行するわけではなく、SKILL.md の本文をユーザーメッセージとして差し込むだけの役割です。本文が差し込まれたあと、Claude はそこに書かれた指示どおり Bash ツールで `date` を叩き、最後に日本語で結果を返しました。

ここで気づいてほしいのは、Skill はツールというより、読めるドキュメントに近いということです。新しいツールを増やしたいわけではなく、新しい知識を Claude に渡したいとき、Skill は軽量な選択肢として働きます。

### 3-2　段階的開示を体感する

Skills の面白さが本当に見えてくるのは、入れ子にしたときです。たとえば db という Skill を作り、SKILL.md には概要だけを置いて、詳細は同じディレクトリの queries.md に分離します。

```bash
mkdir -p /tmp/skills-hooks-lab/.claude/skills/db

cat > /tmp/skills-hooks-lab/.claude/skills/db/SKILL.md << 'EOF'
---
name: db
description: プロジェクトのデータベース操作案内。詳細クエリは queries.md を参照
---

# db skill

よく使うクエリの一覧は、同じディレクトリにある queries.md を参照してください。
注意: queries.md を読むのは、ユーザーが具体的なクエリを求めたときだけにしてください。
EOF

cat > /tmp/skills-hooks-lab/.claude/skills/db/queries.md << 'EOF'
# よく使うクエリ一覧

- 直近 7 日の売上合計:
  SELECT SUM(amount) FROM orders WHERE created_at > NOW() - INTERVAL '7 days';

- 失敗ジョブの一覧:
  SELECT id, error FROM jobs WHERE status = 'failed' ORDER BY updated_at DESC LIMIT 20;
EOF
```

この状態で「db skill を使って、直近 7 日の売上合計を出す SQL を教えて」と頼むと、Claude の動き方はこう変わりました。

```json
{"type":"tool_use","name":"Skill","input":{"skill":"db"}}
{"type":"text","text":"# db skill\nよく使うクエリの一覧は queries.md を参照..."}
{"type":"tool_use","name":"Read","input":{"file_path":".../db/queries.md"}}
{"type":"tool_result","content":"# よく使うクエリ一覧\n- 直近 7 日の売上合計: ..."}
{"type":"text","text":"SELECT SUM(amount) FROM orders WHERE created_at > NOW() - INTERVAL '7 days';"}
```

最初に Skill ツールで SKILL.md の概要だけを読み、具体的なクエリが必要になった段階で Read ツールを使って queries.md を取りに行っています。これが段階的開示、プログレッシブ・ディスクロージャーの動作です。知識の全体を一度にコンテキストへ積むのではなく、必要になったぶんだけ順に取ってくる。コードベースのなかで Grep と Read を使いながら Claude が自分で探索していく、あのやり方と本質は同じです。

Skill は、タイトルと短い要約だけが先に積まれて、本文は必要になるまで開かれません。Claude のコンテキストウィンドウを節約しつつ、必要になった瞬間に必要な知識を渡す。この仕組みがあるから、知識量に対してコンテキストが破綻しません。

もう一つ、SKILL.md がただのマークダウンという点も、わたしは設計の勘所だと感じました。スキーマで縛らず、Claude がもともと得意な形式である自然言語で書ける。書き手の人間にとっても、読ませたい相手である Claude にとっても、知識をそのまま置ける形になっています。ツールとしての厳密さと、ドキュメントとしての柔らかさが、SKILL.md と Skill ツールの二段構えでうまく両立していました。

この二段構えに気づいてから、わたしは「これはツール化すべきか、Skill にすべきか」を迷ったとき、頻度と構造で判断するようになりました。毎回同じ入力と出力を取る処理ならツール、状況によって手順が変わる知識なら Skill、という住み分けです。正解というより、今のところの運用指針という位置づけではありますが、迷った時の判断軸が一つ増えるだけでも作業は楽になりました。

## 第4章　なぜ Hooks が必要になったのか

Skills が「知識をどう渡すか」の問題を解くとすれば、Hooks は「タイミングをどう制御するか」の問題を解きます。

わたしが Hooks の必要性に納得したのは、ファイル編集のあとで必ず走らせたい処理がある、というありふれた場面でした。たとえば Write や Edit のあとで lint をかけたい。Claude にそのまま頼むこともできます。「このファイルを直したら eslint を走らせて」と言えば、たいていの場合は走らせてくれます。ただ、たいていというのが曲者で、たまに忘れるし、「走らせました」と言いながら実際には走っていないケースにも遭遇します。Claude が嘘をついているわけではなく、思考の流れのなかで手順が一つ落ちている。人間がチェックリストなしでフライト前点検をするのに似ています。

Hook はこの穴を塞ぐための道具です。`.claude/settings.json` にこう書きました。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"[hook] file edited at $(date +%H:%M:%S)\" >> /tmp/skills-hooks-lab/hook.log"
          }
        ]
      }
    ]
  }
}
```

分かりやすさのため echo でログに追記していますが、実用では `eslint --fix` でも `pytest` でも、好きなコマンドに差し替えられます。この設定を置いた状態で、Claude にファイルを作らせます。

```bash
claude -p "sample.txt というファイルを作って 'hello hooks' と書いてください" \
  --setting-sources "project,local" \
  --permission-mode bypassPermissions
```

実行後、hook.log の中身を確認します。

```text
$ cat /tmp/skills-hooks-lab/hook.log
[hook] file edited at 09:16:41
```

Write ツールが呼ばれたあと、Hook が決定的に発火しました。Claude が覚えていようが忘れていようが関係ありません。ツールが呼ばれた事実に対して、ハーネス側が自動で追加処理を差し込む仕組みだからです。Hook は Claude を信じないための仕組みではなく、Claude が高い確率でやることを 100% の確率でやらせるための仕組みだと、わたしは受け取りました。航空機のチェックリストが優秀なパイロットを疑っているわけではないのと同じで、チェックリストそのものが安全を作っています。

逆に言えば、Hook のない世界には「必ず実行される」と保証できる場所がありません。CLAUDE.md に「Write のあとは lint を実行」と書くこともできますが、あれは祈りに近い。祈りと保証は違います。Hook はその差を埋めるために置かれた機構です。

もちろん、何でもかんでも Hook にすればいいわけでもありません。Hook は強い仕組みなので、乱用すると Claude の作業の流れが細切れになり、むしろ動きが悪くなることがあります。わたしが置いている基準はこうです。忘れたら事故になるものだけを Hook にし、忘れても後で気づけるものは Claude の判断に任せる。lint や型チェックは事故の芽を早めに潰したいので Hook 向き、コミットメッセージの整形のようなものは後で直せるので Claude 任せでいい、という線引きです。

## 第5章　TodoWrite が枷になった話

ブログのもう一つ面白い観察は、Anthropic 自身が TodoWrite というツールを一度外したという話でした。かつて TodoWrite は、Claude に作業計画をリストとして書かせ、順に消化させるための補助輪でした。モデルがまだ全体を見渡せなかった時代には、この補助輪が計画性を担保していた。ところがモデルが賢くなったあと、同じ補助輪が動きを狭める枷に変わっていったそうです。サブエージェントを並行で立ち上げたほうが速い場面でも、TodoWrite にすべて書き出す癖が邪魔をして、Claude はリストを一項目ずつ律儀に消化しようとしていました。結果として Task ツール中心の設計へ寄せ直した、というのがブログの結論でした。

この話は、Skills や Hooks にとっても他人事ではありません。いまわたしたちが書いている CLAUDE.md や、並べている Skill や Hook の設定は、いまのモデルに対しては合っていても、半年後のモデルには過剰になっているかもしれない。モデルが賢くなるほど、ハーネスは削ったほうが動きがよくなる、ということが普通に起こり得ます。

一度立ち止まって、自分の CLAUDE.md や Skill の中身を眺めてみてもいいかもしれません。あれを書いた時、どのモデルのどの振る舞いを心配していたのか。その心配は今のモデルにも残っているのか。残っていなければ、削る選択肢が出てきます。

## おわりに

この記事で確認したことを、最後にまとめます。

- Claude はツールの返り値を通じてしか外界を見ない、という制約が設計のすべての出発点になっています
- ツール設計の難しさは「モデルが積極的に呼べる形」を作れるかどうかに集中していて、Anthropic 自身も AskUserQuestion を三度作り直してきました
- Skills は読めるドキュメントとして、知識を段階的に渡す仕組みです
- Hooks はツール呼び出しのタイミングで決定的に何かを実行するための、保証の仕組みです
- どちらも「ツールを増やさずに能力を拡張する」という発想から生まれています

使い方の文書は公式ドキュメントにしっかりとあります。ただ、背景の設計意図を知ってから手元で動かすと、同じ機能でも見え方がだいぶ変わります。わたしはこの記事を書きながら、Skills と Hooks の位置づけが前より少しだけクリアになりました。読んでくださった方にも、そのクリアさの一部が伝わっていたらうれしいです。
