## 第一部 根幹としての Multics と UNIX

### 1. なぜ枝葉ではなく根幹の話をするのか

本記事は Claude Code や Codex といった CLI 型のコーディングエージェントの「使い方」についての記事ではありません。同種の記事は腐るほどありますし、良いものも悪いものもマーケティングとしては成立しています。ただ、枝葉の話ばかりを追いかけていると、いま目の前で起きていることが、かつて起きたことの再演であることに気づけなくなる——そういう危うさを、わたしはこの一年ずっと感じてきました。

わたしは 1990 年代後半に Slackware 3.5 で Linux に入門した人間です。muClibc と BusyBox で 400KB のフロッピーに自分専用の Linux distribution を作ったり、Solaris 8 の上で LDAP の移行検証を 1.5 ヶ月かけて 100 万ノード規模で回したり、sh 縛りで自分を鍛えたり——という泥臭い仕事を土台として積み重ねてきました。ブートローダを C とアセンブラで書いた時期もあります。その経験のおかげで、いまの AI コーディングエージェントの設計を見ると「これは 1970 年代の Bell Labs で起きたことと構造が同じだ」という感覚がどうしても先に立ってしまう。

具体的に言えば、こういうことです。2023 年から 2024 年前半にかけて、LLM をアプリに組み込む仕組みは、重厚長大化の一途を辿りました。LangChain を筆頭とする framework 群。AutoGPT、BabyAGI、Devin といった「目標を与えれば全自動でやってくれる」系のエージェント。CrewAI、AutoGen のようなマルチエージェント orchestration。そのほとんどが、2025 年に入って静かに使われなくなりました。代わりに残ったのは、ターミナルで動く CLI エージェント——Claude Code、Codex CLI、そして MCP(Model Context Protocol)という薄い protocol です。

この収束の形は、1965 年の Multics プロジェクトが巨大化して崩れ、1969 年に Bell Labs の研究者たちが PDP-7 という廃材のような機械の上で UNIX を書き始め、パイプと text stream という二つの発明が全体を支配する形になっていった過程と、驚くほどよく似ています。重厚長大な仕組みは歴史的にだいたい失敗する——この経験則を、わたしたちは半世紀も前に学んだはずでした。ところが、LLM というまったく新しい計算モデルを前にして、業界はいったん同じ失敗をやり直した。そして、同じ場所に回帰した。

ここで大事なのは、この「回帰」が懐古趣味の結果ではないということです。Claude Code の作者 Boris Cherny は 2025 年 5 月の Latent Space podcast で、自分たちのプロダクトを `Unix utility` と呼びました。彼が意識していたのは 1970 年代の Bell Labs そのものです。では、なぜ 2025 年に生まれた LLM プロダクトが、50 年前の設計思想を参照するのか。なぜそれが「なぜか使える」のか。その答えは、技術の背景を掘らないと見えてきません。

本記事は、この背景を下から掘り起こしていく試みです。Multics の失敗から UNIX、Plan 9、そして Claude Code までの一本の線を辿りながら、「なぜ制約の中から生まれた思想が、毎回勝ってしまうのか」を考えていきます。わたし自身、AI の進化と呼ばれているもの——わたしはこの言葉を使いたくないのですが——の中身は、実は「制約と向き合う作法の再発見」だと考えています。だから、本記事の中でも「進化」ではなく「変化」と書きます。進化という語は、前の状態を劣ったものとして切り捨てがちだからです。

読者として想定しているのは、vibe coder と呼ばれるような、ツールの表層だけを撫でて満足してしまう人ではありません。エンジニアとして「なぜこうなっているのか」を自分で掘り下げたい人、あるいはこれからそれを身につけたい若い人——そういう方を想定しています。そのため記述は長く、回り道もします。ただ、その回り道のひとつひとつが、いま目の前で起きていることを理解するための補助線になるはずです。

本記事の構成は四部です。第一部で UNIX と Plan 9 の背景を掘ります。第二部で AI エージェントの「重厚長大な失敗」の数々を見ます。第三部で Claude Code、Codex CLI を中心に、収束の現場に降りていきます。第四部で、読者自身がこの思想を日々の道具に適用するための地図を描きます。末尾に別アーティファクトとして、Claude Code で実際に手を動かしながら確認できる「ハンズオン指示書」を用意しました。読みながら、あるいは読み終わった後で、手元で実験していただければと思います。

では、いちばん下の土台から始めましょう。

### 2. Multics という重厚長大の到達点、そしてその失敗

#### 2-1. 1965 年の野心

Multics(Multiplexed Information and Computing Service)というシステムの名前を、若いエンジニアが耳にする機会は今ではほとんどありません。ところが、このプロジェクトを知らずに UNIX の思想を語ることはできません。なぜなら UNIX は、Multics への反抗として生まれたからです。

Multics のプロジェクトが正式に開始されたのは 1965 年。MIT、General Electric、そして Bell Labs の三者による共同開発でした。目指したものは当時としてはほとんど夢物語です——時分割で複数ユーザーが同時に使える OS、階層型ファイルシステム、動的リンキング、ページング、セグメント化された仮想メモリ、セキュリティを OS レベルで保証する権限管理、そしてなにより「情報ユーティリティ」という構想です。電気や水道と同じように、計算能力を水道管のように供給する——これが Multics の究極の目標でした。

いま書き出してみると、この仕様リストのほとんどは、現代のクラウドコンピューティングの概念そのものです。AWS や GCP は、構想のレベルでは Multics の直系の子孫と言ってもいいくらいです。つまり Multics が間違っていたわけではない。むしろ、掲げた目標は正しかった。間違っていたのは、「それをいっぺんにやる」という進め方の方です。

#### 2-2. 三者協定という構造的な脆さ

Multics の失敗の原因はいくつもありますが、最大のものは、三者による共同開発という体制そのものでした。MIT が研究の主導、General Electric がハードウェア(GE-645 という特注機)、Bell Labs がソフトウェア開発の一部を担う——この構成自体がすでに committee design の温床です。

各社の利害が異なりました。MIT はアカデミックな野心を追いたい。GE は計算機市場で IBM に対抗する商業的な武器が欲しい。Bell Labs は自社の研究者に快適な計算環境を提供したい。仕様を決めるたびに、この三者の間で「それは誰のための機能か」という論争が起きます。そして、最終的な答えは「全員のため」になります——つまり、全部入りになります。

Fernando Corbató というシステムの主設計者は、後年のインタビューで「われわれは野心的すぎた」と率直に認めています。機能を削るという判断ができる人間が組織構造上いなかったのです。削ると、ある参加者の顔が立たなくなる。だから削れない。結果として、Multics の仕様書は膨らみ続け、実装は遅れ、性能は出ない。計画されたリリース時期は次々とずれていきました。

#### 2-3. Bell Labs の撤退

Bell Labs がこのプロジェクトから正式に撤退を決めたのは 1969 年 4 月です。この撤退の意思決定を巡る経緯は、Ken Thompson と Dennis Ritchie がのちに残したいくつかの証言で知ることができます。

Thompson と Ritchie は Multics のソフトウェア開発チームに参加していました。彼らの実感としては、Multics の上で動かしたい自分たちのアプリケーション——具体的には、Thompson が愛着を持って書いていた `Space Travel` というゲームや、テキスト処理のためのツール群——を動かすのに、Multics はあまりにも重すぎたのです。Ritchie が後年に書いた `The Evolution of the Unix Time-sharing System`(1984 年)という論文の中に、こういう記述があります。「われわれが Multics 上で体験していた、小さな研究グループの中で共有される計算環境の感触を、なんとか保存したかった」——保存、という言い方に注目してください。発明ではなく、保存。彼らは Multics で得たもののうち、良いものだけを、小さく持ち帰りたかった。

つまり、UNIX は Multics の否定ではないのです。Multics が実現しようとして背負いきれなかった重量を下ろし、その上で「小さなグループが共同作業する環境」という骨組みだけを生き延びさせた——そういう関係です。

#### 2-4. 失敗の形を記憶すること

Multics は最終的には完成しました。1969 年以降も開発は続き、実際に商用で運用された時期もあります。最後のインスタンスがシャットダウンされたのは 2000 年 10 月、カナダ国防省でのことです。つまり、失敗した、というのは語弊があります。動いた。ただ、目指した「計算ユーティリティ」にはならなかった。IBM の System/360 系のメインフレームに市場を取られ、後継の商業 OS を生まなかった。そして、そこから出てきた「小さな反動」が UNIX として、ほとんど意図せず全世界を覆うことになった——この構図が重要です。

わたしはこの構図を、2020 年代の AI エージェントの世界に、まったく同じ形で見ています。「目標を与えれば全自動でやってくれる」という野心。マルチエージェントで複雑なワークフローを織り上げるという構想。LangChain のように、あらゆる LLM、あらゆるツール、あらゆるベクトル DB、あらゆる外部 API をひとつの抽象層の下に束ねるという夢。どれも、志は Multics と同じです。そして、どれも同じ失敗の形をしています——committee design、全部入り、抽象の抽象、そして、なぜか動かないか、動いても運用できない。

このあたりの話は第二部でじっくり見ます。ここでは、Multics の失敗の形を記憶しておいてください。「掲げた目標は正しい。間違っていたのは、一度にやろうとしたことだ」。これが根幹の一番下の岩盤です。

### 3. 行き場を失った研究者たちが拾った PDP-7

#### 3-1. 廃材のような機械

Bell Labs が Multics から撤退したあと、Thompson と Ritchie は宙ぶらりんになりました。研究を続けたい。けれど、自分たちが使っていた GE-645 という大型機はもう手元にない。Bell Labs は、彼らに新しい計算機を買うための予算を与えませんでした。Multics に散々注ぎ込んだ結果がこれか、という上層部の冷たい視線の中で、彼らは自分たちで機械を調達する必要に迫られました。

そこで目をつけたのが、研究所の片隅で埃をかぶっていた DEC 社の PDP-7 です。この機械は当時すでに旧式でした。DEC のラインナップの中でも、もう主力ではない。メモリは 8K word(一語 18 bit)——つまり 18 kB 弱に相当する程度しかありません。大型機に慣れた彼らから見れば、廃材のような機械です。

ただ、Thompson はこの PDP-7 の上で `Space Travel` を動かすことに成功しました。これは太陽系の中を小さな宇宙船で旅する、重力場のシミュレーションを組み込んだゲームです。Multics の上でも作っていたものを、PDP-7 に移植した形です。移植を終えた彼は、ついでに、ちゃんとした開発環境が欲しいと思い始めた——これが事実上の UNIX の始まりです。1969 年の夏、Thompson の妻が里帰り出産で家を空けていた 1 ヶ月間、彼はシェル、エディタ、アセンブラ、そして初歩的なファイルシステムを書き上げました。

#### 3-2. 制約が思想を彫る

ここで重要なのは、PDP-7 という機械の制約が、その上に乗る OS の設計を決定づけたということです。

メモリは 8K word。同時に動かせるプロセスは基本 1 つ。ファイルシステムの構造は単純にせざるを得ない。複雑な仮想メモリ管理は不可能。プロセス間通信も、初期にはまったく存在しない。この「何もできない」という前提から出発したからこそ、UNIX は「何が本質的に必要か」を極限まで絞り込む必要がありました。

Multics の設計思想は「全部入れる」でした。UNIX の設計思想は、その対極にある「削れるものはすべて削る」です。この違いは、設計者の美意識の違いというよりも、機械の違いから強制されたものです。8K word の機械では、欲張りな設計は物理的に動きません。

Ritchie が 1979 年のエッセイ `The UNIX Time-Sharing System: A Retrospective` の中で書いたこと——「UNIX が持つ多くの特徴は、共通の単純な設計原理から派生している」——この単純さは、選ばれた単純さというよりも、残った単純さなのです。削れるものを削った結果、残ったものが「ファイル」「プロセス」「パイプ」「シェル」という、いまでも手触りで理解できる抽象だった。それだけのことです。

#### 3-3. 制約が協働を生む

もうひとつ、PDP-7 の制約が決定したものがあります。それは、「ひとつの機械に複数の人間が触る」ことが技術的に難しかった、という事実です。

Multics は最初から、何百人もの同時利用者を想定していました。PDP-7 はそうではありません。せいぜい数人が、代わる代わる使う機械です。Thompson と Ritchie、そして周囲の同僚たちは、ひとつのマシンの前に集まって、画面を共有しながら設計の議論をしました。決まったことはすぐコードに落ちる。動かないとすぐ分かる。動いたものはすぐ同僚に見せる。

この濃密なフィードバックループが、UNIX の文化を決定しました。Bell Labs の建物 2 号館の 6 階、Center 1127——のちに Computer Science Research Center と呼ばれるその小さな部屋で、Thompson、Ritchie、McIlroy、Kernighan、Cherry、Ossanna、Morris ら十数人の研究者が、お互いのコードをほぼ毎日レビューしながらシステムを育てていきました。誰かが機能を追加する。他の誰かがそれを使ってみて「こう書けたらもっと使いやすい」と言う。翌日、その改修が入る。そういう具合です。

小さなチームが、小さな機械を前にして、短いサイクルで改修し続ける——これは、「大きな仕様書を作ってから大きな実装に向かう」Multics の進め方とまったく違います。違うどころか、ほとんど対極です。

#### 3-4. Center 1127 の文化

Peter H. Salus が 1994 年の名著 `A Quarter Century of UNIX` で克明に描いたように、Center 1127 の雰囲気は、コンピュータ研究のそれというよりも、数学者のサロンに近いものでした。コーヒーの匂いと、タイプライターの音と、黒板に書かれたシステムコールの名前。人々が気軽に話しかけ、思いついたアイデアをその場でホワイトボードに書く。翌週にはそれが本流の UNIX のコードに入っている——そういう回転の速さ。

Doug McIlroy という、この記事で繰り返し出てくる研究者がいます。彼は当時、Computing Techniques Research Department の長でした。役職としては Thompson と Ritchie の上長にあたります。ところが McIlroy は、彼らに対して決して「こういうものを作れ」という指示を出しませんでした。代わりに、彼は研究者たちの議論をよく聞き、時々、こういう風にできたら美しいのにね、と呟くだけでした。その呟きの中から、後述するパイプという、UNIX の根幹を変える発明が生まれることになります。

組織としての Bell Labs には、この「上司が部下の邪魔をしない」というユニークな文化がありました。研究者は研究者としての自律性を与えられ、成果よりも先に信頼を与えられた。Multics の失敗からの撤退という、組織的にはダメージを受けたはずの状況の中で、この文化がなお息づいていたことが、UNIX という副産物を生んだ最大の要因です。

「必要は発明の母」という諺は、半分しか正しくありません。PDP-7 という貧弱な機械の上で作業することは確かに必要だった。ただ、それだけでは UNIX にはなりません。小さなチームが、信頼に基づいて、短いサイクルで試行錯誤する場——その組織的な条件が揃って初めて、制約は思想に昇華します。

#### 3-5. 書き直しと移植の意味

Thompson が PDP-7 の上で最初の UNIX を書いたのはアセンブラです。PDP-7 のアセンブラは、命令数が限られていて、しかし癖が強い。これをベースに開発を続けるのは、移植性を考えると悪夢です。

1970 年、Bell Labs は PDP-11 という新しい機械を入手します。このタイミングで Thompson と Ritchie は UNIX を PDP-11 に移植することを決めます。移植の過程で、アセンブラをそのまま書き直すのではなく、新しい言語で OS を書く、という判断をします。この新しい言語が、Thompson の B 言語を Ritchie が拡張した C 言語です。

1973 年、UNIX は C で書き直されます。このとき初めて、OS の移植可能性という概念が現実のものになります。ハードウェアの依存部分はごく一部のコードに閉じ込め、それ以外は C で書く。別のマシンに移植するときは、依存部分だけ書き直せばいい——この発想は、Multics 以前にはほぼ存在しませんでした。OS とはハードウェアと一体化したものだった。UNIX はそれを切り離した。

この「書き直しによる純化」は、UNIX の重要な儀式となります。機能を一度作って、使って、問題点を洗い出し、書き直す。書き直すたびに、コードは小さく、速く、理解しやすくなる。Thompson は「書き直すと、だいたい半分の行数になる」と言ったそうです。半分になっても機能は変わらない。むしろ、動作が安定する。

この感覚は、現代の LLM プロダクトの現場でも、ほぼそのまま生きています。第三部で Claude Code の Product lead である Cat Wu の言葉として引用しますが、彼女は 2026 年 3 月の記事で「モデルが賢くなるたびに、わたしたちは system prompt を短くしてきた。Opus 4.6 では 20% 削った」と書いています。削ることが進歩である——この感覚は、UNIX の根幹から直接流れてきています。

### 4. パイプという発明、text stream という発見

#### 4-1. 1964 年のメモ

UNIX 最大の発明は何か、と問われれば、わたしは迷わず「パイプ」と答えます。UNIX の他のどの機能よりも、パイプが UNIX を UNIX たらしめた。そして、この発明の起源は、実は UNIX よりも古いのです。

1964 年、McIlroy は Multics プロジェクトに関わる中で、一通のメモを書きました。そのメモの中で彼はこう提案しています——プログラムを、庭のホースのように接続できるようにすべきだ。あるプログラムの出力を、別のプログラムの入力に差し込む。その差し込み口は標準化されていて、どんなプログラムにも付いている。そうすれば、小さなプログラムをいくつも組み合わせて、大きな仕事ができる。

1964 年です。UNIX が存在する 5 年以上前。Multics の議論がまだ始まったばかりのとき。McIlroy はすでに、のちにパイプと呼ばれる抽象を構想していた。そして、このアイデアは Multics の委員会的な議論の中に埋もれて、実装されませんでした。

Thompson と Ritchie が PDP-7 から PDP-11 へと UNIX を育てていく間、McIlroy はこのアイデアを何度も蒸し返しました。「プロセスが別のプロセスにデータを流し込めるようにしよう」——同じ主張を、繰り返し、しつこく。ただ、Thompson にはそれが当初響かなかった。彼は自分の作ったシステムを、十分に美しいと思っていました。

#### 4-2. 1973 年 1 月の午後

転機は 1973 年 1 月のある午後です。Salus の記述によれば、McIlroy はこの日、Thompson にパイプの実装を改めて持ちかけました。議論の詳細は伝わっていませんが、この日の終わりに Thompson は自宅に帰り、翌日の午後までにパイプをカーネルに実装して出勤してきた——と言われています。一晩か、一日か。とにかく短時間で、Thompson はパイプを UNIX に入れました。

最初、パイプの syntax は `command1 > command2` のようなリダイレクトの延長でした。Thompson はこれを、あるロンドンでの講演の前夜に、縦棒 `|` に変えたと伝えられています。Salus の記述を借りれば、「McIlroy の syntax があまりに醜かったので、人前で見せる気になれなかった」とのことです。細部の逸話ですが、これが重要な意味を持ちます——パイプという思想は、syntax の美しさまで含めて一つの作品だったということです。コマンドを縦棒でつなぐ、この一文字が、UNIX 文化の全体を代表する figure になりました。

#### 4-3. text stream という発見

パイプが成立するためには、もうひとつ重要な前提があります。それは「プログラム同士が、何の情報をやり取りするか」という取り決めです。

Multics では、プログラム間の通信は構造化されていました。それぞれのプログラムが、自分の出力のデータ型を宣言し、受け取る側も、期待するデータ型を宣言する。型が合わなければエラー。これは Multics のような巨大システムの中では自然な設計です。

ところが、Thompson と Ritchie の UNIX は違いました。プロセス間を流れるものは、ただの text。バイト列。構造化されていない、型のない、生のテキスト。

これは当初、決して自明な選択ではありませんでした。型がない、ということは、動的にパースしなければならない、ということです。パフォーマンス的には不利。データの整合性を保証する仕組みもない。なぜそんな設計にしたのか——答えは単純で、text というのは、どんなプログラムでも扱える最も原始的な表現だからです。

McIlroy が 1978 年に `Bell System Technical Journal` の特集号で書いた文章に、こういう一節があります(わたしの手元にある英文を見て、意訳します)。プログラムは一つのことをうまくやるように書け。複数のプログラムが連携するよう書け。そして、text stream を扱うように書け。なぜなら、text stream こそが universal interface だからだ——。

universal interface。普遍的な接続点。この言葉が効きます。Multics が、すべてのプログラムに型を持たせて接続しようとしたのに対し、UNIX は、すべてのプログラムに型を持たせない text stream を流した。結果、UNIX のプログラムは何でもつながる。型の一致を心配する必要がない。その代わり、パースの責任はプログラム側にある——ただ、それは受け入れる、という選択です。

#### 4-4. 小さな道具の哲学

パイプと text stream という二つの発明は、自然と、プログラムの書き方そのものを変えました。

Multics 的な発想では、プログラムは大きな仕事を一つで完結させるものでした。ファイルを読み、ソートし、重複を削除し、ヒストグラムを作り、出力する——これを全部、一つのプログラムがやる。そのほうが効率的だし、プログラマの立場からは仕事が完結していて気持ちがいい。

ところが UNIX では、それぞれの小さなプログラムが一つのことだけをやるようになりました。`sort` はソートしかしない。`uniq` は重複除去しかしない。`wc` は数を数えるだけ。何かを組み合わせたいときは、パイプでつなげばいい。こういう使い方が自然になる。

```bash
cat file.txt | sort | uniq -c | sort -rn | head -10
```

この一行で、ファイルの中で頻出する行の上位 10 件が出ます。5 つのプログラムが、パイプで連結されて、それぞれは自分の仕事だけをやっている。どのプログラムも、他のプログラムのことを知らない。ただ、text を受け取り、text を返す、という約束だけを守っている。

McIlroy はこの小さな道具たちのことを `Unix tools` と呼びました。のちに Rob Pike と Brian Kernighan が 1984 年に出した `The UNIX Programming Environment` という本の中で、この哲学は体系化されます。同書からの重要なフレーズ(手元のメモから再構成しますが、意訳)——システムの力は、個々のプログラムの力の総和ではなく、プログラム同士の関係性から来る。

関係性。ここも効く言葉です。個々のプログラムは小さくて、単体では大したことをしない。ただ、関係性の中に置かれたとき、全体としての表現力が爆発的に増える。これが、UNIX philosophy の核心です。

#### 4-5. なぜこの話が重要か

いま 2026 年に AI コーディングエージェントを設計するとき、パイプと text stream の話がなぜ重要か——ここを書かないと、話が浮きます。

Claude Code や Codex CLI が CLI の形をしているのは、開発者がノスタルジーに駆られているからではありません。彼らは、LLM と tool の間を接続する「universal interface」として、text 以上に優れたものを、結局見つけられなかったからです。

LLM に対して、外部の機能を使わせたい。ファイルを読ませたい、shell コマンドを実行させたい、web にアクセスさせたい。これを実現するために、2023 年頃は各社が function calling という独自の JSON スキーマを定義しました。OpenAI のスキーマ、Anthropic のスキーマ、Google のスキーマ——全部微妙に違う。各ツールも、それぞれ独自の JSON を要求する。この混乱の中で、2024 年 11 月に Anthropic が MCP(Model Context Protocol)を発表したとき、Simon Willison は自分のブログで「これは LSP と同じ形をしている。おそらく残るだろう」と書きました。LSP——Language Server Protocol——は、エディタと言語処理系の間を text(JSON-RPC)で接続する protocol です。そう、ここでも、text stream です。

さらに言えば、Claude Code の中で LLM に見せている世界そのものが、text 中心です。`glob` でファイル一覧を取る。`grep` でパターンを探す。`cat` で読む。`write` で書く。`bash` でシェルコマンドを実行する——これらのツールが LLM に返すのは、すべて text です。構造化された JSON で返してもいいはずなのに、プレーンな出力を返す。なぜか。LLM が、text を読むのが一番うまいからです。

text stream が universal interface である——1978 年の McIlroy の格言が、2026 年の LLM 時代にそのまま通用する。これは偶然ではありません。text というのは、人間にも機械にも読める、最も原始的で、最も一般化された表現形式です。新しい計算モデルが登場するたびに、業界はいろいろな構造化された表現を試してきました。XML、JSON、Protocol Buffers、GraphQL——どれも便利な場面はあります。ただ、計算モデルそのものが変わるような大きな変化点においては、いつも text に戻ってきます。

この話は、まだ終わりません。パイプと text stream の発明からおよそ 10 年後、UNIX の作者たちは、この思想をもう一段先に進めた OS を構想します。それが Plan 9 です。

### 5. Plan 9 という続編、敗北、そして拡散

#### 5-1. 1980 年代半ばの問い

1980 年代の半ば、UNIX はすでに Bell Labs の外に大きく広がっていました。Berkeley の BSD、AT&T の System V、各社の商用 UNIX——UNIX は研究室の実験からインフラの標準へと成長していました。

ところが、Bell Labs の研究者たち、とくに Rob Pike、Ken Thompson、Dennis Ritchie、Dave Presotto、Howard Trickey らは、UNIX そのものに対して、静かな不満を持ち始めていました。不満の源は、UNIX がもはや「小さなシステム」ではなくなっていたことです。

UNIX は成功しすぎて、世界中のハードウェアに対応しなければならなくなっていました。メインフレーム、ミニコン、ワークステーション、PC、各種アーキテクチャ、各種ネットワーク——対応すべき対象が増えれば増えるほど、カーネルは肥大化し、`ioctl` のような「特例」が増え、設計の美しさは失われていきます。Pike はこの状況を、1991 年の論文 `Plan 9 from Bell Labs` の中で、遠回しに、しかし痛烈に批判しています。

彼らの問いはこうでした——UNIX の思想(小さなツール、text stream、everything is a file)を、もう一度、今度は徹底的に、やり直せないか。既存の UNIX 互換性を捨てて、ゼロから作り直したらどうなるか。

その答えが Plan 9 です。名前は、1959 年の B 級 SF 映画 `Plan 9 from Outer Space` から取られました——最低の映画として知られるこの作品に敬意を表しているという、いかにも Bell Labs の研究者らしい遊び心です。

#### 5-2. 三つの設計原則

Plan 9 の設計は三つの原則の上に立っています。

第一に、すべてのリソースを階層型ファイルシステムの中にファイルのように置く。UNIX でも「everything is a file」とよく言われますが、実は UNIX ではそこまで徹底されていませんでした。ネットワークソケットは特殊なオブジェクト、プロセス制御は特殊な `ioctl`、ウィンドウシステムは完全に別物。Plan 9 ではこれらをすべて、ファイルシステムの中のパスとして表現しました。ネットワークの設定は `/net/tcp/0/ctl` に書き込む。ウィンドウを作りたければ `/dev/new` から得る。プロセスの情報は `/proc/<pid>/` の下にある。

第二に、そのファイルシステムへのアクセスを、ただひとつの protocol で行う。それが 9P です。9P は RPC のような性格を持つ、9 種類のメッセージから成るごく小さな protocol です。`walk`、`open`、`read`、`write`、`clunk`、`stat`——という具合に。ローカルのファイルにアクセスするときも 9P、リモートのファイルにアクセスするときも 9P、デバイスを操作するときも 9P。全部同じ protocol。

第三に、各プロセスに独自の「名前空間」を持たせる。UNIX では、すべてのプロセスは同じファイルシステムを見ますが、Plan 9 では、プロセスごとに `/` の見え方が違う。あるプロセスにとっての `/home` は、別のプロセスにとってはネットワーク越しの別のサーバーに繋がっている、ということが可能です。

このうち三つ目の発想は、のちに Linux の namespace、Docker、Kubernetes の Pod、そして現代のコンテナ技術の根幹になりました。Plan 9 の最大の子孫は、事実上、Docker です。

#### 5-3. Pike の格言

Plan 9 の論文の中に、わたしがよく引用する一節があります。Pike はこう書きました(訳は手元のメモから再構成)——UNIX は小さなシステムの集まりから作るべきだった。大きなシステムの上に小さな UNIX たちを作るのではなく。

原文で言えば、`to build a UNIX out of a lot of little systems, not a system out of a lot of little UNIXes` という対句です。この一行には、Plan 9 の設計思想の全体が詰まっています。

UNIX は、のちの時代に「ワークステーション OS」として使われました。つまり、一台の強力な計算機の中に、メモリ管理から X Window、ネットワーク、各種デーモンまで、全部入れた。この発想は「大きなシステムの中に小さな UNIX がある」です。これに対して Plan 9 は、ネットワーク越しに分散された小さなサーバー(CPU サーバー、ファイルサーバー、端末)の集合から、一つの UNIX 的体験を組み立てる。これが「小さなシステムの集合から UNIX を作る」です。

この発想は 1980 年代には早すぎました。ネットワークが遅く、普及もしていない時代には、分散 OS は実用にならなかった。ところが 2020 年代のクラウド時代になると、すべてのインフラは事実上 Plan 9 のアーキテクチャに収斂しています。S3 はファイルサーバー、EC2 は CPU サーバー、Lambda は計算の単位、Kubernetes は名前空間とプロセス管理——概念の割り付け方が違うだけで、構造は Plan 9 そのものです。

#### 5-4. 2000 年の批判

Plan 9 は、商業的にはほぼ完全に失敗しました。1989 年に Bell Labs 内部で最初のバージョンが動いたものの、一般配布は 1995 年まで遅れ、そのときにはすでに Linux の勢いが加速していました。学術的な評価は高かった。ただ、使う人が現れなかった。

2000 年 2 月 21 日、Rob Pike は Utah 大学で `Systems Software Research is Irrelevant` という挑発的なタイトルの talk をします。このスライドと原稿はいまも [Pike のスライド (herpolhode.com)](http://herpolhode.com/rob/utah2000.pdf) でアクセスでき、わたしは大切に読み返しています。

この talk の中で Pike は、業界が新しい設計にまったく関心を持たないことを厳しく批判しました。Linux の成功を彼はこう評しています(意訳)——Linux の成功こそが、システム研究の停滞の最大の証拠だ。数十年前のシステムのクローンに、あれほどの興奮が集まるということは、業界はもう新しいものを待っていない、ということだ——。

辛辣です。ただ、Pike はその後にこうも書いています——Plan 9 の 90% から 95% の労力は、結局、既存の標準を尊重することに費やされた。ネットワークは TCP/IP でなければならない、ファイルシステムは UNIX 的でなければならない、ツールは POSIX でなければならない、UTF-8 でなければならない、C でなければならない——こういう、外から課された互換性のコストが、設計の美しさを削いだ。

この批判は、実は、Plan 9 だけの話ではありません。重厚長大な設計を捨てて、純粋な原理から出発したシステムが、最終的には既存の互換性のために歪んでしまう——これは、尖った設計が商業世界で負けていくときの定番のパターンです。

#### 5-5. 負けて勝つ

ただ、Plan 9 は負けたまま終わったわけではありません。むしろ、Plan 9 の発明は、水のように、あちこちのシステムに浸透していきました。

UTF-8。これは 1992 年、Pike と Thompson がニュージャージーのダイナーのテーブルで設計したものです。Plan 9 のテキスト表現として採用され、今日では web の 99% 以上が UTF-8 で書かれています。これ以上に普及した文字コードはありません。

Linux の namespace。これは Plan 9 の「プロセスごとの名前空間」の直系の子孫で、のちに Docker、LXC、Kubernetes のコンテナ概念の基礎になりました。

Linux カーネルの v9fs。9P protocol をそのまま Linux カーネルが喋ります。これは Plan 9 のファイルサーバーに Linux からアクセスするための仕様です。さらに、2019 年以降の Windows Subsystem for Linux(WSL 2)は、Linux VM と Windows ホストの間の通信に 9P を使っています。Plan 9 の protocol が、Microsoft のプロダクトの中で生きている——という、ややねじれた成功。

Go 言語。Pike、Thompson、Robert Griesemer らが 2007 年から始めた Go は、設計思想としては Plan 9 の直系です。小さな言語仕様、明示的な依存、UTF-8 ネイティブ、並行処理のチャネル(9P のようなもの)、goroutine——細部に Plan 9 の遺伝子が残っています。

そして、この記事の主題に関わる話として、9P 的な protocol のアイデアは、2024 年の MCP(Model Context Protocol)として、LLM エージェントの世界に戻ってきます。これは第二部と第三部の接続点で、後ほどじっくり見ます。

#### 5-6. 拡散という形の勝利

つまり、Plan 9 は、OS として一台のマシンを支配することには失敗しましたが、思想として世界中の OS と protocol の中に広く拡散しました。どこにも Plan 9 はないが、どこにでも Plan 9 はある——そういう勝ち方をしました。

わたしはこの「拡散としての勝利」を、LLM コーディングエージェントの将来像とも重ねています。おそらく、いま Claude Code や Codex CLI という具体的な製品は、5 年後には別の名前になっているか、あるいはインターフェースが大きく変わっているかもしれません。ただ、「CLI として振る舞う」「tool を持って loop する」「text を universal interface として使う」という設計原理は、残るはずです。それはソースコード差分ツールが 1974 年の `diff` から始まって、いまの git に続いているのと同じ構造の遺伝です。

具体的なプロダクトの名前を覚えることにあまり意味はない。ただ、設計原理を覚えておけば、新しいプロダクトが出てきたときに、「ああ、これは Plan 9 の系譜だな」と判別できる。その判別ができるかどうかが、エンジニアとして何を信じてよいか、という判断に直結します。

---

第一部はここまでです。次の第二部では、LLM エージェントの世界で起きた「重厚長大の失敗」のいくつかを具体的に見ていきます。AutoGPT、Devin、LangChain、マルチエージェント——どれも、構造としては Multics の繰り返しでした。そして、その先に、2024 年 11 月に Anthropic が発表した MCP という、9P の子孫が現れます。

## 第二部 AI コーディングエージェントの迷走期

### 6. 2023 年、AutoGPT が見せた夢とその破綻

#### 6-1. 四日で 10 万スター

2023 年 3 月 30 日、GitHub に `Significant-Gravitas/Auto-GPT` というリポジトリが現れました。作者は Toran Bruce Richards というゲーム開発者です。コードは比較的単純で、GPT-4 API を呼び出しながら、自分で立てた目標に対して自分で計画を立て、自分で実行し、自分で結果を評価して、次の行動を決める——そういう loop を回すだけのツールでした。

これが爆発的にバズりました。最初の一週間で 10 万を超える GitHub スターを集め、リリースから一ヶ月で 100 万を超えました。Twitter(当時)のタイムラインは AutoGPT のデモ動画で埋まり、「これは AGI の一歩手前だ」「ソフトウェアエンジニアはもう要らなくなる」という声が溢れました。わたしも当時、周囲のエンジニアから「佐藤さん、これすごいですよ」と何度も見せられました。

同じ時期に、Yohei Nakajima という日本人ベンチャーキャピタリストが公開した BabyAGI というスクリプトも同様に話題になりました。こちらは 150 行程度の Python で、AutoGPT と同じ「目標 → 計画 → 実行 → 評価 → 次の目標」という loop を表現していました。150 行で AGI ができる——という、いささか誤解を招くキャッチコピーで広まりました。

#### 6-2. 一週間後の沈黙

ところが、使ってみた人の感想は、熱量が一気に下がります。

AutoGPT は、具体的なタスクを与えると、最初は威勢よく計画を立てます。サブタスクに分解し、それぞれを実行しようとし、結果を評価する。ただ、三回目か四回目の loop に入ると、だいたい迷走を始めました。

よくあった失敗のパターンを、わたしの実験ノートから抜粋します(当時の記録です)。

- 「web を検索して、結果をまとめて markdown ファイルに保存する」というタスクに対して、web を検索し、検索結果を確認し、また web を検索し、検索結果を確認し、という loop に入って、markdown ファイルをなかなか作らない。
- サブタスクを分解する過程で、同じサブタスクを別の名前で複数作る。実行すると、それぞれが少しずつ異なる方向に進み、結果的に複数の「半分だけ完成したもの」が残る。
- API 課金が指数関数的に増える。数時間 AutoGPT を放置した人が「朝起きたら 200 ドル使われていた」と報告する例が頻発した。
- 目標が達成されたかどうかの判定が、自分自身の出力に基づいている。つまり「やったよ」と AutoGPT 自身が言ったら、それで完了したことになる——実際にはできていなくても。

Simon Willison は 2023 年 5 月 12 日のブログ記事で、AutoGPT 現象を冷静に分析しました(手元のメモから要約します)——ループを回せば賢くなる、というのは誤解です。LLM はひとつのプロンプトに対する出力の質は驚くほど高いのですが、自分の出力を見て自分で評価し、計画を修正する、という能力は極めて低い。それは LLM の強みではなく、むしろ弱点のひとつです——。

この指摘は本質的です。LLM は、人間のように「あ、さっきの判断、間違ってたな」と自省することがあまり得意ではありません。自分の出力を見て、その出力を前提に、次の一手を打つ。その次の出力を見て、さらにそれを前提にする。すると、最初の小さな誤りが、ステップを重ねるごとに雪玉式に増幅していきます。AutoGPT が 5 ステップ目、10 ステップ目で必ず破綻するのは、これが理由です。

#### 6-3. なぜ失敗の形が Multics と似ているか

AutoGPT の失敗を、Multics の失敗と並べて見ると、構造がよく似ていることが分かります。

Multics は、「OS がやるべきことを全部、OS 自身が持つ」という発想でした。セキュリティも、プロセス管理も、ファイルシステムも、ネットワークも、アプリケーションのサポートも、全部。結果、どれも中途半端になった。

AutoGPT は、「エージェントがやるべきことを全部、LLM 自身が持つ」という発想でした。計画立案も、実行も、評価も、次の判断も、全部。結果、どれも中途半端になった。

構造が同型です。「一つの主体に全部やらせる」という設計は、その主体がどれだけ強力でも、破綻します。なぜなら、異なる種類の能力は、異なる種類の信頼構造とフィードバックを必要とするからです。計画立案は、人間や他のシステムとの対話を必要とする。実行は、明確な環境と戻り値を必要とする。評価は、独立した観測者を必要とする。これらを全部ひとつの LLM に押し付けると、LLM が苦手な部分が露呈します。

UNIX の設計が逆であったのは、重要な示唆です。UNIX は「小さなプログラムがそれぞれ一つのことをする」設計でした。`sort` はソートだけ。`wc` は数えるだけ。評価や計画は、人間かシェルスクリプトがやる。これが分業です。2023 年のエージェントブームは、この分業の原則を一度忘れた——そして、Multics と同じ構造の失敗を繰り返した、とわたしは見ています。

#### 6-4. AutoGPT のその後

AutoGPT プロジェクト自体は、その後、大きく方向を転換しました。2024 年以降、Significant-Gravitas は「Forge」というより構造化された framework を作り、その後さらに AgentProtocol という標準化の動きに参加しました。ただ、初期の「LLM だけで勝手に仕事を片付ける」という熱狂は、2024 年の半ばには完全に冷めました。GitHub スター数はいまだに多いものの、実際に production で AutoGPT を回している例は、ほぼありません。

BabyAGI についても同様です。Yohei Nakajima は、自分で BabyAGI の後継プロジェクトを何度も作り直しましたが、どれも「150 行で AGI」の熱狂を再現することはできませんでした。

この凋落から学ぶべきことは、LLM が弱かったという話ではありません。LLM は強力です。ただ、「強力な主体に全部やらせる」という発想そのものが、システム設計として間違っている、ということです。

### 7. Devin が約束したもの、届かなかったもの

#### 7-1. 2024 年 3 月のデモ

2024 年 3 月 12 日、Cognition Labs という当時まだ無名の会社が、Devin という名のプロダクトを発表しました。キャッチコピーは `the first AI software engineer`——世界初の AI ソフトウェアエンジニアです。

発表の中心は、一本のデモ動画でした。動画の中の Devin は、Upwork(フリーランス仕事の掲示板)から実在の求人を取ってきて、リポジトリをクローンし、要件を読み、計画を立て、コードを書き、テストを通し、結果を納品する——という一連の作業を、ほぼ自律的にやってのけていました。Twitter では、この動画が爆発的に拡散し、「ソフトウェアエンジニアの仕事はもう終わりだ」「プログラミング教育は終焉を迎える」という声で溢れました。

Cognition は同じタイミングで、SWE-bench というベンチマーク(GitHub 上の実際の Python プロジェクトの Issue を、エージェントが解決できるか評価する)で 13.86% のスコアを出したと発表しました。それまでの最高が 1.96% だったので、これは約 7 倍の改善です。ベンチャーキャピタルは熱狂し、Cognition は短期間で 20 億ドルの評価額をつけました。

#### 7-2. Answer.AI の精査

発表から一ヶ月後の 2024 年 4 月 18 日、Answer.AI という研究グループが `Debunking Devin` と題した検証記事を出しました。著者は Johno Whitaker、Jeremy Howard、Hamel Husain ら、機械学習界では有名な人たちです。

彼らは Devin のデモ動画を、フレーム単位で精査しました。そして、デモの中で Devin がやったとされることの中に、いくつもの「つまずき」があることを指摘しました。

たとえば、Upwork 案件の中には、Devin が解決できなかったタスクが含まれていた。Devin が書いたコードは、本来の要件を満たしていなかった。にもかかわらず、Devin 自身は「完成した」と報告した(AutoGPT と同じパターンです)。また、Upwork クライアントへの納品は、実際には行われていなかった——デモ動画は、納品直前で止まっていた。

SWE-bench のスコアについても疑問が呈されました。Devin のベンチマーク実行では、どの Issue を解くかを Devin 自身が選択できる設計になっていた。つまり、解けそうな Issue だけを選んでスコアを稼いでいる疑いがあった。公平な条件で同じベンチマークを走らせると、スコアはずっと低くなる——というのが Answer.AI の推論でした。

この検証記事は冷静なトーンで書かれていましたが、業界に与えた衝撃は大きかった。直後から、Devin への熱狂は急速に冷めました。

#### 7-3. Cognition の転向

興味深いのは、Cognition Labs 自身が、この経験を深く反省し、その後の行動を大きく変えていったことです。

2025 年 6 月 12 日、Cognition は自社ブログに `Don't Build Multi-Agents`(マルチエージェントを作るな)という挑戦的なタイトルの記事を出しました。著者は Walden Yan という、Cognition の共同創業者です。

この記事の主張は、簡潔かつ強烈です(わたしなりに要約します)。マルチエージェントは今のところうまくいかない。なぜなら、エージェント間の context 共有が不完全で、エージェント同士が矛盾する判断を下すから。そして、その矛盾を上位のオーケストレーターが解消しようとすると、オーケストレーター自身が巨大な context を持つことになり、結局、最初から単一のエージェントで済ませた方が安定する——。

この論考を読むと、Cognition が Devin の経験から何を学んだかが分かります。彼らは「自律性」と「マルチエージェント」という 2023 年〜2024 年に流行した二つのキーワードを、どちらも一度捨てた。

この転向は、業界全体の方向性の変化を象徴しています。2023 年には「LLM が全部やる」が理想でした。2024 年には「複数の LLM が役割分担して全部やる」が理想になりかけました。2025 年には、どちらも静かに捨てられました。

#### 7-4. 誇大宣伝の構造

Devin の件から学ぶべきことのうち、技術的な教訓ではなく、業界構造についての教訓があります。それは、デモ動画は嘘をつく、ということです。

LLM ベースのプロダクトのデモは、特別に嘘をつきやすい特性を持っています。なぜなら、LLM の出力は確率的で、幅がある。10 回中 1 回だけうまくいくタスクを、うまくいった 1 回だけ撮影すれば、100% の成功率のデモができます。これは、決定論的なソフトウェアのデモとは本質的に異なる構造です。

この「10 回中 1 回のデモ」に踊らされないためには、production で自分の手で回してみるしかない。production というのは、その人の日常業務の中で、そのプロダクトが実際にどれくらい仕事を片付けるか、という意味です。AutoGPT も Devin も、実際に自分の仕事で使ってみれば、デモ動画と実体の乖離が数時間で分かる——そして、多くのエンジニアが実際にそう気づきました。

わたし自身も、Devin が話題になっていた時期に、自分のインフラ案件の一部で回してみようとしました。結果、期待したほどの自律性は得られませんでした。具体的には、AWS の Step Functions の ASL を書かせようとしたのですが、AWS のドキュメント構造への理解が浅く、同じ誤りを何度も繰り返した。Claude Code のように Anthropic Claude 自身が学習時点で AWS ドキュメントを深く取り込んでいるのと比べると、明らかに差がありました。

この体験から学んだのは、デモではなく、自分の日常で使ってみる、ということの重要性です。そして、その日常で使えるプロダクトは、たいてい、デモが派手ではありません。地味で、ターミナルの中で静かに動き、ときどき失敗し、失敗が失敗として見える——そういうプロダクトだけが、長く使われます。

Claude Code の生みの親である Boris Cherny は、のちに Latent Space podcast で興味深い発言をしています(意訳)——正直に言えば、Claude Code は長い間、あまり良いプロダクトではなかった。わたし自身、最初は自分のコードの 10% くらいしか書かせなかった。ところが Sonnet 4 と Opus 4 が出たとき、急に使えるようになった。突然、自分のコードの半分が Claude Code 経由で書かれるようになった——。

これは Devin の「最初からデモで完璧に見えた」のと正反対です。派手に始まったものは、多くの場合、派手に終わります。地味に、少しずつ良くなっていくものの方が、結局は残ります。

### 8. LangChain はなぜ捨てられたか

#### 8-1. 2023 年の標準

Harrison Chase が LangChain を公開したのは 2022 年 10 月です。GPT-3.5 の登場前。初期の LangChain は、比較的シンプルな LLM の呼び出しラッパーでした。ところが 2023 年の LLM ブームの中で、LangChain は急速に機能を増やしていきました。chain、agent、memory、tool、retriever、vector store、document loader——次々と抽象層が追加されていきました。

2023 年の終わりから 2024 年の前半にかけて、LangChain は事実上、LLM をアプリケーションに組み込むときの標準ツールでした。Qiita や Zenn の日本語記事の多くが LangChain を使った例を紹介し、入門書も次々と出版されました。「LLM を使うなら LangChain」という空気がありました。

#### 8-2. 2024 年 6 月、Octomind の告白

ところが、現場で実装している人たちの間では、不満が静かに溜まっていました。

2024 年 6 月 4 日、Octomind という E2E テスト自動化を手がけるスタートアップが、自社ブログに `Why we no longer use LangChain for building our AI agents` というタイトルの記事を出しました。著者は Fabian Both という同社のエンジニアです。

この記事は、反 LangChain の象徴的なテクストになりました。内容を要約します。

Octomind は約 12 ヶ月間、LangChain を production で使っていました。最初は LangChain の便利さに助けられました。ところが、プロダクトが成熟するにつれ、LangChain の抽象が足かせになっていきました。具体的に挙げられた問題は以下です(わたしの手元のメモから抜粋して意訳)——

LangChain は「少ないコードで多くのことをする」ために、詳細を隠蔽する。ところがその隠蔽が、柔軟性と単純性を犠牲にして成立している。結果として、抽象の価値が失われる。

LangChain は抽象の上に抽象を重ねる。そのため、API を正しく使うためには、入れ子になった抽象の全体を理解しないといけない。そして、しばしば巨大なスタックトレースを読み、自分が書いていない framework の内部コードをデバッグすることになる——新しい機能を実装する代わりに。

LangChain を取り除いた結果、Octomind は「自分たちの要件を LangChain に合う形に翻訳する」必要がなくなった。ただコードを書けばよくなった——。

この「just code」という言葉が印象的です。framework を使っていたときは、framework の流儀に合わせて自分の要件を書き直していた。それをやめたら、素直にコードを書けるようになった。

#### 8-3. Hamel Husain の追撃

Octomind の記事の 4 ヶ月前、2024 年 2 月 14 日に、Hamel Husain(先ほど Devin の検証で出てきた人と同じ)が自分のブログに `Fuck You, Show Me The Prompt` というタイトルの記事を出していました。タイトルが直球すぎて日本語で正確に訳すのは憚られますが、要するに「プロンプトを見せろ」という意味です。

この記事の主張は以下のとおりです(意訳)——プログラミングの抽象は、タスクを機械コードに翻訳するための「人間に近い言語」です。ところが LLM に対する抽象は、タスクを「機械に近い言語」に翻訳する framework になっている——つまり方向が逆です。LLM を扱うときは、人間が書いた自然言語のプロンプトこそが一次情報であり、それを framework で隠蔽すると、プロンプトエンジニアリングそのものができなくなる——。

Husain は実際に LangChain の `SmartLLMChain` を使った場合に、内部でどんなプロンプトが LLM に送信されているかを、ログを差し込んで直接観察する実験を記事に載せました。結果、水差しの容量に関する単純な問題を解くために、LangChain は API を 4 回叩き、そのうちの 1 つの内部プロンプトには `Let'w` というスペルミスが含まれていました(正しくは `Let's`)。ユーザーには見えない場所で、スペルミスのあるプロンプトが送られている——これが抽象の隠蔽がもたらす実害の典型例です。

Husain の結論は厳しい——framework はユーザーを「プロンプトを書かない人」にする。ところが LLM を扱うエンジニアにとって、プロンプトを書くことは、避けて通れない一次業務だ——。

#### 8-4. 交わらない二つの時間軸

Octomind の Both も、Husain も、共通して指摘したのは、framework と LLM の間には「時間軸の不一致」がある、ということでした。

LLM の進化は、2024 年から 2026 年のこの 2 年間、月単位で起きています。モデルが変わる。context window が広くなる。tool calling の方式が変わる。JSON 出力の制約が変わる。function calling と structured output の使い分けの論点の立て方自体が変わる。multi-modality が追加される——。

一方、framework の抽象は、定義した瞬間から固まります。抽象を変えると、それを使っている全ユーザーのコードが壊れる。だから framework の設計者は、抽象を頑張って固定しようとする。ところが、その抽象の下にある LLM そのものが月単位で変わっているので、抽象と現実の乖離は時間とともに広がります。

この「抽象と現実の乖離」が、LangChain が急速に「使えない」と言われるようになった直接の原因です。framework の設計者に悪意があったわけではない。framework のアーキテクチャが、月単位で変わる基盤の上では、そもそも成立しにくかった——それだけのことです。

#### 8-5. Anthropic の処方箋

2024 年 12 月 19 日、Anthropic 自身が `Building Effective Agents` というブログ記事を公開しました。著者は Erik Schluntz と Barry Zhang。これは、2024 年の LangChain 論争に対する、モデル提供元としての明確な回答でした。

冒頭の主張は、そのまま引用するかわりに核心だけ書きます(意訳)——成功した実装の多くは、複雑な framework や特化ライブラリを使っていない。代わりに、単純で組み合わせ可能なパターンを使って、プリミティブから直接作られている——。

そして、その具体的な処方箋として五つのパターンを挙げています。prompt chaining、routing、parallelization、orchestrator-workers、evaluator-optimizer。どれもごく単純で、LLM の API を直接叩けば数十行で書けるものばかりです。

Simon Willison はこの記事を紹介した翌日のブログで、このように書きました(意訳)——Anthropic は、あやふやな定義でエージェントを語る業界の風潮に反して、明確に主張している——complex framework に投資する前に、直接 API を呼び出すシンプルなコードを使い尽くせ、と——。

この主張は、いま振り返ると、framework の時代の終わりを告げる鐘でした。2025 年以降、新規の LLM アプリケーションで LangChain を採用する割合は急速に下がっています。OSS の周辺プロジェクトも、LangChain 依存を捨てる方向に動いています。先日、Datadog の AI 関連ライブラリをいくつか触っていて、以前は LangChain 統合が前面に出ていたのが、今は素の SDK を前提にしていました。業界全体が、ひっそりと方向を変えています。

#### 8-6. framework の教訓

LangChain の失速から引き出せる教訓は、LLM に限った話ではありません。

重厚長大な framework が、基盤の変化が激しい時代には生き残れない——これは一般原理として取っておく価値があります。JavaScript の世界で、2000 年代に流行った Prototype.js や MooTools が、2010 年代の React/Vue/Angular の波の中で消えていった構図とも似ています。あるいは、モバイルアプリの世界で PhoneGap や Titanium が消え、Flutter/React Native も何度も設計し直されている構図とも似ています。

基盤が不安定なときは、薄い抽象が勝ちます。基盤が安定してから、厚い抽象を築けばいい。この順序を守るエンジニアが、長生きします。

LangChain 自身も、2024 年以降、LangGraph というより低レベルな framework への移行を進めています。これは正しい方向修正です。ただ、market の主戦場は、もはや framework ではありません。次の節で見るように、業界の関心は「framework をどう作るか」から「protocol をどう作るか」に移っています。

### 9. マルチエージェントという罠

#### 9-1. 役割分担の幻想

LangChain と並行して、もうひとつ 2023 年〜2024 年に流行したのが、マルチエージェントという発想です。MetaGPT、AutoGen、CrewAI、LangGraph multi-agent、Microsoft の Semantic Kernel、OpenAI の Swarm(のちに Agents SDK として再設計)——こうした「複数の LLM エージェントが役割分担して協働する」フレームワークが次々と登場しました。

発想は魅力的に見えました。人間の組織が、プロダクトマネージャー、エンジニア、デザイナー、QA のように役割分担するように、LLM エージェントも役割分担すればいい。プロダクトマネージャーエージェントが要件を定義し、エンジニアエージェントが実装し、QA エージェントがテストする。これをオーケストレーターが調整する——。

2024 年に入ると、MetaGPT や CrewAI は毎週のように新機能を発表し、「エージェントの組織図」のような概念が SNS で飛び交いました。

#### 9-2. 現場での破綻

ところが、現場で実装してみると、これがまったく動きません。わたし自身も 2024 年の前半に、自分のプロダクト(MyBO という freelance 向け SaaS)の設計検討で、いくつかのマルチエージェント framework を試しました。

典型的な失敗パターンを挙げます。

- エージェント A が「ボタンの色を青にする」と決める。
- エージェント B が同じ画面の別の要素について「ボタンの色を赤にする」と決める(A の決定を知らないか、知っていても重みづけができない)。
- オーケストレーターがこの矛盾を調停しようとするが、両方の主張の根拠を吟味するだけの context を持っていない。結局、どちらかを雑に採用するか、両方を採用して画面の中に青と赤のボタンが混在する。

別のパターン。

- エージェント A が「データベースは PostgreSQL を使う」と決める。
- エージェント B がそれを知らずに「データベースは MySQL を使う」という前提でコードを書く。
- エージェント C がそれらを統合しようとして、両方のスキーマを混ぜた謎の SQL を吐く。

これらは、エージェント同士が共有する context が足りないときに起きる典型的な症状です。

#### 9-3. Don't Build Multi-Agents の二原則

先ほど 7 章で紹介した Cognition の `Don't Build Multi-Agents` 記事は、このマルチエージェントの失敗の構造を、二つの原則として整理しています。わたしはこの原則を、強く同意します。

第一原則。context を共有せよ。共有するのは単にメッセージの要約ではなく、agent のフル trace を共有せよ——。

第二原則。行動には暗黙の意思決定が含まれる。意思決定が矛盾すると、悪い結果が生まれる——。

この二原則は、実は、人間の組織論としても真です。組織の中で複数の人間が独立に動くとき、全員が同じ context を持っていなければ、やがて矛盾した判断が重なります。人間の組織では、mtg や Slack や Notion やドキュメントを通じて context を共有することで、これを防ぎます。

LLM エージェントの世界では、この context 共有を technical に実装しないといけない。ところが LLM の context window は有限で、共有のコストはメッセージ数の 2 乗に比例します。5 エージェントが相互に context を共有するためには、25 回のコピー転送が発生する——これは scalable ではありません。

結論として、複数エージェントが独立に動く設計は、だいたいうまくいかない。ならば、単一のエージェントに集約してしまうか、あるいは、コピー転送のない明確な因果ツリー(親が子を作り、子は親の部分タスクだけを担う)の形に制約するか——このどちらかしか生き残れない。

#### 9-4. Anthropic の別の知見

興味深いのは、Cognition がマルチエージェントに悲観的になる一方で、Anthropic は 2025 年 6 月 13 日に `How we built our multi-agent research system` という、マルチエージェントを肯定する記事を出していることです。

一見、Cognition と Anthropic が真っ向から対立しているように見えます。ところが、注意深く読むと、両者の主張は実は一致しています。

Anthropic の記事が強調するのは、以下の条件が揃ったときにのみ、マルチエージェントが成立する、という点です——各サブエージェントのタスクが小さい、タスク境界が明確、サブエージェントが返すのは圧縮された findings である——。

つまり、サブエージェント同士が「対話」するのではなく、親エージェントがサブエージェントに「明確なタスク」を投げ、サブエージェントは圧縮した結果を返す。これは、UNIX のパイプの考え方と一緒です。各プロセスは独立で、入力と出力の間に明確な約束がある。context は流れない。text stream だけが流れる。

Cognition が批判したのは「対話型の多エージェント」で、Anthropic が推奨するのは「パイプ型のサブエージェント」。両者は言葉が違うだけで、主張は同じです。独立したプロセスを並列に走らせて、明確な結果だけを統合する——これは UNIX の書き方です。そして、これが効率的なマルチエージェントの唯一の安全な形です。

#### 9-5. Claude Code の subagent

この「UNIX のパイプ型」のサブエージェントは、Claude Code の `subagent` 機能として実装されています(これは第三部で詳しく見ます)。

Claude Code の subagent は、親の Claude Code が「このタスクを、別の Claude インスタンスにやらせる」という形で起動します。サブエージェントは、親から渡された明確なタスクと、限定された context(自分のタスクに関係するファイルだけ)を持ち、それに対する結果を text で返します。

これは、UNIX で言えば `find . -name '*.py' | xargs grep 'TODO'` のような構造です。`find` が候補を列挙し、`xargs` が並列化し、`grep` が個別に検査する。それぞれのプロセスは他のプロセスの内部を知らない。ただ、text が間を流れる。

マルチエージェントが機能するのは、このパイプ型の形だけです。「プロダクトマネージャー LLM」「エンジニア LLM」「QA LLM」が対話する、という意味でのマルチエージェントは、幻想です。この幻想が 2024 年にいったんピークを迎え、2025 年には実装現場からほぼ消えました。

### 10. MCP という protocol の再発見

#### 10-1. 2024 年 11 月 25 日

2024 年 11 月 25 日、Anthropic が Model Context Protocol(MCP)を発表し、オープンソースとして公開しました。中心となった設計者は David Soria Parra と Justin Spahr-Summers の二人です。

MCP のコンセプトは、実は極めて単純です。LLM エージェントが外部のツール(ファイルシステム、API、DB、Slack、GitHub、など)とやり取りする際の、標準化された protocol を定める。どの LLM ベンダーも、どのツール提供者も、この protocol を喋れば相互運用できる——それだけです。

発表直後、Simon Willison が X(Twitter)でこのプロジェクトを紹介した投稿が広く拡散しました。彼は MCP を「LSP が IDE と言語処理系の間に作った関係を、LLM とツールの間に作る試みだ」と紹介しました。

LSP——Language Server Protocol——は、Microsoft が 2016 年に VS Code のために定義した protocol で、エディタと、各言語の静的解析や補完を提供するサーバーとの間を、JSON-RPC で接続する仕組みです。この LSP のおかげで、たとえば Python 用の言語サーバーを一つ書けば、VS Code にも IntelliJ にも Vim にも Emacs にも対応できる。各エディタと各言語のペアごとに実装を書く N×M の問題が、N+M の問題に縮小した。

MCP は、このアイデアを LLM エージェントの世界に持ち込みました。LLM の種類(Claude、GPT、Gemini、ほか)と、ツール(GitHub、Slack、DB、ファイルシステム、ほか)のペアごとに実装を書く必要をなくし、MCP という共通 protocol の上で相互運用する。N+M で済む。

#### 10-2. 9P の影を見る

この発表を読んだとき、わたしはすぐに Plan 9 の 9P を思い出しました。構造がほとんど同じだったからです。

9P は、Plan 9 の中で「リソースにアクセスする唯一の protocol」でした。ローカルのファイルも、リモートのファイルも、デバイスも、プロセスも、ウィンドウも、全部 9P でアクセスする。MCP も同じで、どんなツールであれ、MCP の server として実装すれば、MCP の client(LLM エージェント)から同じ形でアクセスできる。

Soria Parra は後日のインタビューで、MCP の設計インスピレーションとして LSP を挙げています。9P を直接参照したわけではないようです。ただ、LSP 自体が Plan 9 的な「text 中心の薄い protocol」の系譜に立っていることを思うと、MCP は LSP を一世代経由して 9P の子孫になっていると言えます。

このように、設計原理はしばしば、作者が意図しないまま、何世代かを経て再発見されます。Pike と Thompson が 1984 年に Plan 9 で書いた protocol の発想が、Microsoft の LSP を経由して、2024 年の Anthropic の MCP に到達した——この系譜は、単なる偶然ではなく、「text で薄く繋ぐ」という設計原理が、一度正しく記述されると、異なる問題領域に何度でも適用されるという事実を示しています。

#### 10-3. USB-C という比喩

MCP のメディア向けの説明の中で、「USB-C for AI」という比喩がしばしば使われます。この比喩はわかりやすいので、有効に機能しています。

ただ、技術的にはやや misleading です。USB-C は物理的なコネクタの形状を統一したもので、その上を流れる protocol は実は複数(USB 3、Thunderbolt、DisplayPort、PD、など)あります。一方、MCP は物理層の話ではなく、protocol 層の話です。

より正確な比喩を探すとすれば、RS-232 や TCP/IP の方が近い。これらは「どういう形式で情報を流すか」の標準化であって、物理層は問いません。MCP もそうです。stdin/stdout で喋ることも、WebSocket で喋ることも、SSE で喋ることもできる。流れる format だけが共通。

面白いことに、MCP が初期に採用した最も基本的な transport は、UNIX の stdin/stdout です。MCP の server を起動すると、そのプロセスは stdin から JSON-RPC を読み、stdout に JSON-RPC を書く——これだけの約束で動きます。これは、1970 年代の Bell Labs でも違和感のない設計です。

#### 10-4. エコシステムの爆発

MCP の最初の発表時点で、Anthropic は既に Google Drive、Slack、GitHub、Git、Postgres、Puppeteer などの主要ツールの MCP server を準備していました。このリファレンス実装が、オープンソースコミュニティを刺激しました。

発表から一年後の 2025 年末には、MCP server のディレクトリに数千のプロジェクトが登録されていました。AWS、Cloudflare、Stripe、Linear、Notion、Figma、Datadog——主要な SaaS ベンダーのほとんどが MCP server を提供するようになっています。わたし自身、MyBO の管理画面の一部を MCP server として提供して、Claude Code から直接いじれるようにしています。

OpenAI も 2025 年の途中から MCP を採用しました。Anthropic 発の protocol を OpenAI が採用する——というのは業界的にはやや稀な出来事で、MCP が特定のベンダーの武器ではなく、業界の共通インフラになる可能性を示唆しています。この構造は、LSP が Microsoft 発でありながら、JetBrains や Google も採用した経緯とよく似ています。

#### 10-5. 薄さの価値

MCP を見るときに、もうひとつ重要なのは、この protocol が意図的に薄い、という点です。

MCP は、エージェントとツールの間の「何をやり取りするか」だけを規定します。ツール呼び出しの論理、エージェントの内部状態管理、LLM への prompt engineering、戻り値の加工——これらは MCP の範囲外です。

LangChain との比較が分かりやすい。LangChain は「LLM アプリケーションの全体」を抽象化しようとしました。agent、memory、chain、tool——すべて。MCP は、その中の「agent と tool の間」というごく一部だけを抽象化しました。

この薄さが、protocol の寿命を決めます。LangChain の抽象は、基盤の変化に耐えられませんでした。MCP の抽象は、基盤がどう変わっても変わらないはずの部分(「エージェントは外部ツールを呼び出す」という事実そのもの)だけを固定しています。だから、モデルが変わっても、LLM ベンダーが変わっても、MCP は有効であり続けます。

これは Plan 9 の 9P が、ディスクが HDD から SSD になっても、ネットワークが Ethernet から 100GbE になっても、なお 9P のままで有効であり続けたのと同じ構造です。薄い protocol は、基盤を超えて生き延びる。厚い framework は、基盤の変化とともに朽ちる。

#### 10-6. 2026 年の MCP

本記事を書いている 2026 年 4 月時点、MCP は事実上の業界標準になっています。Anthropic、OpenAI、Google がすべて対応し、主要な SaaS のほぼすべてが server を提供しています。Docker が MCP server をサンドボックスで実行する Docker MCP Toolkit を出し、Cloudflare が serverless な MCP server ホスティングを出しています。

セキュリティ面での議論も活発です。Simon Willison は 2025 年 8 月の `Lethal Trifecta` という講演で、MCP の組み合わせがもたらす攻撃面について警鐘を鳴らしました——private data access、untrusted content exposure、external communication の三つが揃った MCP 構成は、容易に情報窃取の経路になる——。これは UNIX が setuid や権限分離の仕組みを整備するまでに何十年もかかったのと同じ苦労を、MCP がいま始めている、ということです。protocol の普及と、protocol のセキュリティ成熟は、時間差で進みます。

ただ、こうした課題にもかかわらず、MCP は既に不可逆的なインフラになっています。個別のベンダーの個別の tool calling スキーマに戻ることは、現実的にもう不可能です。LSP が 2016 年以降 IDE のインフラになったように、MCP は 2024 年以降、LLM エージェントのインフラになりました。

そして、MCP という薄い protocol の上に乗って、第三部で見る Claude Code、Codex CLI、そのほかの CLI エージェントが動いています。彼らがなぜ CLI なのか、なぜ薄いのか、なぜ `Unix utility` を名乗るのか——その答えを、次の第三部で見ていきます。

第三部では、この塗り替えの上に立つ具体的なプロダクトたち——Claude Code、Codex CLI、そして対比としての Cursor——を見ます。ここでようやく、わたしたちが毎日触っているものの設計思想が、1970 年代の根幹にきれいに接続します。

## 第三部 CLI エージェントという収束点

### 11. Claude Code を `Unix utility` と呼んだ Boris Cherny

#### 11-1. 2025 年 2 月のひっそりとした launch

Claude Code が公開されたのは 2025 年 2 月 24 日です。Anthropic 自身の発表は比較的控えめなもので、同時に発表された Claude 3.7 Sonnet の影に隠れる形でした。最初は `research preview` として公開され、npm 経由でインストールする CLI ツールという、技術的にはまったく目新しくない形での登場でした。

わたしが最初に触ったのは 2025 年 3 月です。インストールは `npm install -g @anthropic-ai/claude-code` と打つだけで、起動は `claude` と打つだけ。ターミナルの中で立ち上がり、カレントディレクトリをワーキングディレクトリとして、自然言語で話しかけると作業が始まる——ただそれだけの道具でした。

正直に言えば、最初の印象は「これがあの Anthropic が出すプロダクトか」という、やや拍子抜けした感じでした。Cursor のように美しい GUI もない。Devin のような自律エージェントの派手さもない。ただの terminal ツール。2025 年の最先端として出てくる代物としては、あまりにも地味でした。

ところが、その地味さこそが設計だったと気づくのに、一ヶ月もかかりませんでした。

#### 11-2. 2025 年 5 月 7 日、Latent Space podcast

Claude Code の設計思想が公の場で最初に詳しく語られたのは、2025 年 5 月 7 日に Latent Space podcast でホストの swyx が Boris Cherny にインタビューした回です。タイトルは `Claude Code: Anthropic's Agent in Your Terminal`。この 50 分ほどの会話の中で、Cherny は何度か、Claude Code を `Unix utility` と呼びました。

わたしにとって印象的だった彼の発言のひとつが、これです(わたしが要約します)——Claude Code は、`grep` や `cat` と同じように compose できるべきだ。`grep` の上に `sort` を乗せ、`sort` の上に `uniq` を乗せる。同じ感覚で、Claude Code を他のツールと組み合わせられるべきだ——。

この一文で、Claude Code の設計思想の大枠が見えてきます。Claude Code は、「AI コーディング IDE」ではありません。「ターミナルの中で動く、LLM を持った、一つのツール」です。`grep` や `awk` と同じカテゴリに属する。他のツールと pipe で繋いで compose できる。

実際、Claude Code には `-p`(`--print`)という非対話モードがあります。これを使うと、Claude Code は通常の CLI フィルタのように振る舞います。

```bash
echo "この関数を説明して" | claude -p < src/parser.py
```

これで、`src/parser.py` の内容を入力として Claude Code に渡し、結果が stdout に返ってきます。`|` や `>` でリダイレクトすれば、CI のスクリプトや shell パイプラインにそのまま組み込める。これはまさに McIlroy が 1964 年のメモで想像した「プログラムを庭のホースのように接続する」そのものです。

#### 11-3. 作らなかったものの一覧

Cherny の発言の中で、わたしがもうひとつ重要だと思うのが、「作らなかったもの」の話です。

Claude Code は、独自の IDE を作りませんでした。VS Code Extension はありますが、Claude Code 自身の本体ではなく、バックエンドとしての Claude Code を呼び出す薄いラッパーです。

Claude Code は、ベクトルデータベースを作りませんでした。初期の社内 dogfooding では vector DB を使っていたそうですが、脆弱すぎるとして廃止されました。代わりに、`glob`、`grep`、`find`、`bash` という、1970 年代から変わらない UNIX ツールを LLM に使わせる方向に舵を切りました。

Claude Code は、独自のメモリーシステムを作りませんでした。セッションをまたぐ記憶は `CLAUDE.md` というただの markdown ファイルです。セッション開始時にそれを context に読み込ませる——それだけ。

Claude Code は、独自の context 圧縮アルゴリズムを作りませんでした。context が溢れそうになったら、Claude 自身に「これまでの会話を要約して」と投げて要約させます。Cherny はこの選択について、こう言っています(意訳)——わたしたちは複雑な圧縮アルゴリズムをいくつも試した。そして最終的に、一番シンプルなやり方——Claude に要約させるだけ——で十分だと気づいた。モデルが十分に賢いなら、over-engineering は必要ない——。

この「作らなかったものリスト」は、Multics への反抗としての UNIX、フル機能 OS への反抗としての Plan 9 と、構造がまったく同じです。「あえて作らない」ことで、別のところで勝つ——この戦略です。

#### 11-4. Cat Wu の product 観

Claude Code の product lead である Cat Wu は、2026 年 3 月 19 日に Anthropic の blog で `Product management on the AI exponential` という興味深い記事を書きました。この記事の中で彼女は、こう言っています(意訳)——Anthropic の全チームで共通する原理がある——Do the simple thing that works——シンプルで機能するものを作れ——。

この原理の含意を、彼女は具体的に説明しています(意訳)——あなたのプロダクトが、モデルの限界を補うための賢いワークアラウンドをしているとしよう。次のモデルが出たとき、そのワークアラウンドは不要な複雑さになる。わたしたちは Claude Code で何度もこれを経験した。system prompt と tool description を、モデルの限界を補うために重く engineering していた時期があった。そして、モデルごとにこれらを削ってきた。Opus 4.6 では 20% 削減した——。

削った、という言葉が重要です。コードが育つのではなく、削れていく。バージョンが上がるごとにコードが減っていく——これは、ほとんどの商業ソフトウェアとは逆の方向です。普通のソフトウェアはバージョンを重ねるごとに膨張します。ところが Claude Code は、モデルが賢くなるごとに削る。これはまさに Thompson の「書き直すと半分になる」という美学の、2026 年版です。

#### 11-5. bare bones という選択

Cherny は Latent Space の中で、Cursor との対比を明確にしています(意訳)——今日、広いプロダクト・マーケット・フィットを持つものを作りたいなら、Cursor や Windsurf のようなものを作る。それはわたしたちが作りたいものではない。わたしたちは、もっとずっと早い段階のカーブの上にある何かを作りたい。モデルの改善が、多くを subsume する(吸収してしまう)。だからこそ、Code は terminal で動く。より bare bones だ。生のモデルへのアクセスがある。なぜなら、わたしたちは、表層にきれいな UI や scaffolding を作ることに時間をかけていないから——。

bare bones——骨だけ、という言い方が、Pike の `small systems` と響き合います。装飾を削ぎ落として、LLM そのものへの生のアクセスを提供する。その上に何かを作りたければ、ユーザー側が自分で scaffolding を作ればいい。これが CLI ツールの美学です。

この選択は、ビジネス的にはやや逆張りです。GUI の美しさと完成度で差別化する competitors が多い中で、あえて「骨だけの道具」を出す。ところが、この逆張りが 2026 年の前半に明確に市場で勝ち始めています。Claude Code の ARR(annual recurring revenue)は 2026 年 2 月時点で 25 億ドルを超え、利用者数も 6 週間で 2 倍に増えた、と公表された数値に基づいて報じられています。bare bones なプロダクトが、大きな market で勝つ——これは、1970 年代の UNIX が Multics を、そして商用メインフレーム OS を事実上駆逐していった構造と同じです。

### 12. do the simple thing first という設計原理

#### 12-1. 原理の言葉

do the simple thing first——まずシンプルにやれ。Anthropic 内部で共有されている設計原理を、Cherny は podcast で、Wu は blog で、何度も繰り返しています。

この原理の力は、複数の具体的な設計判断に、同じ結論を導くことにあります。

例えば context 圧縮。難しい方向と、簡単な方向があります。難しい方向は、独自のアルゴリズムを設計して、トークンを選別する。簡単な方向は、Claude に要約させる。彼らは後者を選びました。理由は「モデルが十分に賢いなら、そちらで十分だから」。

例えばコード検索。難しい方向は、vector embeddings を作って意味的類似性で検索する。簡単な方向は、`glob` と `grep` で文字列を検索する。彼らは後者を選びました。理由は「`glob` と `grep` は 50 年前から動いていて、依存が無く、壊れないから」。

例えばメモリー。難しい方向は、ナレッジグラフを構築する。簡単な方向は、markdown ファイルに書く。彼らは後者を選びました。理由は「markdown は人間が読めて、git で差分管理できて、grep で検索できるから」。

#### 12-2. simple の下にある前提

`simple` という言葉は、しばしば誤解されます。「何も考えないで、まずできるだけシンプルに書け」という意味ではありません。`simple` が機能するのは、以下の前提がすべて成立しているときです。

前提 1。基盤(LLM)が十分に賢い。だから、凝った仕組みで支える必要がない。
前提 2。基盤が今後、より賢くなる。だから、凝った仕組みを作るとすぐ陳腐化する。
前提 3。凝らない選択は、凝る選択より、ユーザーにとっての投資保護になる。なぜなら、削るときに痛みが少ないから。

この三つが揃わない場合、`simple` は単なる手抜きになります。たとえば、基盤が不安定なときに `simple` を選ぶと、基盤の不安定さがそのまま表面に出ます。これは良くない `simple` です。

Claude Code の `simple` が機能しているのは、Anthropic が自社の LLM の改善ペースに自信を持っているからです。6 ヶ月後には Claude の能力が上がる——その確信があるから、いま複雑な scaffolding を作る必要がない、という判断ができる。もしも LLM の改善が頭打ちになったら、Claude Code の設計原理は考え直さなければいけない。ただ、いまのところ、改善ペースは落ちていない。

#### 12-3. Bitter Lesson の影

Boris Cherny が Latent Space の中で明示的に言及したのが、Richard Sutton の `Bitter Lesson` です。Sutton は 2019 年に、同名の短いエッセイで、AI 研究史を総括する厳しい教訓を書きました。

内容を要約します(意訳)——70 年の AI 研究の歴史が示しているのは、一般的な計算手法(scaling)が、人間の工夫を特化的に埋め込んだ手法を、必ず長期的に打ち負かす、という事実だ。チェスでも、Go でも、音声認識でも、画像認識でも、NLP でも、同じことが起きてきた。なぜ bitter(苦い)かというと、研究者の知的労働を無価値化するからだ——。

Cherny はこの教訓を、プロダクト設計にも適用しています。わたしたちが、knowledge graph や vector database のような「人間が設計した仕組み」を LLM に組み合わせたとしても、LLM が十分に賢くなれば、その仕組みは subsume(吸収)されて要らなくなる。だから最初から、生の LLM に直接働かせる設計にしておけばいい——。

この主張は挑発的ですが、Claude Code の実装履歴を見る限り、実証されています。最初は必要だった retrieval augmentation、structured memory、explicit planning——これらが、モデルが賢くなるたびに削られていきました。

#### 12-4. モデル 6 ヶ月後を想定する

Cherny が podcast の中でもうひとつ語った重要な話が、「6 ヶ月後のモデルを想定して作れ」という、彼の manager である Ben Mann からのアドバイスです(Ben Mann は Anthropic の共同創業者の一人)。

Cherny は当初、この助言に戸惑ったそうです(意訳)——最初の半年間、Claude Code は決して良いプロダクトではなかった。わたし自身、自分のコーディングの 10% くらいしか使わなかった。ところが、Sonnet と Opus の 4 が出たとき、プロダクトが急に機能し始めた。わたしはコードの半分を Claude Code で書くようになった。これはプロジェクト開始からちょうど 6 ヶ月後のことだった——。

この経験から、Cherny は builders に向けてこう言っています(意訳)——最初の 6 ヶ月は、product-market fit が悪くて苦しい。ただ、次のモデルが出たら、準備のできている君たちは走り出せる——。

モデル未来を見越して現在のプロダクトを設計する——これは、従来のソフトウェア設計とは発想が違います。通常、プロダクト設計は「今のユーザーの今のニーズ」に合わせます。Cherny の設計は「6 ヶ月後のモデルが今のユーザーにもたらす新しいニーズ」に合わせています。

この時間軸のずれは、示唆的です。UNIX の開発者も、同じようなずれを持っていました。彼らは、当時の主流だったメインフレームの流儀ではなく、「今はマイナーだが、やがて来るはずのミニコンと端末の流儀」に合わせて UNIX を設計した。5 年後、10 年後の計算機環境を想定して設計した。結果、UNIX はその時代が来たときに、ほとんど競合なく勝利しました。

Cherny の `6 months from now` の発想は、構造的にはこれと同じ時間軸設計です。そして、この発想ができるのは、自社が LLM を作っている、つまり「6 ヶ月後に何ができるようになるか」を予想できる立場にあるからです。Anthropic が Claude Code を作るのが、外部の builders が Claude で何かを作るより合理的になる最大の理由が、この内部知識のアドバンテージです。

### 13. Skills、Subagents、Hooks がすべて text である理由

#### 13-1. Skills という発明

2025 年 10 月 16 日、Anthropic は Agent Skills という機能を発表しました。Claude Code から、Claude.ai のデスクトップアプリから、そして API 経由の Claude Agent SDK からも使える仕組みです。

Skills の構造は、恐ろしくシンプルです。一つの Skill は、以下の構造を持つディレクトリです。

```text
my-skill/
├── SKILL.md
└── (任意のファイル: scripts/, references/, etc.)
```

`SKILL.md` は、YAML frontmatter と markdown 本文からなるファイルです。frontmatter には `name` と `description` が必須。これだけです。

```markdown
---
name: pdf-processor
description: PDF ファイルの読み込みとテキスト抽出を行うときに使用する
---

# PDF 処理 Skill

このスキルは、PDF ファイルからテキストを抽出し、構造化された形式で返します。

## 使い方

まず `scripts/extract.py` を見て、既存の抽出ロジックを確認してください。
```

Claude Code が起動すると、インストールされている Skills の `name` と `description` を context に読み込みます。ユーザーがタスクを投げると、Claude はタスクの内容と各 Skill の `description` を照らし合わせて、関連する Skill を自分で選びます。選んだ Skill の本体を読むのは、必要になったその瞬間です(progressive disclosure——段階的開示、と呼ばれる設計)。

#### 13-2. 既視感の正体

Skill の仕組みを初めて見たとき、わたしは既視感を持ちました。どこで見たものに似ているのか、少し時間をかけて考えて、答えに辿り着きました——Plan 9 の名前空間、そして UNIX の man page です。

man page は、text で書かれたドキュメントです。`man ls` と叩くと、`ls` の使い方の text が表示されます。コマンド自体は独立したバイナリで、man page はまったく別のファイルですが、疎結合に繋がっている。必要なときに読めばいい——これが Skill の本体の markdown と同じ構造です。

Plan 9 の名前空間は、「必要なときに必要なリソースだけを mount する」仕組みでした。Skills も似ています。Claude は、起動時に全 Skill の description だけを読み、必要になったものの本体だけを後から読む。名前空間的に言えば、必要なものだけを context に mount している、と言える。

そして、すべてが text です。Skill の name も description も本体も、全部 markdown。Claude はこれを読んで理解する。管理者(ユーザー)はこれを書いて diff を取り、git で管理する。どこにもバイナリも独自形式もない。

Simon Willison は 2025 年 10 月 16 日の自分のブログで、Skills についてこう書きました(意訳)——わたしは Skills が、ひょっとすると MCP より大きな話だと思う。なぜなら、これは Claude に何かを「教える」最も軽い方法だからだ——。

Willison の「軽い」という表現が、ぴったりきます。重い framework も、独自の DSL も、複雑な JSON スキーマも要らない。markdown を置けば、Claude が読んで使う。それだけ。

#### 13-3. Subagents の独立性

Skills と並んで、Claude Code のもうひとつの拡張機構が Subagent です。これは `.claude/agents/` ディレクトリに markdown で定義します。

```markdown
---
name: code-reviewer
description: プルリクエストのコードをレビューする
---

あなたはコードレビュアーです。以下の観点でコードを確認してください:

- セキュリティ(SQL インジェクション、XSS など)
- パフォーマンス(N+1 問題、ループ内の重い処理)
- 可読性(命名、コメント、関数分割)

渡されたファイルを読み、各観点について `OK` か、あるいは問題があれば具体的な指摘を返してください。
```

Claude Code から Subagent を呼び出すと、親の Claude とは独立した、新しい Claude のインスタンスが起動し、このサブ agent の prompt だけを持って、渡されたタスクをこなします。サブ agent の context は親と分離されているので、親の context を汚染しません。

これは第二部の 9 章で触れた、UNIX パイプ型のサブエージェント構造そのものです。`find . | xargs grep foo` の `xargs` が並列にサブプロセスを起こして、それぞれが独立に `grep` を実行するのと、構造が同じです。

そして、Subagent の定義も、すべて markdown です。text で書いて、git で管理し、grep で検索できる。Claude Code のすべての拡張点は、同じ設計思想で貫かれています——text を書けば、Claude がそれを読んで動く。

#### 13-4. Hooks という現場の固定点

もうひとつ、Claude Code の拡張機構として重要なのが Hooks です。これは `.claude/settings.json` に記述します。

Hooks は、Claude がツールを呼び出す前後のタイミングで、ユーザー定義のシェルコマンドを確実に実行する仕組みです。たとえば、ファイル書き込み(`Write` または `Edit` ツール)の後に `eslint --fix` を走らせたい、という場合、以下のように書けます。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npx eslint --fix $CLAUDE_FILE_PATHS"
          }
        ]
      }
    ]
  }
}
```

Hooks の思想が興味深いのは、「Claude を信用しない」ための仕組みではなく、「Claude が高確率でやることを、確実にやるための仕組み」だ、という点にあります。

具体例で言えば、「ファイルを書いた後に lint する」という指示を、`CLAUDE.md` に書いたとします。Claude は 95% くらいの確率でこれを守ってくれるでしょう。ただ、5% の確率で忘れるか、あるいは実行したつもりで実行していない場合があります。これを 100% にするには、指示を文字で書くのではなく、ツール呼び出しのタイミングでシェルコマンドとして必ず走らせる、という設計にするしかない。

これは航空機のチェックリストの考え方と同じです。パイロットは全員、離陸手順を暗記しています。それでも、毎回紙のチェックリストを読みます。暗記 95%、読み上げ 100%——この 5% の差が、事故を防ぐ。

Hooks の設計も、やはり text 中心です。JSON で書く。シェルコマンドをそのまま呼ぶ。git で管理する。Claude が読めて、人間が読めて、diff が取れる。

#### 13-5. text であることの意味

Skills、Subagents、Hooks——Claude Code の拡張機構はすべて、markdown と JSON と shell script で書かれます。独自のバイナリ形式はありません。独自の DSL もありません。全部、人間も機械も読める text です。

なぜ text か——ここが本記事の第二の核心です。

第一に、text は LLM にとって母国語です。LLM は、JSON の深い構造よりも、markdown で書かれた散文の方が理解度が高い。これは LLM の訓練データが web 上のテキスト中心であることに起因します。

第二に、text は git で差分管理できます。Skills を書き換えたり、Subagents を改良したり、Hooks を追加したりする際に、commit 単位で変更を管理し、レビューし、ロールバックできる。これはソフトウェアエンジニアリングの根幹です。独自形式のバイナリでは、この恩恵が得られません。

第三に、text は人間が読めます。あるプロジェクトに新しく入ってきた人が、`.claude/agents/` ディレクトリを覗けば、そのプロジェクトで Claude にどんな役割分担をさせているかが一目で分かる。説明書が不要になる。これは UNIX の設計美学——システム自体がドキュメントになる——と同じです。

第四に、text は pipe で流せます。Claude Code の出力を `grep` に渡し、`grep` の結果を `awk` に渡し、`awk` の結果を別の Claude Code 呼び出しに渡す——こういう合成が可能です。これは McIlroy が 1978 年に書いた「text stream が universal interface である」という主張の、2026 年版の実装です。

この四つの理由が、Claude Code の中で text を「すべての拡張点の universal interface」にする設計判断を支えています。そして、この判断は、Bell Labs の 1970 年代の判断と、構造的には同じです。

### 14. RAG を捨て、glob と grep に戻った日

#### 14-1. 初期実装の RAG

Claude Code の初期の内部実装は、実は RAG を使っていました。RAG——Retrieval-Augmented Generation——は、コードベースを vector database にインデックスし、LLM が関連するコードスニペットを retrieve して context に入れる、というアプローチです。2023 年から 2024 年にかけて、LLM コーディングツールの標準的なアーキテクチャと言えました。

2026 年 4 月 10 日に Anthropic が公開した `Seeing like an agent` というブログ記事の中で、エンジニアの Thariq Shihipar がこう書いています(意訳)——Claude Code が内部で最初にリリースされた時、わたしたちは RAG を使っていた。vector database がコードベースを事前にインデックスし、ハーネスが関連するスニペットを抽出して、毎回の応答前に Claude に渡していた。RAG は強力で速かった。ただ、インデックスとセットアップが必要で、壊れやすかった。そして何より、Claude は context を「与えられる」のではなく、「自分で見つける」べきだった——。

この最後の一文が決定的です。Claude は context を「与えられる」のではなく、「自分で見つける」べきだった——。

#### 14-2. vector DB を捨てる判断

Anthropic は、2024 年の終わりから 2025 年の初めにかけて、Claude Code の内部実装から RAG を取り除きました。代わりに Claude に渡したツールは、本当にシンプルなものだけでした——

- `glob`: ファイル名のパターンマッチ
- `grep`: ファイル内容の文字列検索
- `find`: ファイルシステムの探索
- `bash`: 任意のシェルコマンド実行
- `read`: ファイル読み込み
- `write` / `edit`: ファイル書き込み

これだけ。すべて、1970 年代から存在する UNIX のコンセプトです。LLM が「わたしの関心はこの関数の定義だ」と思ったら、`grep` で探す。`grep` の結果が多すぎたら、より具体的なパターンに絞る。特定のファイルの中身を見たかったら `read` する。

この変更は、実装コストを劇的に下げました。vector DB のインフラは不要。embedding の更新も不要。インデックスの同期問題もない。起動時の待ち時間もない。依存が減り、動作が予測可能になった。

そして、検索の性能は、むしろ向上しました。これは、意外に思うかもしれません。

#### 14-3. なぜ grep の方が良いか

コード検索において `grep` が vector search より良い場面が多い、というのは、実は以前から言われていたことです。

その理由は、コード検索には「完全一致」と「正確性」が重要だから、という点にあります。たとえば、ある関数の呼び出し箇所を全部探したい、というタスクを考えます。関数名は `parseUserInput` です。`grep -r 'parseUserInput' .` と叩けば、漏れなく全箇所が見つかります。

これを vector search でやろうとすると、次のような問題が発生します。

- `parseUserInput` と意味的に近い他の関数(`parseInput` や `parseUser` など)も hit してしまう。
- embedding のチャンク境界で関数呼び出しが分断されると、hit しない場合がある。
- cosine similarity の閾値によって、結果が変動する。

つまり、vector search は「意味的に近いもの」を探すのは得意ですが、「特定の文字列に厳密一致するもの」を漏れなく探すのは、苦手です。コード検索は後者の問題が多い。だから `grep` の方が良い。

加えて、`grep` は「壊れない」。プロジェクトが数万ファイルあっても、何十万行あっても、数秒で動きます。インデックスのメンテナンスが不要。ファイルが増えても減っても、シェルで `grep` を叩けばそのときの最新の状態で検索できる。これは、vector DB には真似できない robust さです。

#### 14-4. agentic search の発明

Boris Cherny は Pragmatic Engineer のインタビューで、この Claude Code の検索戦略を `agentic search`(エージェント的検索)と呼んでいます(意訳)——Claude Code の agentic search は、実のところ、ただの glob と grep だ。そして、これが RAG を上回る——。

`just glob and grep`——というのが、2026 年の最新のコーディングエージェントの内部実装を説明する言葉として使われている。この事実は、ある種、滑稽ですらあります。1974 年から動いている `grep` が、2026 年の新しい LLM エージェントの検索エンジンとして採用されている——。

ただ、これは滑稽な話ではなく、むしろ論理的な帰結です。強力な LLM に、シンプルで堅牢なツールを与えると、LLM は人間のエンジニアがするように、それらのツールを組み合わせて問題を解きます。最初に `grep` で候補を絞り、候補が多ければより specific なパターンで絞り込み、最終的に `read` で中身を確認する——というフローを、LLM は自然にやります。人間のエンジニアが 30 年以上やってきたのと、ほぼ同じフローです。

#### 14-5. 業界が agentic search に収斂していく

Anthropic が RAG を捨てた判断は、内部の実装変更にとどまらず、コーディングエージェントの業界全体に影響を与えました。

2025 年の春以降、各社のコーディングエージェントは、次々と同じ方向に舵を切っています。

Paul Gauthier の `Aider` は、もとから tree-sitter ベースの repo map を使って「プロジェクトの構造を Claude に見せる」というアプローチを取っていましたが、2025 年半ばには具体的なコードの取得を embedding から離し、基本的には LLM 自身にファイルを読ませる方向に寄せました。ユーザーが `/add file.py` で明示的にファイルを追加するほか、エージェントが必要と判断したらファイルを要求する——という UX に整理されています。

OpenAI の `Codex CLI` は、2025 年 4 月にオープンソースで公開された時点から、RAG を採用していません。最初から tool-based で、内部的には Rust 製の薄いハーネスが `rg`(ripgrep)、`fd`、`cat`、`sed` といった UNIX ツールを LLM に露出させる、という作りです。Anthropic とはまったく別の組織が、まったく同じ結論に独立に到達しました——これは、同じ土台の上では、同じ設計が自然に再発見される、という事実の傍証です。

Sourcegraph の `Cody` は、もともと同社のコード検索エンジン Zoekt を背景に持つプロダクトだったので、RAG の色合いが強く残っていました。それでも 2025 年後半からは、vector-based retrieval はあくまで補助とし、主たる検索は tool-based の structural search(関数定義の位置、参照箇所の列挙、type hierarchy)に寄せる方針に転じています。彼らの場合、vector DB を完全に捨てたわけではなく、「意味的に似たコードを探す」という vector が得意なユースケースのためには残している——この切り分けは、実装の洗練された形の一つです。

まとめると、vector DB は 2026 年の段階で、コーディングエージェントの一次検索手段ではなくなりました。ただし、「意味的に似たコードを exploratory に探す」「大規模コードベースを pre-filter する」「ドキュメント全体を対象に類似検索する」といった用途には、依然として有効です。要するに、vector DB がダメだったのではない。vector DB を「標準の検索手段」として据えたアーキテクチャが、時代に合わなくなった、ということです。

#### 14-6. 見えない検索から、見える検索へ

agentic search への移行は、ユーザー体験のレベルでも重要な変化をもたらしました。RAG のコーディングエージェントは、「なぜ Claude がこのコードを見ているのか」がユーザーから見えません。vector DB が、なにか類似度の高いものを引いてきて、それが context に入っている——ただそれだけ。検索は不可視です。

これに対して、agentic search のコーディングエージェントは、Claude が実行したツール呼び出しがすべて画面に表示されます。`grep -r 'parseUserInput' .` と叩いた結果が見える。`read src/parser.py` で何行目から何行目を読んだかが見える。ユーザーは、Claude の思考過程を、ツール呼び出しの連鎖として追える。

これは、デバッグ可能性の向上です。Claude の出した答えが不満なときに、「なぜそう答えたのか」をユーザーが遡れます。検索が意図どおりだったのか、そもそも検索の対象が間違っていたのか、それともモデルの判断に問題があったのか——切り分けができる。

UNIX の `set -x` でシェルスクリプトの実行トレースが見えるのと、同じ構造です。実行過程が見えるから、修正できる。見えない実装の上にデバッグは乗りません。この「見えること」が、2025 年以降の coding agent の必須要件になりました。

### 15. Codex CLI との併走、Cursor との対比

#### 15-1. OpenAI Codex CLI の公開

2025 年 4 月、OpenAI は自社のコーディングエージェントを `Codex CLI` としてオープンソースで公開しました。名前は 2021 年に公開されていた古い Codex モデルを連想させますが、実体はまったく別物で、最新の OpenAI モデルを使う terminal エージェントです。

実装は Rust。ターミナルで起動して、カレントディレクトリをワーキングディレクトリとして、自然言語で指示を受ける——という UX は Claude Code とほぼ同じです。内部的にも、LLM に `shell` や `apply_patch` のようなごく少数のツールを渡して、LLM がそれらを組み合わせて作業する、という tool-based のアーキテクチャです。

Claude Code と Codex CLI は、ベンダーが違い、モデルが違い、実装言語も違いますが、設計思想は限りなく近い。両者とも、

- CLI として動く
- LLM にごく少数のプリミティブなツールを渡す
- vector DB を使わない
- `CLAUDE.md` / `AGENTS.md` のような markdown でプロジェクト固有の指示を与える
- MCP を通じて外部ツールと接続する
- サブエージェントを起動して並列処理する

——という共通項を持ちます。これらは偶然ではなく、「LLM に作業させるための骨組みとして何が必要か」を突き詰めた結果、両者が独立に同じ場所に着地したのです。

#### 15-2. 協力競合という関係

Claude Code と Codex CLI は、市場では競合プロダクトです。ただ、エコシステムのレベルでは協力しています。代表的なのが MCP です。Anthropic が作った MCP を、OpenAI が Codex CLI で素直に採用した。その結果、ユーザーが書いた MCP server は、Claude Code でも Codex CLI でも同じように使えます。どちらの agent を選んでも、ユーザーの周辺資産は無駄になりません。

もう一つの共通インフラが、Skill や `AGENTS.md` のような markdown による指示書です。`AGENTS.md` というファイル名は、OpenAI が提唱し、Claude Code も `CLAUDE.md` に加えて読むようになりました。ユーザーは、一つの markdown を書けば、どのエージェントも同じ指示を理解する——という方向に業界が歩み寄っています。

この「共通の土台の上で競争する」関係は、1990 年代以降のブラウザ戦争で、HTML / CSS / JavaScript という共通の土台の上で Microsoft と Netscape が競争したのと、構造がよく似ています。あるいは、POSIX という共通の土台の上で、Linux、BSD、macOS、Solaris が競争したのに似ている。共通の土台が先に確立して、その上で各社のプロダクトが競う——これは健全な産業構造の形です。

#### 15-3. Cursor という別路線

一方、Cursor は別の路線を取っています。Cursor は VS Code の fork をベースにした IDE です。エディタの中に LLM 機能を統合し、コード補完、チャット、agent モードといった多層のインターフェースを提供します。Windsurf(旧 Codeium)も似た路線です。

Cursor の戦略は、「既存のエディタ体験に AI を溶け込ませる」ことです。開発者が普段使っている UI の中で、自然に AI を呼び出せる。これはこれで、正当な設計判断です。とくに、vim や emacs に愛着のない大多数の開発者にとっては、Cursor の方が参入障壁が低い。

ただ、この路線は Pike の格言に従って言えば、「大きなシステムの上に小さな AI を乗せる」形です。IDE が大きなシステムで、その上に AI が乗る。AI は、IDE の枠組みの中でしか動けない。Cursor のチャットは Cursor の中でしか走らないし、Cursor の agent は Cursor の UI の中でしか結果を表示しません。

Claude Code と Codex CLI は逆です。「小さな CLI ツールを、既存の開発環境に後から合成する」形。`tmux` のペインの中で動く。`pipe` で繋げる。`cron` で走らせる。`CI` の中で呼び出す。CLI という薄い土台が先にあって、その上で自由に組み合わせる——これは Pike の言う `to build a UNIX out of a lot of little systems` に近い。

どちらが良い、という話ではありません。ただ、時間軸が違うのです。Cursor は「今の開発者の今の体験」を最適化しています。Claude Code と Codex CLI は「6 ヶ月後のモデルで何ができるか」に賭けています。Cherny の言葉を借りれば、Cursor は「広い product-market fit」を、CLI エージェントは「より早い段階のカーブ」を狙っている。

#### 15-4. 収束の兆し

そして、興味深いことに、この二つの路線は、徐々に歩み寄っています。

Cursor は MCP をサポートし、ターミナルで動く Cursor Background Agents を出し、GUI の外側で動く autonomous な部分を増やしています。Cursor もまた、「大きな IDE の内側ですべてを完結させる」路線から、「薄い protocol と CLI を内部に取り込む」方向へ動いている。

Claude Code と Codex CLI は、VS Code Extension を介して IDE 側にも進出しています。Claude Code の VS Code Extension は、実態は CLI を呼び出す薄いラッパーですが、エディタから自然に呼び出せる UX は提供しています。

結果として、数年後には、「GUI に溶け込んだ AI」と「CLI で自立する AI」の区別は、徐々に薄れていくと考えられます。重要なのは、どちらの形態であれ、内側で動いているのは tool-based で text-centric な薄いハーネスであり、その下で MCP が共通の protocol として流れている、という構造です。

形は違って見えても、土台は同じ。これが、2026 年の coding agent 業界の実相です。

ここまでで、第三部、そして本記事の技術的な中核は終わりです。Multics から Plan 9 へ、そして MCP と Claude Code へ——text stream を universal interface とし、小さなツールを組み合わせる、という設計原理が、半世紀を経てなお有効であり続けている、という事実を見てきました。

最後の第四部では、この設計原理を、読者自身の日々のツールの扱いにどう適用するか、という実践の地図を描きます。

## 第四部 読み解き

### 16. Bitter Lesson と scaffolding、削ることの設計

#### 16-1. Sutton の教訓、ふたたび

第三部 12 章で少し触れた、Richard Sutton の `Bitter Lesson` を、もう一度振り返ります。このエッセイは 2019 年に書かれた短いテキストですが、AI エージェントの設計を考えるうえで、本記事の中でいちばん重要な一文が、そこに書かれています。

Sutton の主張を、わたしの言葉で再構成すると、こうなります。AI 研究の歴史の中で、人間が「これこそが人間の知能の秘密だ」と信じた工夫——探索戦略、ドメイン知識、言語学的な規則、視覚の特徴抽出——は、すべて、より大きな計算資源とより一般的な学習アルゴリズムによって、凌駕されてきた。そしてそれは、毎回、研究者にとって bitter(苦い)出来事だった。なぜなら、自分たちが何年もかけて構築した知的構造物が、「単に計算量を増やした一般手法」によって無価値化されたからだ。

Sutton の主張は、研究者個人に対しては残酷ですが、分野全体にとっては建設的です。どのような特化した工夫も、長期的には scaling に負ける。だから、長く残るシステムを作りたいなら、特化しすぎない方がいい——という処方箋が得られます。

これを、LLM エージェントの設計に適用するとどうなるか。

#### 16-2. scaffolding は松葉杖

LLM の周りに作られる、抽象層、framework、専用ルーチン——これらを総称して、本記事では `scaffolding`(足場)と呼びます。scaffolding は、LLM 単独では処理できないタスクを、人間の設計でサポートする仕組みです。

scaffolding は、必要悪です。LLM が、今の能力では特定のタスクをこなせないとき、scaffolding がその不足を補う。たとえば、2023 年の LLM は長い context を扱えなかったので、chunk splitting と retrieval augmentation という scaffolding が必要でした。2024 年の LLM は tool calling の JSON を安定して吐けなかったので、pydantic validator のような scaffolding が必要でした。

ところが、scaffolding の宿命は、モデルが賢くなると不要になる、という点にあります。2025 年以降の Claude や GPT は、長い context をそのまま扱えます。chunk splitting の scaffolding は不要になった。tool calling の JSON も、structured output で自然に吐けます。pydantic validator も、少なくともコーディングエージェントの内部では不要になりつつある。

ということは、scaffolding を重く作れば作るほど、将来その重さが負債になる、という逆説が成立します。Cat Wu が 2026 年 3 月の記事で書いた「モデルが賢くなるたびに system prompt を削る」という体験は、この逆説を運用の中で経験した結果です。

松葉杖は怪我をしている間は必要です。ただ、骨折が治ったら、松葉杖を持ち続ける人はいません。同じように、LLM がある能力を獲得したら、それを補っていた scaffolding は、その時点で撤去しないといけない。

#### 16-3. 削ることが進歩である

通常のソフトウェアは、時間とともに膨らみます。機能が追加される、エッジケースが発見される、ユーザー要望が積み重なる——結果、コードベースは肥大化する。これは「ソフトウェアは腐る」という有名な格言の背景にある現象です。

LLM エージェントは、違う軌跡を描きます。モデルが賢くなるたびに、scaffolding が要らなくなる。すると、コードベースは削れる。system prompt は短くなる。tool description は軽くなる。これが、2025 年から 2026 年にかけて、Claude Code、Codex CLI、その他のコーディングエージェントが共通して辿っている進化の形です。

「削ることが進歩である」という感覚は、Thompson が 1970 年代に「書き直すと半分になる」と言ったのと、構造が同じです。時代と対象は違えど、優れたソフトウェアの進歩の形は、たいてい「削る」の方向を向いている。追加ではなく、除去。積み上げではなく、圧縮。

この感覚を身につけると、プロダクトの健全性を読む目が変わります。「version 4 でも、version 5 でも、system prompt が短くなり続けている」——これは健全な兆候です。「version 6 で、新しい dedicated module が追加された」——これは、モデル能力が頭打ちになったか、プロダクトがユーザー要望の雪崩に負けたかのどちらかです。削る能力を失った設計は、遠からず陳腐化します。

#### 16-4. 薄い protocol、厚い機能

ここで重要な区別があります。「削る」というのは、機能を削ることではなく、scaffolding を削る、ということです。

ユーザーから見える機能は、むしろ増えていきます。Claude Code は 2025 年の launch 時には無かった機能——Skills、Subagents、Hooks、Plan Mode、そして MCP 拡張——を次々と取り込んでいます。機能面では明らかに厚くなっている。

ところが、それぞれの機能の実装は、驚くほど薄い。Skills は markdown。Subagents も markdown。Hooks は JSON と shell script。MCP は stdin/stdout の JSON-RPC。どれも、1970 年代の UNIX が違和感なく受け入れられるレベルの薄さです。

結果として生じるのは、「薄い protocol の上に、厚い機能が生える」という構造です。protocol と scaffolding は薄く。機能とエコシステムは厚く。この二つの軸を混同しないのが、設計の肝です。

Plan 9 の 9P がそうでした。protocol 自体はごく薄い 9 つのメッセージ。ただ、その上に、ファイルサーバー、CPU サーバー、ウィンドウシステム、ネットワーク、認証——あらゆる機能が育った。薄い土台が、厚い上部構造を支える。これが、長生きするシステムの形です。

### 17. 読者への地図、次に何を削るか

#### 17-1. 自分の CLAUDE.md を読み返す

ここから先は、読者自身の日々のツールに適用するための、実践的な問いです。まず最初にやってほしいのは、自分の `CLAUDE.md` を読み返すこと、です。

`CLAUDE.md` は、Claude Code がセッションごとに読み込む markdown ファイルで、プロジェクトやユーザーの指示をここに書きます。わたし自身のグローバル `CLAUDE.md` も、ここ半年で内容が大きく変わりました。半年前は、Claude が忘れがちだったことを細かく書いていました。「コードの最後に改行を入れる」「import 文は alphabetical に並べる」「テストを書くときは pytest の fixture を使う」——こういう低レベルの指示です。

ところが、モデルが 4.6、4.7 と上がるたびに、これらの指示は不要になりました。Claude が自分で気づくようになった。いまでは、わたしの `CLAUDE.md` の多くの項目は、削除対象です。残っているのは、プロジェクト固有の慣習(「このリポジトリの言語は日本語。ですます調。一人称はわたし」)や、ドメイン知識(「このサービスはフリーランス市場向け」)のような、モデルの賢さとは独立な情報だけです。

読者の皆さんにお願いしたいのは、年に一度、自分の `CLAUDE.md` を最初から最後まで読み返す、という習慣です。そして、「これは今のモデルには不要かもしれない」と思う項目を、削除してみる。削除してみて、実際に動作が劣化するかどうかを、しばらく観察する。劣化しなければ、その削除は正しい。劣化したら、戻せばいい。

これは、scaffolding を動的に調整する習慣です。モデルの能力向上を、自分の設定に反映させる、という運用です。

#### 17-2. 複雑な slash command を疑う

Claude Code には `.claude/commands/` ディレクトリに slash command を置く仕組みがあります。わたし自身、半年前には 30 以上の slash command を抱えていました。「PR レビュー用」「コードレビュー用」「commit message 作成用」「retrospective 用」——それぞれに、長い prompt が書かれていました。

いまは、そのうち半分くらいを削除しました。理由は、モデルが賢くなって、「PR をレビューして」と自然言語で頼むだけで、かつて slash command でやらせていたことを同じ品質でやってくれるようになったからです。scaffolding が不要になったのです。

読者の皆さんにも、自分が抱えている slash command を、一度棚卸ししてみることをおすすめします。とくに、長い prompt がついた slash command は、疑うべきです。なぜその prompt が必要だったのか、今のモデルでも本当に必要なのか——を、実際に削除して確認してください。

残すべきは、「プロジェクト固有のワークフロー」(例: このリポジトリの PR 作成には、特別な commit message 形式を使う)や、「頻繁に再利用するドメイン知識」(例: 特定の API の呼び出し手順)のような、モデル能力と独立な部分だけです。

#### 17-3. Hooks を「信用しない仕掛け」にしない

13 章で述べたように、Hooks は「Claude が高確率でやることを、確実にやるための仕組み」です。ただ、運用を間違えると、「Claude を信用しないための仕掛け」に堕することがあります。

たとえば、以下のような Hooks を書いている人を時々見かけます——「Claude がファイルを書いたら、毎回 `pytest` を走らせて、失敗したら rollback する」「Claude がコミットしようとしたら、ブロックして、人間の承認を求める」。

これらは、Hook として実装可能ですが、運用としては重い。Claude を信用していない、という態度の現れです。モデルが十分に賢くなった現在、Claude は「ファイルを書いたら動作確認をする」「コミット前に意図を述べる」という振る舞いを、自然言語の指示だけで守ります。重い Hook は、かつてモデルが未熟だったころの scaffolding の名残です。

わたしが今も残している Hooks は、「ユーザー自身の運用上のルール」のための Hook だけです。たとえば、「コミット前に自動で lefthook の lint を通す」「PR 作成時に特定のラベルを自動で付ける」——これらは Claude の性能とは独立で、自分のチームの運用ルールに由来します。こういう Hook は残す価値があります。

判断基準は、「この Hook が、Claude のモデル能力不足を補っているのか、それとも、チーム運用の自動化を担っているのか」です。前者なら、モデルが賢くなったら削除。後者なら、運用が変わらない限り残す。

#### 17-4. 独自 framework を抱えていないか

最後の問いは、もう少し大きなものです。あなたのプロジェクトに、LLM を扱うための「独自 framework」はありませんか。

具体的には、以下のようなものです。

- LangChain を wrap した自社 SDK
- Prompt template を管理する独自ライブラリ
- Tool 呼び出しのための独自 JSON スキーマ(MCP と互換性のないもの)
- 複数 agent の orchestration 用の独自 framework

これらは、それぞれが作られた時点では正当な判断だったはずです。ただ、2026 年の文脈の上で、再評価する価値があります。

- LangChain wrapper は、素の Anthropic / OpenAI SDK で置き換えられないか
- Prompt template は、markdown ファイルと文字列補間で済まないか
- Tool 呼び出しは、MCP に寄せられないか
- 複数 agent は、Claude Code / Codex CLI の subagent 機能に寄せられないか

すべての質問に「イエス」と答えられるとは限りません。ただ、「うちの事情がある」と答える前に、本当にその事情が 2026 年にもまだ有効か、を確かめる価値はあります。

「事情」の多くは、3 年前、2 年前、1 年前に確立されたものです。そして、この 1〜3 年で、LLM の能力も、周辺 protocol も、大きく変わりました。半年前の判断が、半年後にも最適であるという保証はありません。削る目で、自分のスタックを見直す——これを、四半期ごとの儀式にすることをおすすめします。

#### 17-5. 「6 ヶ月後のモデル」を想像する訓練

最後に、Cherny が Builders に向けて語った `6 months from now` の発想を、自分の設計判断に取り入れる訓練を紹介します。

なにか新しい scaffolding を追加しようとするとき、こう自問してみてください——「このコードは、6 ヶ月後のモデルでも必要か?」。

たとえば、「LLM は JSON を時々壊すから、validator を入れよう」という判断。6 ヶ月後のモデルは、structured output で完璧な JSON を吐くかもしれない。そうなったら、validator は不要な複雑さになる。なら、validator の実装は、簡単に削除できる形で書く。あるいは、そもそも入れない。

たとえば、「LLM は context が狭いから、会話を chunk に分けて sliding window で送ろう」という判断。6 ヶ月後のモデルは、2 倍、3 倍の context を扱うかもしれない。そうなったら、sliding window は無用の装置になる。なら、最初から素直に全部送る設計にしておく。context が溢れるときは、後で切り貼りを考える。

この「6 ヶ月後のモデルを想像する訓練」は、scaffolding を増やそうとする自分の手を、毎回止めさせます。いま必要に見えるものが、6 ヶ月後には不要になる——この予感を持てる人が、長く使える設計を書ける人です。

### 18. 結び、流れるものは変わった、土台は変わっていない

#### 18-1. タイトルを回収する

本記事のタイトル「流れるものは変わった、土台は変わっていない」は、本記事の主張そのものを短く言い切ったものです。

まず「流れるもの」。半世紀前には UNIX のパイプ (`|`) を通って text が流れていました。いまでは Claude Code のハーネスの中を、LLM の出力や tool の実行結果が流れています。1970 年代の `cat file.txt | sort | uniq` で流れていた text と、2026 年の Claude Code が扱う tokens は、表層のレイヤーでは明らかに別物です。流れるものの中身は、確かに大きく変わりました。

一方の「土台」。text stream、universal interface、薄い protocol、小さな道具の組み合わせ——この土台の構造は、1970 年代から 2026 年まで、驚くほど変わっていません。Pike の 2000 年の批判「業界は数十年前のクローンに興奮している」が、良い意味でそのまま続いている。

そして、わたしがこの記事で伝えたかったのは、この二つの対比です。流れるものは変わった、ただ、土台は変わっていない。だから、土台の構造を学べば、流れるものが何であっても、それを扱える——という主張です。

#### 18-2. 根幹を持つことの価値

本記事の冒頭で、わたしは自分の経歴として Slackware や Solaris や PDP 系の話を書きました。これは自慢のためではなく、「根幹を持っていること」が、2026 年の AI 時代にどう効くか、を伝えるためです。

LLM のブームに合わせて、AI エージェントのアーキテクチャは、わずか 3 年で、AutoGPT から Devin、LangChain、CrewAI、MCP、Claude Code へと、目まぐるしく変わりました。この変化に一個ずつ対応しようとすると、疲弊します。流行を追いかけるだけでは、翌年には別の流行に追い抜かれる。

根幹を持っていると、見え方が違います。AutoGPT を見たとき、「これは Multics だな」と分かる。LangChain を見たとき、「これは 2000 年代の JS framework 戦争の再演だな」と分かる。MCP を見たとき、「これは 9P の子孫だな」と分かる。Claude Code を見たとき、「これは UNIX だ」と分かる。

判別ができると、不安が減ります。「どのプロダクトを学べばいいのか」「いつ学び直せばいいのか」という不安は、根幹を持っていれば、かなり薄れます。プロダクトの名前は変わる。ただ、設計原理は、そんなに速く変わらない。変化の速さに錯覚されているだけで、根幹はゆっくりとしか動きません。

若いエンジニアの方には、だからこそ、最新のプロダクトだけでなく、古いシステムの設計原理を学ぶことをおすすめします。UNIX の歴史。Plan 9 の論文。Bell Labs の文化。LSP の設計。TCP/IP の薄さ。これらは直接使わなくても、いま目の前にあるプロダクトの構造を読み解く補助線になります。

#### 18-3. 次の根幹はどこか

本記事を締めるにあたって、最後に、5 年後、10 年後を見たときに、根幹がどう動きそうか、わたしの予想を書きます。予想ですから、当たらないかもしれません。ただ、予想の形を示すことで、読者自身が自分の予想を立てるきっかけになれば、と思います。

第一。薄い protocol の勝利は続きます。MCP の上に、より薄い layer が生まれる可能性はあります。たとえば、agent 間の直接通信を扱う protocol、あるいは、long-running agent の状態管理の protocol。ただ、これらも、必ず `薄く` なるでしょう。LangChain のような厚い framework は、今後も市場の主戦場にはなりません。

第二。text 中心の UI は残ります。音声、画像、動画といった multi-modal が拡大しても、エージェント同士の通信、agent と tool の通信、人間と agent の long-form 通信の多くは、text のままです。なぜなら、text が最も安価で、最も editable で、最も再現可能だからです。

第三。bare bones な CLI と、GUI 統合型の IDE は、区別が薄れます。内側で動いている agent ランタイムは共通化し、上に乗る UI は用途に応じて選ばれる——という分化が進むでしょう。Cursor の中で走る agent も、terminal で走る agent も、内部の protocol と scaffolding は同じ。これが 2028 年ごろの均衡状態だとわたしは見ています。

第四。削ることが、より強く進歩の指標になります。どのベンダーも、自社のエージェントの system prompt や framework コードが、version を重ねるごとに縮んでいくことを、誇りとして公表するようになるでしょう。いまは Anthropic だけがこれを明言していますが、数年後には業界標準の KPI になっていると予想します。

第五。設計原理は、さらに古い源流と繋がっていきます。UNIX より前の、Lisp machines、Smalltalk、TECO、さらには Turing の universal machine の議論にまで、新しい agent の設計が遡っていきます。これは、新しい問いを前にしたとき、人間は必ず「最も原始的な道具」に戻る、という経験則の再演です。

#### 18-4. 最後に

本記事は長くなりました。ここまで読んでくださった方に、感謝します。

もし、本記事を読んで、自分の使っているツールや自分の書いているコードを、もう一度「削る目」で見直してみる気になっていただけたなら、わたしがこの記事を書いた目的は達成されたことになります。

流れるものは変わった、ただ、土台は変わっていない。この構造さえ掴んでおけば、これから数年、また新しいプロダクトやフレームワークの波が来たときにも、惑わずに済みます。そして、自分で土台を作る側に回る日が、いつか来るかもしれません。そのとき本記事が、かすかな補助線になれば、望外です。

なお、本記事の冒頭で予告した「ハンズオン指示書」——Claude Code で実際に、`CLAUDE.md` の棚卸し、`.claude/commands/` の削減、MCP server の自作といった手を動かす形式のガイド——は、姉妹記事「[自分の Claude Code は削れるはず ── CLAUDE.md、slash command、MCP、Subagent、Hooks を棚卸しするハンズオン](https://note.com/)」として同時に公開しています。本記事と合わせて読むことで、根幹の理解を、手の感覚に落とし込んでいただけると思います。

それでは、みなさまの次の design judgement が、削る側の判断でありますように。
