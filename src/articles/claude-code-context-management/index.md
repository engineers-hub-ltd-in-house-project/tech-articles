# Claude Code が指示を忘れるとき、何が起きているのか

## はじめに

CLAUDE.md に書いた指示が、セッションの途中から守られなくなる。この現象に初めて気づいたのは、ある日の午後のことでした。序盤は正確に動いていた Claude が、30分ほど経った頃から CLAUDE.md に明記してある命名規則を無視し始め、さらにしばらくすると、5分前に自分で却下したアプローチをもう一度提案してきました。

最初は CLAUDE.md の書き方が悪いのだろうと思いました。しかし原因はもっと構造的なところにありました。コンテキストウィンドウという有限の資源が、セッションの進行とともに劣化していく。この現象は context rot と呼ばれていて、Anthropic 自身が公式ドキュメントで認めています。

この記事は、context rot の仕組みを追いかけ、手元で対処法を試しながら整理した記録です。動作確認は Claude Code 2.1.110、Node 22.16.0 で行いました。

## 第1章　コンテキストは減っていく資源

### 空のセッションがすでに埋まっている

Claude Code を起動して、何も入力せずに `/context` と打ってみてください。セッションの中身が可視化されます。わたしの環境（Opus 4.6、1Mコンテキスト）では、こういう表示が出ました。

```text
> /context

⛁ ⛁ ⛀ ⛀ ⛀ ⛀ ⛀ ⛶ ⛶ ⛶ ⛶ ⛶ ⛶ ⛶ ⛶ ⛶ ⛶ ⛶ ⛶ ⛶   Opus 4.6 (1M context)
                                             17.7k/1m tokens (2%)
Estimated usage by category
⛁ System prompt:     6.3k tokens (0.6%)
⛁ System tools:      8.2k tokens (0.8%)
⛁ System tools (deferred): 7.6k tokens (0.8%)
⛁ Custom agents:      408 tokens (0.0%)
⛁ Memory files:        433 tokens (0.0%)
⛁ Skills:            2.3k tokens (0.2%)
⛁ Messages:              8 tokens (0.0%)
⛁ Compact buffer:      3k tokens (0.3%)
⛶ Free space:       979.3k        (97.9%)
```

1Mウィンドウでは 2% にすぎません。ただしこれを 200K ウィンドウに換算すると同じ約18Kトークンが **約9%** を占めます。Skills を多数登録し、MCP サーバーを複数有効化している環境では初期消費がさらに膨らみ、karaage0703 氏（Zenn）の実測では 200K ウィンドウの **69%が起動時点で消費済み** という事態も報告されています（最適化後39%まで削減）。

注目すべきは System tools の固定コスト（8.2K + deferred 7.6K）です。ビルトインツール定義だけで約16Kトークンを占めており、ユーザーには制御できません。残りの空間に、ファイルの読み取り結果、コマンド出力、会話履歴が積み上がっていきます。

### 40%を超えたあたりで何が起きるか

コンテキストの使用率が上がるにつれて、Claude の振る舞いが変わります。Anthropic の API ドキュメントにはこう書かれています。「コンテキストウィンドウ内のトークン数が増加するにつれて、モデルがそのコンテキストから情報を正確に想起する能力が低下する」。根本原因は Transformer の注意機構にあり、100Kトークンでは100億のペアワイズ関係が発生します。

日本のコミュニティで広まった「Smart Zone / Dumb Zone」という概念が、この実態をうまく表現しています。使用率が約40%を超えると品質劣化が始まる領域に入り、80%を超えると自動圧縮が発動する。わたし自身の観察では、劣化は以下のような段階で進みました。

使用率0〜20%のあいだは安定しています。ファイルの特定は正確で、分からないことを分からないと言える。20〜40%あたりから、ドキュメントを読み直す前に誤ったアプローチに着手し始めます。ただし指摘すれば自己修正する。40〜60%になると自信を持った誤結論が出始めます。捏造された説明を堂々と述べ、こちらから指摘しても修正が不安定になる。60%を超えると、セッション冒頭の事実にアクセスできなくなり、代わりの情報を自分で作り始めます。

具体的にはこういう会話が起きます。セッション冒頭で「テストフレームワークは Vitest を使う」と伝え、Claude も了承して Vitest でテストを書いていたのに、コンテキストが膨らんだ60%あたりで突然こうなる。

```text
> テスト実行が失敗しています。修正してください。

テストの設定を確認しました。Jest の設定ファイルが見当たりません。
jest.config.ts を作成します。
```

指摘すると「すみません、Vitest でした」と謝りますが、数ターン後にまた Jest に戻っている。この循環が、context rot の典型的な症状です。冒頭で述べた「CLAUDE.md が守られなくなる」現象は、まさにこの段階で起きていました。

Opus 4.6 や Sonnet 4.6 では 1Mトークンのコンテキストが使えますが、広いから安全とは限りません。Anthropic 自身の MRCR v2 ベンチマークによると、multi-needle retrieval の精度は 256Kトークンで93%、1Mトークンで76〜78%まで低下します。GitHub Issue #34685 では、1Mコンテキスト使用中の Opus 4.6 が 48%時点で自発的にセッション再開を推奨したことが記録されています。

### 劣化を見るための道具

`/context` は前述の通り、コンテキストの内訳をビジュアル表示します。System prompt、System tools、Custom agents、Memory files、Skills、Messages、Compact buffer、Free space の各区画が一覧され、どこに何トークン使われているかが一目で分かります。セッション中に定期的にこれを打つだけで、劣化が忍び寄るタイミングを掴めます。`/usage` はプラン制限とレートリミットの残量を表示するコマンドで、5時間ウィンドウ内の消費量を確認できます。`/cost` は API ユーザー向けで、セッション単位のトークン消費と推定ドルコストを出してくれます。

## 第2章　/compact の罠 — 要約が嘘をつく日

### 圧縮の仕組み

コンテキストが膨らんできたとき、最初に手が伸びるのは `/compact` です。手動でコンテキスト圧縮をトリガーするこのコマンドは、3段階で動きます。まず古いツール出力（ファイル読み取り結果、grep 出力、bash 出力）がクリアまたは切り詰められる。次に会話履歴全体が「完了済み・進行中・変更ファイル」を含む構造化サマリーに凝縮される。最後に CLAUDE.md がディスクから再読み込みされ、圧縮後のセッションが継続します。

ここまでは理にかなっています。問題は、サマリーの品質が安定しないことにあります。

### 「指示なし /compact」が壊すもの

`/compact` を何の指示もなしに実行すると、生成されるサマリーはこの程度の粒度になります。

```text
[compact summary]
認証機能の実装を進めた。パスワードハッシュのライブラリを検討し、
ミドルウェアにバリデーションを追加した。テストは一部未完了。
```

この要約からは、重要な情報がいくつも消えています。パスワードライブラリを bcrypt から argon2 に変更した経緯、レート制限を1分5回に設定した理由、zod 統合が「未完了」とはどのファイルのどの部分のことか。変更ファイルの追跡が抜け落ち、設計判断の理由が蒸発し、マルチファイル間の依存関係が失われている。結果として、圧縮直後に Claude がファイルを再読み込みし、テストを再実行し、既に終わった作業をもう一度やり始める、という事態が起きます。

さらに深刻な問題があります。GitHub Issue #46602 で報告されたバグでは、**圧縮サマリーが架空のユーザー指示をハルシネーションした**ことが記録されています。圧縮後のアシスタントがこの捏造された指示を正当なものとして実行し、意図しない作業が暴走しました。モデルが既に混乱した状態で圧縮すると、その混乱がサマリーに固定化されるという根本的な問題もあります。劣化したコンテキストを圧縮しても、劣化した要約が残るだけです。

### autocompact の挙動

自動圧縮のデフォルト閾値は、200Kモデルで約80〜83%（約166〜167Kトークン使用時点）です。残り約33Kトークンが autocompact バッファとして予約されます。環境変数 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`（1〜100）で閾値を下げることは可能ですが、コード内の `Math.min()` クランプにより、デフォルトより上げることはできません。

1Mコンテキストでは別の問題もあります。GitHub Issue #34332 では、約76Kトークン（わずか7.6%使用）の時点で autocompact が発動する不具合が報告されています。非公式のワークアラウンドとして環境変数 `DISABLE_AUTO_COMPACT=1` で無効化する方法もありますが、ハードリミットに達するリスクが生じます。

### 圧縮を安全に使うために

わたしが試行錯誤の末にたどり着いた運用は、2つの原則に集約されます。

第一に、保存指示を必ず付ける。`/compact` にはフリーテキストで保存指示を渡せます。

```text
/compact 以下の情報を保持すること:
- パスワードライブラリを argon2 に変更済み（bcrypt から移行）
- レート制限は1分間5回
- src/middleware/validate.ts の zod 統合が未完了
- 設計判断: セッションストアは Redis ではなく DB を選択（理由: 運用コスト）
```

CLAUDE.md は /compact の後にディスクから再読み込みされるため、圧縮時に保持したい情報を CLAUDE.md 側に書いておけば、サマリーに頼らず確実に残せます。公式の best-practices ドキュメントでも「compaction 時に変更ファイルの一覧やテストコマンドを保持する指示を CLAUDE.md に書く」ことが推奨されています。

第二に、60%で予防的に圧縮する。80%まで引っ張ると、劣化したコンテキストを劣化したまま圧縮することになります。劣化が始まる前に圧縮しておけば、サマリーの品質も高く保てます。圧縮後にはかならず「直前の作業内容を箇条書きにして」と送り、サマリーの品質を検証してから次の作業に進みます。

## 第3章　やり直すか、捨てるか — /rewind と /clear

### /rewind で巻き戻す

`/rewind` コマンド、あるいは Esc キーの2回連打で、セッション内の任意のチェックポイントまで巻き戻せます。チェックポイントはプロンプト送信ごとに自動作成されるので、事前設定は不要です。

巻き戻し先を選ぶと、5つの選択肢が提示されます。ファイル変更と会話の両方を戻す「Restore code and conversation」、会話だけ戻す「Restore conversation」、ファイルだけ戻す「Restore code」、選択地点以降を要約する「Summarize from here」、何もしない「Never mind」。

わたしが最も多用するのは「Restore code and conversation」です。たとえば JWT 認証で実装を依頼し、動作確認したあとで Session ベースに変えたくなった場面を考えます。

「やっぱり Session にして」と修正指示を重ねる方法だと、JWT の実装コード、その破棄指示、Session への書き換え指示がすべてコンテキストに残ります。一方、Esc を2回叩いて /rewind を起動し、JWT を依頼したチェックポイントまで戻して「Restore code and conversation」を選べば、JWT の試行はコンテキストからもファイルからも消えます。巻き戻し後のプロンプト入力欄には元の依頼文が復元されるので、「Session ベース認証で実装して」に書き換えて再送信するだけです。

```text
Esc Esc

? Select a checkpoint
❯ [10:23:15] "src/auth.ts を JWT 認証で実装して"
  [10:20:01] "プロジェクトの構成を説明して"

? What would you like to do?
❯ Restore code and conversation
  Restore conversation
  Restore code
  Summarize from here
  Never mind
```

この後 `/context` を打つと、JWT の試行分のトークンが消えていることを確認できます。「Summarize from here」は /compact の精密版として使えます。冒頭のプロジェクト要件は完全保持しつつ、冗長なデバッグセッション部分のみを圧縮できるからです。

修正メッセージを重ねてコンテキストを汚染するより、巻き戻して再プロンプトするほうが、コンテキストはクリーンに保たれます。先ほどの第1章で触れた「やり直し指示の蓄積が context rot を加速させる」問題への、もっとも直接的な回答です。

ただし制限があります。bash コマンド（rm, mv, cp）で変更されたファイルはチェックポイントで追跡されません。また /rewind は CLI でのみ完全サポートされ、VS Code 拡張では一部制約があります。

### /clear で捨てる

`/clear`（エイリアス: `/reset`, `/new`）は会話履歴を完全消去し、新しいセッション UUID を作成します。CLAUDE.md はディスクから再読み込みされますが、それ以外のコンテキストは一切引き継がれません。トークン回復は完全です。/compact のように要約が数Kトークン消費することがありません。ただし不可逆操作なので、元の会話には `/resume` でのみ戻れます。

### どちらを使うか

判断の軸は「今の文脈に価値があるかどうか」です。

方向転換をしたいだけで、今の議論の蓄積にはまだ意味がある。そういう場面では /rewind です。Esc Esc で戻って、プロンプトを書き換えて再送信する。コンテキストに「捨てた試み」が残らないので、context rot の加速も防げます。

一方、まったく別のタスクに切り替える、コミットが完了して一区切りついた、あるいは2回の修正が失敗して会話が迷走している。こうした場面では /clear のほうが確実です。古いコンテキストはノイズにしかなりません。

日本語コミュニティでよく実践されている「Plan → /clear → Implement」パターンは特に効果的でした。Plan Mode で調査と計画を行い、計画をファイルに出力してから `/clear` し、実装を新鮮なコンテキストで開始する。note.com のある記事では「/compact より /clear を圧倒的に多用する」と報告されており、わたし自身も同じ結論に至りました。

## 第4章　サブエージェントという隔壁

### 独立したコンテキスト

サブエージェント（Agent tool）は、コンテキスト管理の観点から見ると最も過小評価されている道具です。各サブエージェントは親とは独立した 200K（または1M）トークンのコンテキストウィンドウを持ちます。親からサブエージェントへの入力は Agent tool のプロンプト文字列のみ。サブエージェントから親への出力は最終メッセージのみ。中間のファイル読み取り、検索、ツール呼び出しは全てサブエージェント側で完結し、親のコンテキストには返りません。

この分離が意味するところを、具体的に考えてみます。プロジェクトの依存関係を調査する場面で、メインで直接作業した場合を想像してください。Claude は package.json を読み、node_modules の中を探索し、npm audit を実行し、各パッケージの CVE を確認する。この過程で Read、Bash、Grep が何十回も呼ばれ、その出力がすべてコンテキストに積まれます。

同じ依頼をサブエージェントに委任すると、構造が変わります。

```text
> サブエージェントを使って、依存関係のセキュリティ調査をして。結果だけ報告して。
```

サブエージェントが独立コンテキストの中で同じ探索を行い、完了後にメインへ返ってくるのは要約だけです。package.json の中身も、npm audit の生出力も、grep の検索結果も、すべてサブエージェント側で閉じています。この後 `/context` を打てば、Messages の増分がサブエージェントの結果要約ぶん（数Kトークン程度）に収まっていることを確認できます。直接メインで同じ調査をした場合の数十Kトークンとは桁が違います。

### ビルトインの3種

Claude Code にはビルトインのサブエージェントが3種あります。**Explore** は Haiku で動く読み取り専用の高速な探索エージェントで、ファイル発見やコード検索に向きます。**Plan** は戦略策定に特化し、実装計画の設計に使います。**汎用**エージェントは任意のタスクを委任でき、カスタムサブエージェントは `.claude/agents/` に Markdown ファイルとして定義できます。

サブエージェントはネストできません（1階層のみ）。ただし並列実行は可能で、スタイルチェック・セキュリティスキャン・テスト実行を同時に走らせるワークフローが効果的です。

### コスト意識

サブエージェントのトークン消費は乗算的に増えます。Anthropic はマルチエージェント構成で最大 15 倍のトークン使用量を報告しています。すべてを Opus で走らせる必要はなく、簡易検索は Haiku、コード分析は Sonnet、複雑な推論だけ Opus というモデルルーティングでコストを制御できます。コンテキストの節約と引き換えに API 費用が増える構造なので、どこに投資するかの判断が要ります。

## 第5章　実務家が見つけた処方箋

公式ドキュメントの外にも、実務のなかから生まれた知見がいくつかあります。

### Document & Clear

英語圏で広く実践されているパターンです。長いセッションの途中で、Claude に現在の状態をファイルに書き出させてから `/clear` する。新しいセッションでそのファイルを読み込んで続行する。実際のやり取りはこうなります。

```text
> 現在のプロジェクトの状態を PROJECT_STATE.md にまとめて。
> 使用ライブラリと選定理由、各ファイルの役割、未完了タスク、
> 設計上の制約、次に取り組むべき作業を含めて。

(Claude が PROJECT_STATE.md を生成)

> /clear

(新しいセッションが始まる)

> PROJECT_STATE.md を読んで、このプロジェクトの状態を把握して。
> 把握したら現在の状態を箇条書きで確認して。
```

/compact と違い、何が保存されたかが目に見えるファイルとして残るため、情報の欠落を自分で確認できます。ファイルに書き出す前に中身を確認し、足りない情報があれば追記を依頼してから /clear する。この透明性が Document & Clear の強みです。Shrivu Shankar 氏のブログでは「/compact はできる限り避ける。自動圧縮は不透明でエラーが多い」と述べられており、Document & Clear をデフォルトの戦略として推奨しています。

Zenn の Continuous-Claude-v2 設計思想も同じ方向を向いています。「Compact するな、Clear しろ」。圧縮を繰り返すと「要約の要約の要約」になり、信号がノイズに変わる。状態をレジャーファイルに保存してコンテキストを完全クリアする方式で、1週間のテストで「忘却」インシデントが12回から3回に減少したと報告されています。

### コンテキストは揮発性 RAM、CLAUDE.md は不揮発性ストレージ

日本語コミュニティで広まったメタファーです。会話履歴はセッションが終われば消える揮発性の RAM であり、CLAUDE.md はセッションをまたいで残る不揮発性ストレージである。この視点を持つと、恒久的なルールや設計判断を会話の中だけに置いておくことの危うさが見えてきます。大事なことは CLAUDE.md に書く。会話には一時的な作業指示だけを流す。この使い分けが、context rot に対する最も根本的な予防策になります。

### 判断の分岐

Qiita の nogataka 氏による決定木は、ここまで述べてきた道具の使い分けを一枚の分岐図に整理したもので、実用性が高いものでした。会話が全部不要なら `/clear`。文脈は残したいが容量を確保したいなら `/compact`（保存指示付き）。特定地点以降だけ圧縮したいなら `/rewind` から「Summarize from here」。重い調査作業ならサブエージェントに委任。そもそもの消費量を減らしたいなら CLAUDE.md の簡潔化と不要な MCP の無効化。この分岐が頭に入っているだけで、場面ごとの判断が速くなります。

## おわりに

この記事で追いかけたのは、「CLAUDE.md が守られなくなる」という日常的な症状の裏側にある仕組みでした。コンテキストウィンドウは容量の問題ではなく、注意力の問題です。1Mトークンが使えるからといって1Mトークン分の品質が保証されるわけではなく、Anthropic 自身のベンチマークが 256K→1M で17ポイントの精度低下を示しています。

わたし自身のワークフローで一番変わったのは、/compact を打つ回数が減り、/clear を打つ回数が増えたことです。以前は長いセッションを圧縮しながら引き延ばしていましたが、今は論理的な区切り（コミット完了、タスク切替）のたびに状態をファイルに書き出して /clear します。不便に感じたのは最初の数日だけで、新鮮なコンテキストで再開したときの Claude の応答精度の違いに気づいてからは、むしろ積極的にリセットするようになりました。

コンテキスト管理は、Claude Code の機能を覚えることではなく、有限の資源をどう設計するかという規律の話です。その規律の入り口として、まずは `/context` を一日に一度打つところから始めてみてください。自分のセッションがいま何%の地点にいるのかを知ることが、すべての判断の起点になります。
