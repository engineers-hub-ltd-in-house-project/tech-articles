# Claude Code 実戦入門

Claude Code を入れてはみたものの、結局自分で全部考えている気がする。
そう感じている人に向けて、この記事を書きます。

ゴールはひとつだけ置きます。読み終えたとき、あなたのプロジェクトに何かひとつ動くものが残っていることです。読んだだけでは何も変わりません。読みながら、開いた手元のエディタで小さな仕掛けを作り、その日のうちに走らせる。そういう体験を一緒にやります。

筆者の体験談は省きます。主語は常に「あなたのプロジェクト」です。

## 第1部　Claude Codeの設計思想

設計思想を先に渡します。Tips から入ると、覚えた小技が増えていくだけで全体像が見えてきません。逆に、なぜそういう作りになっているかを先に押さえておくと、後の Command や Skill や Agent の話が、ばらばらの機能の集合ではなくひとつの体系として浮かび上がります。

### 1-1　Claude Codeはなぜ「エディタのプラグイン」ではないのか

GitHub Copilot や Cursor を毎日使っている人ほど、最初の戸惑いがあるはずです。Claude Code は IDE の補完を提供しません。CLI を立ち上げて自然言語でやり取りをします。なぜそういう作りなのか。

補完系のツールは、いま開いているファイルのカーソル位置を中心にした半径数百行のコンテキストで動きます。これは便利ですが、できることの上限が「カーソル付近を埋める」に張り付きます。プロジェクト全体を見渡したり、複数ファイルを横断したり、ターミナルで何かを実行してその結果を踏まえて次の手を打ったり、ということが苦手です。

Claude Code はその逆を行きます。エディタの中ではなくシェルの中にいて、ファイルの読み書き、コマンドの実行、Web の取得、git の操作、テスト実行を自分で行えます。コードを「書く」のではなく、タスクを「進める」存在として作られています。

この発想の差から、いくつかの設計上の帰結が出てきます。

まず、コンテキストの持ち方が違います。補完系ツールはステートレスに近く、毎回その場のコードを読んで提案を返します。Claude Code はセッションを持ち、何を読んだか、何を編集したか、何を実行したかを覚えています。これにより「先ほど見たあのファイルの方針に合わせて」と頼める一方で、コンテキストが汚れていくと判断の質が落ちるという副作用も生まれます。コンテキストは資産であると同時に管理対象です。1-3 で詳しく扱います。

次に、IDE に依存しないことのメリットです。エディタの拡張機能として作られていれば、その IDE にいるあいだだけ使えます。CLI ベースなら、SSH 越しのサーバ、CI のジョブ、Docker コンテナの中、tmux の別ペイン、どこにでも連れていけます。プロジェクトに `.claude/` ディレクトリが置いてあるかぎり、誰がどの環境で起動しても同じ Claude Code が立ち上がります。これは後で見るチーム運用の話に効いてきます。

そして、Claude を「呼び出して使う」という感覚から、「Claude が自律的に進めるのを設計する」という感覚への移行が起きます。Copilot は補完候補を出してくれる便利な道具です。Claude Code は、目的を伝えると自分で計画を立て、必要な情報を集め、ファイルを書き換え、テストを回し、終わったら結果を報告する協働者です。後者の使い方をするには、こちらが「何をしてほしいか」だけでなく「どこまで任せるか、どこで止まってもらうか」を考える必要があります。

ここが Claude Code の最初の壁です。任せ方を考えるという発想が抜けたまま使い始めると、手元の Copilot のような期待値で接してしまい、Claude Code の良さが出ません。逆に、任せ方を設計するという発想を持って入っていけば、CLAUDE.md も Skill も Sub-agent も Hook も、全部「任せ方を細かく決めるための道具」として一直線に見えてきます。

### 1-2　CLAUDE.mdという設計思想

`CLAUDE.md` は Claude Code がプロジェクトを起動するときに自動で読み込むファイルです。多くの解説では「ルールファイル」や「プロジェクト規約を書く場所」と説明されます。それは間違いではないのですが、その捉え方だと書き方を見失いやすいので、別の比喩を提案します。

`CLAUDE.md` は、新しくチームに入ったメンバーが最初に読む「環境設定ガイド」です。

このプロジェクトはどう動かすのか。テストはどう走らせるのか。デプロイ前に何を確認するのか。何をしてはいけないのか。新しい人が一日目に知っておく必要があることを、過不足なく書く場所だと思ってください。Claude Code もまた「新しいチームメンバー」なので、同じものを読みます。

この比喩には自然な制約がひとつついてきます。新しい人が一日目に読むファイルは、長すぎてはいけません。具体的には、200 行を超えると Claude が読み流すようになります。これは公式のメモリドキュメントでも示唆されている目安で、ファイルが膨れていくほど一行あたりの重みが薄まり、結果として書いてあるはずの規約が守られなくなります。

200 行を超えてきたときの対処法は分割です。Claude Code は `.claude/rules/` 配下の Markdown ファイルを追加のメモリとして読み込めます。グローバルな規約は `CLAUDE.md` に、ドメインや言語ごとの細則は `.claude/rules/python.md` や `.claude/rules/database.md` のように分けます。`paths` フロントマターを使えば、特定のディレクトリの作業をしているときだけ読み込ませる、といった絞り込みも可能です。

もうひとつ、`CLAUDE.md` の良し悪しを判断する具体的なテストを置いておきます。新しいメンバーが Claude Code を起動して「テストを走らせて」と頼んだら、追加の質問なしで一発で正しいコマンドが走るか。これが通らないなら、`CLAUDE.md` にビルドコマンドとテストコマンドが書かれていないか、書いてあっても優先度が分かる形になっていません。

書くべきことの優先度は次の通り考えると整理がつきます。

- 動かし方（依存のインストール、ビルド、テスト、ローカル起動）
- やってはいけないこと（消してはいけないファイル、触ってはいけない外部リソース、本番に直接デプロイしないなど）
- 言語やフレームワークの選択にまつわる固有の癖（モノレポの構造、Python のバージョン、Node のバージョン）
- レビュー観点や PR の作り方（ブランチ戦略、コミットメッセージの規約）

逆に、書かないほうがいいものもあります。コードを読めば分かること、git log を見れば分かること、当たり前の Python の書き方。これらを書いてしまうとファイルが膨らみ、本当に大事なことの密度が落ちます。

実践課題

いま開いているプロジェクトに `CLAUDE.md` がなければ、最小構成で作ってみます。最小とは、ビルドコマンド、テストコマンド、禁止事項三つです。それだけで構いません。すでに `CLAUDE.md` がある場合は行数を数えてみて、200 行を超えていたら、`.claude/rules/` への分割を検討してください。

### 1-3　コンテキストという資源の使い方

Claude Code を使い込むほど、ある日「最初は冴えていたのに途中から判断がおかしくなってきた」という経験をします。その正体の多くはコンテキストの汚染です。

セッションを進めると、Claude が読んだファイル、実行したコマンドの出力、エラーメッセージ、検討して却下した選択肢、すべてがコンテキストに積み上がっていきます。これは便利ですが、一定量を超えると、新しい依頼に対して関係のない情報まで参照してしまい、過去の試行錯誤を引きずったまま判断を下すようになります。

この感覚を持つことが、Claude Code を「手放せない道具」にする第一歩です。コンテキストは無限ではなく、有限の資源です。資源は計測し、管理し、節約する必要があります。

道具は三つ用意されています。

`/context` を打つと、いま自分のセッションがどれだけのトークンを使っているかが表示されます。可視化できないものは管理できません。一日に一度はこれを見る癖をつけてください。

`/compact` は、これまでの会話を要約して圧縮します。完全に消すのではなく「あの話があったね」程度の解像度に落として、新しい会話の余地を作ります。コンテキストの使用量が 50% を超えたあたりで手動実行するのが目安です。自動圧縮もありますが、自分のタイミングで切れるほうが、要約のされ方を予測しやすくなります。

`/clear` は、コンテキストを完全に消去して新しいセッションを始めます。長い試行錯誤の末に、もう過去の文脈を引きずりたくないとき、別のタスクに切り替えるとき、これを使います。`/compact` との違いは、過去のサマリすら残さないことです。

ここで多くの人がつまずく感覚があります。`/clear` してしまうと、それまで Claude が学んだプロジェクトの暗黙知も全部消えるのではないか、という恐れです。実はそれが、`CLAUDE.md` や Skill が必要な理由です。コンテキストが消えても消えない場所に、プロジェクトの知識を移しておく。コンテキストは流れる水で、`CLAUDE.md` はダムです。流れている水を増やすのではなく、流れに頼らなくていい貯水池を作ることが、長い目で見れば一番楽になります。

実践課題

いま動いているセッションで `/context` を打ってみてください。何パーセント使っているかを確認します。50% を超えていたら、いったんこの記事を脇に置いて `/compact` を実行し、何が要約されたかを見ます。要約されてから判断の質が上がる経験をしておくと、以降のセッションで自然と `/compact` を切れるようになります。

ここまでが第1部です。Claude Code が自律的に動くものとして設計されていること、CLAUDE.md は規約ではなく「新メンバー向け環境設定ガイド」であること、コンテキストは管理対象の資源であること。この三つが押さえられていれば、第2部の道具立ての話が、機能の羅列ではなく目的のある体系として読めるはずです。

## 第2部　Command / Skill / Agent ─ 3つの道具と使い分け

第2部からが、自分のプロジェクトに仕掛けを作る話の中心です。混乱が最も多い部分なので、最初にざっくりした地図を渡します。

### 2-1　3つの道具の違いを一言で

Claude Code を拡張する道具は、大きく三つに分かれます。先に対比を出します。

```text
Command  既存のコンテキストに知識や手順を注入する。新しいコンテキストは生まれない。
Skill    Command と同じく知識を注入する。加えて、隔離された別コンテキストでの実行もできる。
Agent    最初から独立した新しいコンテキストで動く。メインに結果のサマリだけが返る。
```

ここで一点、現在の Claude Code 公式ドキュメント上の事情を補足しておきます。かつて Custom Command と呼ばれていたものは、いまは Skill の一形態として整理されています。`.claude/commands/<name>.md` というファイルと `.claude/skills/<name>/SKILL.md` というファイルは、どちらも `/<name>` というスラッシュコマンドを生成します。ファイル単体の単純な形が前者、ディレクトリと付随ファイルを持てる強化版が後者です。この記事では混乱を避けるため、ファイル一枚で済むものを「Command 形式」、ディレクトリ構造を持つものを「Skill 形式」と呼びます。仕組みとしては同じ系統の道具です。

判断基準を、用途で並べておきます。

- 毎日の繰り返し操作で、引数を渡して即実行したい → Command 形式
- 手順が長く、補助スクリプトや参考ファイルも一緒に配りたい、あるいは Claude が自動的に呼び出してくれてほしい → Skill 形式
- 長い処理や重い処理を、メインのコンテキストを汚さずに別空間で走らせたい → Sub-agent
- 一度しかやらない単発の依頼 → 何も使わず素のチャットで頼む

ここがピンと来れば、もう半分は越えています。第2部の残りでは、それぞれの道具を実際に書いて動かす作法を見ていきます。

### 2-2　Commandを自分で書いてみる

最初の Command を書きます。Skill 形式の機能をすべて使うとややこしいので、ファイル一枚で完結する形から入ります。

`.claude/commands/pr-review.md` というファイルを作ります。中身はこうです。

```yaml
---
description: 指定した PR の差分を取り込んでレビューする
argument-hint: [PR番号]
allowed-tools: Bash(gh pr *)
---

GitHub PR #$ARGUMENTS のレビューを行います。

## 取得する情報

- PRの差分: !`gh pr diff $ARGUMENTS`
- PRの説明: !`gh pr view $ARGUMENTS`
- 変更ファイル一覧: !`gh pr diff $ARGUMENTS --name-only`

## レビュー観点

1. ロジックの正当性
2. エッジケースの取りこぼし
3. テストの過不足
4. このプロジェクトの既存パターンとの整合

レビュー結果は、ファイルパスと行番号を添えて指摘ごとに分けて報告してください。
```

これだけです。`/pr-review 123` と打てば、Claude Code が以下の手順を自動で踏みます。まず Bash でバッククォート付きで囲まれた `gh pr diff 123` などを実行します。次に出力結果がファイル本文の対応箇所に埋め込まれます。最後に、すでに差分情報を抱えた状態のプロンプトとして Claude に渡されます。

ここで効いている仕掛けを分解します。

`description` フィールドは、ユーザーが `/` を叩いたときに表示される説明文です。argument-hint は引数の見本です。`/pr-review` まで打つと「[PR番号]」が表示され、何を渡せばいいかが分かります。`allowed-tools` は、この Command が呼ばれているあいだだけ追加で許可されるツールの指定です。`Bash(gh pr *)` という書き方で、`gh pr` で始まる Bash 実行だけを事前承認しています。これがないと、Bash の都度確認ダイアログが出ます。

`$ARGUMENTS` は、`/pr-review 123` の `123` 部分が展開されるプレースホルダです。複数の引数を渡したい場合は `$0`、`$1` で位置指定もできます。`/migrate-component SearchBar React Vue` と書いたとき、Skill 内では `$0` が `SearchBar`、`$1` が `React`、`$2` が `Vue` に展開されます。

そしてこの Command の真価は、本文中の `` !`gh pr diff $ARGUMENTS` `` の部分です。バッククォートの直前に `!` を置くと、その中身は Claude にプロンプトとして送られる前に、シェルで実行されてしまいます。実行結果が本文中の該当箇所に差し込まれた状態で Claude に渡されます。Claude から見れば、自分でツールを呼んだのではなく、最初から本文に PR の差分が書かれていたかのように見えるわけです。

この前処理パターンを覚えると、Command の表現力が一気に広がります。git のログ、ファイルの一覧、テストの直近の結果、何でも先に取得して本文に差し込めます。情報を取りに行くのは Claude の仕事ではなく、Command の役目になります。

実践課題

毎日やっている操作を一つ思い浮かべてください。作業ログの記録、定型フォーマットでの報告、依存関係の確認、何でもかまいません。それを `.claude/commands/<名前>.md` に変換します。引数を取る形にすること、`!` を使って動的な情報を埋め込むこと、この二つを満たせば Command として一人前です。最初の Command は完璧を目指さないで構いません。動いたら使ってみて、不満を感じた場所だけ後で直せば十分です。

### 2-3　Skillの設計

Command 形式は、ファイル一枚で動作する手軽さが魅力です。ただし、対象とする操作が大きくなるにつれて、こう感じる場面が出てきます。スクリプトを一緒に置きたい。長い参考資料を別ファイルに切り出したい。Claude が自動的に呼び出してくれるようにしたい。これらをサポートするのが Skill 形式です。

Skill は `.claude/skills/<名前>/SKILL.md` というディレクトリ構造を取ります。SKILL.md がエントリーポイントで、同じディレクトリの中に補助ファイル、サンプル、スクリプトを自由に置けます。

最小の Skill はこんな見た目です。

```yaml
---
name: api-conventions
description: このコードベースの API 設計規約。新しいエンドポイントを書くとき、既存の API の意図を確認したいときに使う。
---
API エンドポイントを書くときは、次の規約に従います。

1. RESTful な命名（複数形のリソース名）
2. エラーは一貫した JSON 形式で返す（code, message, details の三層）
3. リクエストバリデーションは Pydantic のモデルで宣言する

エンドポイントを追加するときは、既存の `app/api/` 配下のパターンを必ず一つ参考にしてから書き始めてください。
```

`description` の書き方が、Skill の品質を決めます。ここを軽視すると、Claude が「いつ自分を呼ぶべきか」を判断できず、せっかく作った Skill が眠ってしまいます。良い description は条件を含みます。

「コードをレビューする Skill」ではなく「Pull Request の差分が渡されたとき、セキュリティと品質の観点でレビューを行う Skill」と書きます。前者は Claude にとってあいまいすぎて発火タイミングが判断できません。後者なら「PR の差分が話題に出たら呼ぶ」という条件が読み取れます。description は 250 文字を超えると Skill 一覧で切り詰められるので、最初の 250 文字に最も使ってほしい条件を入れることも忘れないでください。

Skill のフロントマターには、覚えておくと便利なフィールドがいくつかあります。

`disable-model-invocation: true` を設定すると、ユーザーが手動で `/<name>` と打ったときだけ起動します。`/deploy` や `/commit` のような副作用の大きいものに使います。Claude が「コードができたから自動でデプロイしておいた」となるのを防ぎます。

`user-invocable: false` は逆で、ユーザーの `/` メニューには出さず、Claude が必要なときに自分で呼ぶ背景知識として使います。たとえば「このプロジェクトの旧システム連携の歴史的経緯」のような、ユーザーが叩く意味のない知識集を Claude に持たせたいときに使います。

`allowed-tools` は、その Skill が動いているあいだだけ事前承認するツールの指定です。Command の例で見た `Bash(gh pr *)` のような書き方で、ツールを絞り込みます。

`context: fork` を指定すると、その Skill は新しい隔離コンテキストで実行されます。詳しくは 2-5 のオーケストレーションのところで扱います。

`paths` には glob を渡せます。たとえば `paths: src/api/**/*.py` と書いておくと、`src/api/` 配下の Python ファイルを触っている文脈でだけ Claude が自動で読み込むようになります。広く使う Skill より、狭く使う Skill のほうが結果として活きます。

書くときに気をつけたいことが二つあります。

ひとつは、当たり前のことを書かないことです。「変数名は意味のあるものを使うこと」「テストを書きましょう」のような一般論は、Claude がもともと知っていることです。書けば書くほどコンテキストを消費するだけで、行動は変わりません。書くべきは、このプロジェクト固有の判断、過去に踏んだ地雷、社内の特殊な用語、外部からは分からない制約です。

もうひとつは、手順を細かく規定しすぎないことです。「まず A をして、次に B をして、その次に C をして」と全部書いてしまうと、Claude は判断する余地を失います。書くべきは、ゴールと制約と境界条件です。「最終的にこの状態にしてほしい、ただし X だけはやらないこと」と渡せば、Claude が自分で道筋を考えます。Claude を手続き的に縛るのではなく、宣言的に方向づけるのが、結果として一番安定します。

実践課題

「Claude に何度も同じ説明をしている気がする」と感じる知識を一つ思い出してください。プロジェクト固有の何かです。それを Skill にします。description には「いつこの Skill を使うか」を必ず書きます。書き終えたら、その条件に合致するような依頼を一回出してみて、Claude が自動で呼んでくるかを確認します。呼んでこなかったら、description の条件部分の表現を見直します。

### 2-4　Sub-agentのフロントマター全解説

Sub-agent は、独立した別コンテキストで動く Claude です。メインの会話とは別の Window で動き、終わるとサマリだけがメインに返ります。これにより、長大な検索結果や試行錯誤のログでメインを汚染することなく、重い作業を任せられます。

Sub-agent のファイルは `.claude/agents/<名前>.md` に置きます。最小の例はこうです。

```yaml
---
name: code-reviewer
description: 書かれたばかりのコードに対して、品質とベストプラクティスの観点でレビューを行う。実装が一段落したタイミングで使う。
tools: Read, Glob, Grep
model: sonnet
---
あなたはコードレビュアーです。渡されたコードを読み、品質、セキュリティ、ベストプラクティスの観点で具体的かつ実行可能なフィードバックを返してください。

レビューはファイルパスと行番号を添えた指摘の形式で出力します。
```

`name` と `description` だけが必須です。description は、Claude がいつこの Sub-agent に委譲すべきかを判断するために読む文章なので、Skill と同じく条件を明確に書きます。

ここから、覚えておくと使い勝手が変わるフィールドを順に紹介します。

`tools` は、その Sub-agent が使えるツールの許可リストです。指定しなければメインの会話と同じツールセットを継承します。レビュー専用にしたいなら `Read, Glob, Grep` だけにすれば、誤ってファイルを書き換えてしまう事故を構造的に防げます。許可を絞ることはセキュリティであり、責任分界の表明でもあります。

`disallowedTools` は逆で、特定のツールだけを禁止する形のブラックリストです。Write と Edit だけ禁止して他は全部使わせたい、というときに使います。`tools` と `disallowedTools` の両方を指定した場合は、まず disallowed が引かれてから allowed が解決されます。

`model` には `sonnet` `opus` `haiku` のエイリアス、または `claude-haiku-4-5` のような完全なモデル ID、あるいは `inherit` を指定できます。重い思考を要する Sub-agent には Opus を、定型的で速度が大事な Sub-agent には Haiku を、と使い分けるとコストにも効きます。読み取り専用の探索 Sub-agent なら Haiku で十分、というのが目安です。

`permissionMode` は、その Sub-agent の権限挙動を切り替えます。値は `default` `acceptEdits` `auto` `dontAsk` `bypassPermissions` `plan` の六つです。普段は `default` で構いません。`acceptEdits` を選ぶと、ファイル編集や標準的なファイルシステムコマンドを自動承認するようになり、毎回ダイアログが出るのを抑えられます。`bypassPermissions` は権限プロンプトを完全にスキップする強力な設定で、信頼できる読み取り専用 Sub-agent や、ワークツリーで隔離されている前提の Sub-agent にだけ使うべきです。

`maxTurns` は、その Sub-agent が止まるまでに踏める最大ターン数の上限です。暴走対策として効きます。読み取り専用のレビュアーなら 5、複雑な実装系でも 30 ぐらいに収めておくと、無限ループや延々と試行錯誤を続ける状態に陥ったときに自分で止まってくれます。

`skills` は、その Sub-agent の起動時にプリロードする Skill の指定です。通常のセッションでは Skill の本文は呼び出されたときだけ読み込まれますが、Sub-agent ではここに指定した Skill の中身が起動時にまるごと注入されます。デプロイ手順や API の規約のように「この Sub-agent は常にこの知識を持っていてほしい」というものを指定します。

`mcpServers` には、その Sub-agent から使える MCP サーバを列挙できます。すでに設定済みのサーバ名を渡してもいいですし、その場でインライン定義を書くこともできます。Slack 通知 Sub-agent には Slack の MCP だけを渡す、というように責務に応じて MCP の露出を絞れます。

`hooks` は、その Sub-agent のライフサイクルにスコープしたフック定義です。たとえば Sub-agent の Stop イベントだけに反応するフックを書けば、その Sub-agent が終わったときだけ通知スクリプトを叩く、ということができます。Sub-agent では Stop は実行時に SubagentStop に自動変換されます。第3部で詳しく扱います。

`memory` を指定すると、その Sub-agent は実行のたびに永続的なメモリを持つようになります。`user` を指定すると `~/.claude/agent-memory/<agent-name>/` に保存されてユーザー全プロジェクトで共有されます。`project` だと `.claude/agent-memory/<agent-name>/` 配下に保存され、リポジトリにコミットしてチームで共有できます。`local` はプロジェクト固有だがコミットしない位置に保存されます。Sub-agent が前回の作業を覚えていてほしい場合のための仕組みです。

`background: true` を指定すると、その Sub-agent は常にバックグラウンドジョブとして実行されます。長時間動くドキュメント生成や、待っているあいだに別の作業を進めたい用途に向きます。

`isolation: worktree` は、Sub-agent の作業を一時的な git worktree の中で行わせます。リポジトリの隔離コピーで動くので、もし Sub-agent が変な変更を入れても、メインの作業ツリーには影響しません。Sub-agent が何も変更しなかった場合は、ワークツリーは自動でクリーンアップされます。実装を試させたい、けれどメインのブランチを汚したくない、という場面で活きます。

`color` は、CLI 上で Sub-agent の存在を視認するための表示色です。`red` `blue` `green` `yellow` `purple` `orange` `pink` `cyan` から選びます。Sub-agent が複数並走しているとき、どれがどれかを一瞬で見分けるためのささやかな配慮です。

`initialPrompt` は、その Sub-agent が `--agent` フラグでメインのセッションエージェントとして起動されたときに、自動で送信される最初のユーザーメッセージです。コマンドや Skill の参照も処理されます。特定の起動シナリオを定型化したいときに使います。

組み込み Sub-agent

ここまで自分で書く Sub-agent の話をしてきましたが、Claude Code には最初から用意されている組み込み Sub-agent があります。代表的なものを挙げておきます。

- Explore は Haiku で動く読み取り専用の高速な探索 Sub-agent です。ファイルの発見、コードの検索、コードベース全体の見通しを取る用途に向きます。Claude が「ちょっと調べたいけれどメインのコンテキストを汚したくない」と判断したとき、自動で Explore に委譲します。
- Plan は Plan モード中に呼ばれる調査用 Sub-agent です。読み取り専用のツールだけを持ち、計画立案のための情報収集を行います。
- General-purpose は、すべてのツールを持つ汎用 Sub-agent です。複雑な多段階タスク、調査と変更の両方が必要な作業を任せます。
- statusline-setup は `/statusline` を実行したときに呼ばれる、ステータスラインの設定を支援する Sonnet ベースの Sub-agent です。
- Claude Code Guide は、ユーザーが Claude Code 自体の使い方を質問したときに自動で呼ばれる Haiku ベースの Sub-agent です。

組み込み Sub-agent はそれぞれ意図が明確で、Claude が自分で適切なものを選んで使ってくれます。あなたが意識して呼び出す必要はありませんが、「Claude が何かを調べているときは Explore に丸投げしている」という前提を知っておくと、ログを見たときの理解が速くなります。

ひとつ注意点があります。Sub-agent は他の Sub-agent を spawn することはできません。無限ネストを防ぐための制約です。Sub-agent の中で Plan モード相当の調査をしたいときは、`isolation: worktree` などの仕組みを併用しつつ、Sub-agent 自身が直接ツールを使う形になります。

実践課題

組み込み Sub-agent を一度確認してみてください。Claude に「何か大きめの調査タスク」を渡すと、ログのどこかに Explore が呼ばれた痕跡が残るはずです。それを観察してから、自分の最初のカスタム Sub-agent を作ります。題材は「Read だけを許可された PR レビュアー」がおすすめです。`tools: Read, Glob, Grep` と `model: haiku` と `maxTurns: 10` の三つを設定すれば、安全で軽い読み取り専用 Sub-agent ができます。

### 2-5　Command → Sub-agent → Skill のオーケストレーション

ここまで Command、Skill、Sub-agent をそれぞれ単体で見てきました。実戦では、これらを組み合わせると初めて Claude Code が「手放せない道具」になります。組み合わせ方の典型を一つ示します。

```text
[ユーザー] /review-pr 123
   ↓
[Command/Skill] gh pr diff 123 を ! で実行 → 差分を本文に埋め込む
   ↓
                 Sub-agent (code-reviewer) を呼ぶ
                  ↓
                  独立したコンテキストで code-reviewer が動く
                  ↑ skills フィールドで coding-conventions Skill がプリロード済み
                  ↓
                  レビュー結果のサマリだけがメインに返る
   ↓
[メインのClaude] 結果を受け取り、ユーザーに整形して提示
```

この流れの何が嬉しいか。Command の側で動的情報の取得が完了しているので、Sub-agent が呼ばれた時点で「何をレビューするか」のコンテキストが整っています。Sub-agent は独立した Window で動くため、長い差分や試行錯誤のログがメインを汚しません。プリロードされた Skill によって、Sub-agent はチームのコーディング規約を最初から知った状態で動きます。

これは設計の話に聞こえるかもしれませんが、書くコード量はそんなに多くありません。Command 形式の Skill のなかから Sub-agent を呼ぶには、本文中で「次の差分について、code-reviewer Sub-agent に依頼してください」と指示するだけです。Claude Code が Agent ツール（旧 Task ツール）を使って自動的に Sub-agent を spawn します。

`context: fork` を指定した Skill のパターンも便利です。

```yaml
---
name: deep-research
description: 与えられたテーマについて、コードベースを徹底的に調査して報告する
context: fork
agent: Explore
---
$ARGUMENTS について、コードベースを調査します。

1. 関連するファイルを Glob と Grep で発見
2. 重要なファイルを読み込み、構造を把握
3. 依存関係や使われ方を辿る
4. 最後に、ファイルパスと行番号付きの報告書を作る

報告は要点を絞り、リンクできる箇所にはすべて参照を添えてください。
```

`context: fork` と `agent: Explore` の二行を入れると、この Skill は呼ばれた瞬間に Explore Sub-agent の中で実行されます。Skill の本文がそのまま Sub-agent への最初のプロンプトになります。Skill 単体でメインを汚さない調査タスクが完成します。

実践課題

「Command が Sub-agent を呼ぶ」または「Skill が `context: fork` で隔離実行する」のどちらかを一つ作ってみてください。最初は完璧でなくて構いません。動いたら、メインのコンテキスト使用量を `/context` で確認します。直接同じ作業をメインでやった場合と比べて、コンテキストの汚染がどれだけ抑えられるかを見ると、隔離の価値が体感できます。

ここまでが第2部です。第3部では、Hook を使った自動化と運用の話に移ります。

## 第3部　Hooksと実用パターン

第2部までは Claude Code に何かを「依頼する」ための道具立てでした。第3部は逆向きです。Claude Code が何かをするたびに、自動で別の処理を差し込む道具を扱います。Hook です。

### 3-1　Hooksとは何か

Hook は、Claude Code のエージェントループの外側で動くイベントハンドラです。Claude が動作の節目に達するたびに、設定されたシェルコマンドが発火します。Claude 自身は Hook を「呼んでいる」つもりはなく、Hook はあくまで Claude の挙動を観測し、必要なら止めたり情報を返したりする外部の存在です。

これが何を可能にするか。Claude の判断には揺らぎがありますが、Hook の挙動は決定的です。「コードを書いたら必ずフォーマッタを走らせる」「危険なコマンドを実行しようとしたら必ず止める」「ターンが終わったら必ずチェックを通す」というような、確実に保証したい動作を Claude の意思決定の外側に置けます。

主な発火タイミング

Hook が反応できるイベントはたくさんあります。代表的なものを挙げます。

- `PreToolUse` ─ Claude がツール（Bash, Edit, Write など）を実行する直前
- `PostToolUse` ─ ツールが成功して結果が返った直後
- `Stop` ─ Claude のターンが終わる時点（「もう返すことはありません」と言った直後）
- `UserPromptSubmit` ─ ユーザーが何か入力した直後、Claude が処理を始める前
- `SessionStart` ─ 新しいセッションを開いた瞬間
- `SessionEnd` ─ セッションを終了する瞬間
- `PreCompact` / `PostCompact` ─ コンテキストの自動圧縮の前後
- `SubagentStart` / `SubagentStop` ─ Sub-agent の起動と終了
- `FileChanged` ─ 監視対象のファイルが外部から変更されたとき
- `Notification` ─ Claude Code が通知を送るとき

このうち、まず使いやすいのは `PreToolUse` `PostToolUse` `Stop` の三つです。それぞれの典型的な活用例を、次の節で具体的に見ていきます。

設定の基本形

Hook は `.claude/settings.json` に書きます。最小例はこれです。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/validate-bash.sh"
          }
        ]
      }
    ]
  }
}
```

`matcher` でツール名や条件を絞り、該当した場合に `command` が呼ばれます。コマンドの標準入力には、その時点のイベント情報が JSON で渡されます。コマンドの終了コードや標準出力で、Claude の挙動を変えることができます。たとえば終了コードを 2 にして標準エラーにメッセージを書くと、Claude にツール実行を中止させ、エラーメッセージを Claude のコンテキストに戻すことができます。

Hook はシェルコマンドなので、好きな言語で書けます。bash でも Python でも Node でもかまいません。重要なのは、Hook が短時間で終わることです。Claude のターンの応答性に直結するので、長時間の処理は Hook の中ではなく、Hook が起動するバックグラウンドジョブの方に押し出してください。

### 3-2　実用Hookパターン3つ

実戦で効くパターンを三つ紹介します。

パターン1: 自動フォーマット（PostToolUse）

Claude が書いたコードを、書いた直後にフォーマットします。CI のフォーマットチェックで落ちる「最後の 10%」を Hook 側で吸収できます。

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/format-after-write.sh"
          }
        ]
      }
    ]
  }
}
```

`format-after-write.sh` の中身は、ファイルパスを受け取って拡張子別にフォーマッタを呼び出す程度の単純なスクリプトで構いません。Python なら ruff、TypeScript なら biome、Markdown なら prettier、というように振り分けます。Claude がフォーマットを忘れても、Hook が必ず後始末してくれます。

このパターンで一番嬉しいのは、Claude にフォーマットの細則を覚えさせる必要がなくなることです。「インデントは 2 か 4 か」「セミコロンを付けるか付けないか」のような細かい好みは、CLAUDE.md に書くと書くだけ濃度が薄まります。Hook で機械的に整えてしまえば、CLAUDE.md の本文は本質的な規約だけに集中できます。

パターン2: 危険な操作のガード（PreToolUse）

Claude がツールを実行する直前に Hook を挟んで、危険な操作を止めます。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/block-dangerous-bash.sh"
          }
        ]
      }
    ]
  }
}
```

`block-dangerous-bash.sh` の中で、標準入力に渡された JSON を読み、コマンド文字列に `rm -rf /`、`git reset --hard`、`drop database`、`force push` などの危険語が含まれていたら、終了コード 2 で標準エラーに「禁止された操作です」と書いて落とします。Claude はこのエラーをコンテキストに受け取り、別の手段を考え直します。

このパターンは、Claude を信頼するか信頼しないかという二択ではなく、信頼を多層化する考え方です。Claude はほとんどの場合まともに振る舞いますが、稀にミスをします。そのミスがプロジェクトに致命的な影響を及ぼす操作であれば、確率がいくら低くても発生してはならない。だから、Claude の判断とは独立した防衛線を引きます。Hook はその防衛線として機能します。

危険語のリストはプロジェクトごとに違います。本番 DB に触れるコマンド、特定の S3 バケットへの書き込み、特定のファイルの削除など、自分のプロジェクトで「これだけは絶対に意図せず実行されてはまずい」というものを列挙してください。

パターン3: ターン終了時のチェック（Stop）

Claude が「終わりました」と返したタイミングで、本当に終わっているかを検証します。

```json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/post-turn-checks.sh"
          }
        ]
      }
    ]
  }
}
```

`post-turn-checks.sh` の中では、たとえば次のようなことを行います。git status を見て中途半端なステージング状態がないか確認する、テストが走る状態か軽くチェックする、TODO コメントが新しく増えていたら警告する、Lint エラーがないか確認する。何か問題があれば終了コード 2 で標準エラーに状況を書きます。Claude はその情報を読んでもう一度ターンを回し、本当に「終わった状態」にしようとします。

このパターンの効きどころは、Claude が「完了した」と宣言する精度のばらつきを吸収できることです。Claude Code は基本的に正直ですが、時々「動くと思います」と言ってまだ動かない状態を完了扱いにしてしまいます。Stop Hook は、その種の油断を機械的に検出します。

three の Hook をまとめて運用すると、Claude Code に対する信頼が一段上がります。書いたコードはフォーマットが整っていて、危険な操作はブロックされ、終わった宣言には根拠がある。Claude を制約していると言うより、自分が安心して任せるための環境を整えていると考えるほうが正しい捉え方です。

### 3-3　SkillとSub-agentにHookを埋め込む

Hook は `.claude/settings.json` に書くだけでなく、Skill や Sub-agent のフロントマターに直接埋め込むこともできます。これが地味に便利です。

```yaml
---
name: deploy
description: 本番環境にデプロイする
disable-model-invocation: true
allowed-tools: Bash(./scripts/deploy.sh)
hooks:
  Stop:
    - hooks:
        - type: command
          command: "./scripts/notify-slack-deploy-complete.sh"
---
本番にデプロイします。

1. テストを走らせる
2. ビルドする
3. デプロイスクリプトを実行する
4. 動作確認する
```

この `deploy` Skill が呼び出されてターンが終わった瞬間にだけ、Slack 通知のスクリプトが走ります。グローバルな Stop Hook で全ターンに通知を出すのではなく、デプロイ Skill が走ったときだけ通知する、という局所的なフックが書けます。

Sub-agent 側でも同じことができます。Sub-agent のフロントマターに `hooks` を書いておけば、その Sub-agent のライフサイクル中だけ発火する Hook を定義できます。Sub-agent では Stop は実行時に SubagentStop に自動変換されます。Sub-agent が終わったら親に通知する、レビュー Sub-agent が完了したら Markdown レポートを保存する、といった用途で使えます。

Hook の責務を局所化すると、グローバル設定が膨らむのを防げます。「この処理はこの Skill のためのもの」と分かる場所に書いてあれば、半年後に読んだ自分が混乱しません。

### 3-4　Hooksでスキル使用状況を計測する

最後に、運用に役立つ Hook の使い方をひとつ紹介します。Hook を使うと、自分が作った Skill や Command が実際にどれぐらい使われているかを計測できます。

`PreToolUse` Hook を仕込んでおき、Skill ツールが呼ばれた瞬間にどの Skill が呼ばれたかをログファイルに追記します。一週間続けると、自分のローカルで一番使われている Skill と、まったく使われていない Skill が浮き彫りになります。

使われていない Skill は二種類に分かれます。description が悪くて Claude が選ばないもの、本当に必要なかったもの。前者は description を改善する余地があり、後者は削除すべきです。よく使われている Skill は、もっと充実させる価値があります。これを計測なしで判断するのは難しく、感覚で「使ってる気がする」と言うとだいたい外れます。

具体的なログ取りのスクリプトはここでは省きますが、要点は、ログを取ること自体ではなく、定期的に振り返って Skill の棚卸しをすることです。Claude Code の運用は、書き捨てではなく、育てるものです。Hook はその育成のための観測手段になります。

ここまでが第3部です。第4部では、これまで見てきた道具立てを実際に自分のプロジェクトに根付かせる話に移ります。

## 第4部　自分のプロジェクトに根付かせる

第3部までで道具は揃いました。第4部は道具を組み合わせて「日々使い続けられる仕組み」にする話です。

### 4-1　何もない状態から始める最小構成

Claude Code を入れた直後の人がやるべきことを、一週間に分解してみます。

一日目は CLAUDE.md を作ります。書く内容は三つだけです。ビルドコマンド、テストコマンド、禁止事項三つ。それ以上は書きません。長い CLAUDE.md は最初から書こうとせず、必要を感じたときに足していきます。

二日目は、毎日やっている操作を一つ Command 形式にします。題材は何でも構いません。テストの実行、PR の作成、ログの取得、デプロイ前のチェック。引数を取る形にして、`!` で動的情報を埋め込む、これだけ守れば十分です。

三日目はその Command を実際に使います。一日使ってみて、不満を感じた点をメモします。引数の渡し方が分かりにくい、出力が冗長、もっと事前情報が要る、何でも構いません。

四日目と五日目は、Command の中で繰り返している説明を Skill 形式に分離します。Command の本文が膨らんできたら、その一部は「知識」であって「手順」ではありません。知識は Skill に切り出します。Skill にすれば、別の Command や Sub-agent からも参照できる資産になります。

六日目と七日目は、重い処理を Sub-agent に分離してみます。長い検索や試行錯誤を伴う作業を一つ選び、`tools` を絞った読み取り専用の Sub-agent にしてみます。メインのコンテキスト使用量がどう変わるかを `/context` で観察します。

この一週間で、CLAUDE.md と Command と Skill と Sub-agent をすべて触ったことになります。ここからは、自分の業務の中で「またこれをやっている」と思った瞬間に、Command か Skill か Sub-agent かを判断しながら積み上げていきます。

### 4-2　Commandのレシピ集

ここからは、すぐコピーして手を入れて使えるレシピを並べます。題材は十個。それぞれフロントマターと本文の骨格を示します。

レシピ1: PRレビュー依頼

```yaml
---
description: 指定した PR の差分を取り込んでレビューする
argument-hint: [PR番号]
allowed-tools: Bash(gh pr *)
---

PR #$ARGUMENTS のレビューです。

- 差分: !`gh pr diff $ARGUMENTS`
- 説明: !`gh pr view $ARGUMENTS`

ロジック、エッジケース、テストの過不足、既存パターンとの整合の四点で見て、ファイルパスと行番号付きで報告してください。
```

allowed-tools で `gh pr` 以下に絞っていることがポイントです。`gh repo delete` のような破壊的操作までは許可されません。

レシピ2: Issue修正

```yaml
---
description: 指定した issue を読み、修正計画を立てる
argument-hint: [issue番号]
allowed-tools: Bash(gh issue *)
---

GitHub Issue #$ARGUMENTS の対応計画を立てます。

- Issue 本文: !`gh issue view $ARGUMENTS`
- コメント: !`gh issue view $ARGUMENTS --comments`

まず再現条件を整理し、次に修正方針を三つ挙げ、最後に推奨案を選んで根拠を示してください。実装には進まず、計画だけで止めます。
```

「実装には進まず、計画だけで止める」という制約を明示的に書いているのがポイントです。Claude Code は親切なので、放っておくと勝手に実装まで進めようとします。

レシピ3: Changelog生成

```yaml
---
description: 直近のコミットから Changelog エントリを生成する
argument-hint: [タグ名]
allowed-tools: Bash(git log *)
---

リリース $ARGUMENTS のための Changelog エントリを作ります。

- 直近のタグからのコミット一覧: !`git log --oneline $(git describe --tags --abbrev=0)..HEAD`
- 詳細: !`git log $(git describe --tags --abbrev=0)..HEAD --format='%h %s%n%b'`

カテゴリ（追加・変更・修正・破壊的変更）に分類して、ユーザー目線で書き直します。コミットメッセージをそのまま使わず、何が嬉しいかを書いてください。
```

git log の出力をそのまま貼り付けるのではなく、Claude にユーザー目線で書き直させているところが、人手で書いた Changelog に近づける鍵です。

レシピ4: テスト失敗の分析

```yaml
---
description: 直近のテスト結果を取り込んで、失敗の原因を分析する
allowed-tools: Bash(pnpm test *)
---

テストを走らせて失敗を分析します。

- テスト出力: !`pnpm test 2>&1 | tail -200`

失敗したテストを一つずつ見て、原因をコードの根本に遡って説明してください。修正案がある場合は、最も小さく確実な変更を提案します。
```

`tail -200` で取り込む量を絞り、コンテキストの汚染を抑えています。テストの全出力を入れると、すぐに数千行になります。

レシピ5: デプロイ前チェック

```yaml
---
description: デプロイ前のチェックリストを実行する
allowed-tools: Bash(git status *) Bash(git log *) Bash(pnpm test) Bash(pnpm lint)
---

デプロイ前チェックです。

- 作業ツリーの状態: !`git status --short`
- main との差分: !`git log main..HEAD --oneline`
- テスト: !`pnpm test`
- Lint: !`pnpm lint`

各項目について、デプロイしてよいかをイエス・ノーで答え、ノーならその理由を最初に書いてください。
```

イエス・ノーの即答を最初に求めるのが、現場で使うときに効きます。Claude が長い説明から始めると、結論を読み逃しがちです。

レシピ6: コードリファクタリング提案

```yaml
---
description: 指定ファイルのリファクタリング案を出す
argument-hint: [ファイルパス]
allowed-tools: Read
---
ファイル `$ARGUMENTS` のリファクタリング案を出します。

ファイルを読み、関数の責務、命名、依存の方向、テスト容易性の四点で改善提案を出してください。提案は破壊的な大手術ではなく、独立して取り込める小さな変更の単位に分解してください。実装はせず、提案だけにとどめます。
```

allowed-tools を `Read` に絞ってあるので、誤って書き換えてしまう事故が起きません。

レシピ7: ドキュメント更新

```yaml
---
description: 直近のコミットに合わせて README を更新する
allowed-tools: Bash(git diff *) Read
---

README を最新の実装に合わせて更新します。

- 直近の差分: !`git diff main..HEAD`
- 現在の README: README.md を読んでください

新しく追加された機能、変更されたコマンド、消えた機能をすべて反映します。実際にない機能を書かないこと、すでにある機能を二重に書かないことを最優先にしてください。
```

「ないものを書かない」「二重に書かない」という制約は、ドキュメント生成系の Skill では必ず入れたい文言です。これがないと Claude は親切心で見栄えのいい嘘を書きがちです。

レシピ8: セキュリティスキャン

```yaml
---
description: 指定ディレクトリのセキュリティ観点レビュー
argument-hint: [ディレクトリパス]
allowed-tools: Read Glob Grep
---
ディレクトリ `$ARGUMENTS` のセキュリティ観点レビューです。

次の観点で見ます。
- 認証・認可の漏れ
- 入力バリデーションの欠落
- SQL インジェクションや XSS の可能性
- 機密情報のハードコード
- 依存ライブラリの既知の脆弱な使い方

検出した問題はファイルパスと行番号、深刻度、修正案の三点で報告してください。
```

レシピ9: パフォーマンス分析

```yaml
---
description: 指定ファイルのパフォーマンス観点レビュー
argument-hint: [ファイルパス]
allowed-tools: Read
---
ファイル `$ARGUMENTS` のパフォーマンス観点レビューです。

- N+1 クエリ
- 不要なループ
- 大きいデータの全件読み込み
- 同期的な I/O ブロッキング
- メモリリークになりうる参照保持

それぞれについて、検出したらまず計測の必要性を述べ、その上で改善案を提示してください。最初から最適化に走らないでください。
```

「最初から最適化に走らない」という制約が大事です。Claude は親切なので、見つけた問題を全部直そうとします。

レシピ10: 技術的負債の洗い出し

```yaml
---
description: 指定ディレクトリの技術的負債を洗い出す
argument-hint: [ディレクトリパス]
allowed-tools: Read Glob Grep
---
ディレクトリ `$ARGUMENTS` の技術的負債を洗い出します。

次のサインを探してください。
- TODO / FIXME / HACK コメント
- 長すぎる関数（80 行以上を目安）
- 深いネスト（4 段以上）
- 重複したロジック
- テストのない複雑な分岐

検出したものを「すぐ直すべき」「次のリファクタリング機会で」「許容範囲」の三段階で評価し、根拠と共に示してください。
```

allowed-tools を Read 系に絞り、評価と分類だけを求めています。Claude を「分析する人」として使うときの典型形です。

これらのレシピを並べてみて気づくのは、フロントマターの allowed-tools と本文の制約句の二つでほとんどの安全性と品質が決まる、ということです。Skill の本文にどれだけ凝っても、ツール許可が広すぎれば事故は起きるし、制約を書き忘れれば Claude は親切心で余計なことをします。

### 4-3　Skillのレシピ集

Command 形式より、ディレクトリと付随ファイルを持つ Skill 形式が向くケースを五つ挙げます。

レシピA: デプロイチェックリスト（Sub-agent プリロード用）

```yaml
---
name: deployment-checklist
description: 本番デプロイ前に必ず確認する事項。Sub-agent のプリロード用に user-invocable は false にする。
user-invocable: false
---
本番デプロイ前のチェックリストです。

1. main ブランチが最新か
2. ステージング環境で動作確認済みか
3. データベースのマイグレーションが必要なら、ロールバック手順が用意されているか
4. オンコール担当者に連絡済みか
5. 緊急時のロールバックコマンドを手元に控えているか
```

`user-invocable: false` にして、ユーザーの `/` メニューには出しません。デプロイ系の Sub-agent の `skills` フィールドにこれを指定すると、Sub-agent は起動時にこの内容を全部知った状態で動き始めます。

レシピB: コーディング規約（Sub-agent プリロード用）

```yaml
---
name: coding-conventions
description: このプロジェクトのコーディング規約。コードレビュー Sub-agent やコード生成 Sub-agent のプリロード用。
user-invocable: false
---
このプロジェクトのコーディング規約です。

- 関数は単一責任を保つ
- 早期 return を優先し、ネストを浅く保つ
- 変数名はドメインに即した具体名を使う
- テストはまず正常系から書き、次に境界、最後に異常系
- コミットメッセージは Conventional Commits に従う

詳細は .claude/rules/ 配下のドメイン別ファイルを参照してください。
```

これも user-invocable は false です。Sub-agent から参照されることを前提にしています。

レシピC: PR 説明テンプレート（ユーザー呼び出し）

```yaml
---
name: pr-description
description: 直近のコミットから PR 説明を生成する
allowed-tools: Bash(git log *) Bash(git diff *)
---

直近のコミットから PR 説明を生成します。

- コミット一覧: !`git log main..HEAD --oneline`
- 差分の要約: !`git diff main..HEAD --stat`

次のテンプレートに沿って書いてください。

## 概要
何をするための変更か、二行で。

## 変更点
箇条書きで。

## 影響範囲
このプロジェクトのどこに影響するか。

## テスト
どう確認したか。
```

ユーザーが `/pr-description` を打って呼び出します。出力をそのまま PR 作成画面にコピペできる形式にしておくことが重要です。

レシピD: ログ分析（fork で隔離）

```yaml
---
name: analyze-logs
description: 指定ファイルのログを読み、エラーパターンを抽出する
context: fork
agent: Explore
argument-hint: [ログファイルパス]
allowed-tools: Read Bash(grep *) Bash(wc *)
---
ログファイル $ARGUMENTS を分析します。

1. 全体の行数と期間を把握
2. エラーレベルごとに件数を集計
3. 頻出するエラーメッセージを上位 5 件抽出
4. 時間帯別の分布を確認
5. 異常パターンを発見したら、コードのどこから出ているかを推測

ファイルが大きいので、必要な部分だけを段階的に読み込んでください。全文を一度に読まないこと。
```

`context: fork` と `agent: Explore` を指定しています。ログ分析は読み込み量が多くなりがちなので、メインのコンテキストを汚さないための隔離が効きます。

レシピE: 型安全性チェック（fork で隔離）

```yaml
---
name: type-safety-audit
description: 指定モジュールの型安全性を点検する
context: fork
agent: Explore
argument-hint: [モジュールパス]
allowed-tools: Read Glob Grep
---
モジュール $ARGUMENTS の型安全性を点検します。

次のパターンを探してください。
- any や unknown の使用
- @ts-ignore や @ts-expect-error の使用
- 型アサーション (as) の濫用
- 暗黙的な any
- ジェネリクスの境界が緩い箇所

検出した箇所をファイルパスと行番号で挙げ、それぞれに「なぜ問題か」「どう直すか」を添えてください。
```

これも fork で動かすことで、メインのセッションが型情報の海に飲まれるのを防ぎます。

### 4-4　Sub-agentのレシピ集

Sub-agent は Command や Skill よりも「資産」としての性格が強くなります。三つだけ、フロントマター込みで具体例を出します。

レシピX: コードレビュー Sub-agent

```yaml
---
name: code-reviewer
description: 書かれたばかりのコードを、品質・セキュリティ・ベストプラクティスの観点でレビューする。実装が一段落したタイミングで使う。
tools: Read, Glob, Grep
model: haiku
maxTurns: 10
permissionMode: default
color: blue
---
あなたはコードレビュアーです。

渡された変更を読み、次の四点で具体的かつ実行可能なフィードバックを返します。

1. ロジックの正当性
2. エッジケースの取りこぼし
3. テストの過不足
4. 既存パターンとの整合

各指摘はファイルパスと行番号を添え、深刻度を「致命的・要改善・提案」の三段階で示してください。レビューだけを行い、コードの修正には進まないこと。
```

`tools` で読み取り系だけを許可、`model: haiku` で軽量化、`maxTurns: 10` で暴走防止、`color: blue` で視認性確保。最低限の安全装置を全部入れた構成です。

レシピY: デプロイ管理 Sub-agent

```yaml
---
name: deploy-manager
description: ステージング環境へのデプロイを管理する。デプロイ実行と動作確認を行う。本番には触らない。
tools: Bash, Read
model: sonnet
maxTurns: 20
permissionMode: acceptEdits
isolation: worktree
memory: project
skills:
  - deployment-checklist
  - coding-conventions
mcpServers:
  - slack
color: orange
---
あなたはデプロイ管理者です。

ステージング環境へのデプロイを実行し、動作確認まで行います。本番環境には絶対に触れません。

デプロイ前に deployment-checklist を必ず確認し、すべてのチェックが通ってから実行してください。デプロイ後はステージング環境にアクセスして主要画面の動作を確認します。

完了したら、Slack の #deploy チャンネルに結果を投稿してください。
```

`isolation: worktree` で隔離コピー、`memory: project` で過去のデプロイ知識を蓄積、`skills` で必要な知識をプリロード、`mcpServers` で Slack 通知の権限だけを限定的に与える。実用的な Sub-agent はこのぐらいフロントマターが厚くなります。

レシピZ: ドキュメント生成 Sub-agent（バックグラウンド実行）

```yaml
---
name: doc-generator
description: 指定したモジュールの API ドキュメントを生成する。長時間動くのでバックグラウンドで走る。
tools: Read, Glob, Grep, Write
model: sonnet
maxTurns: 50
background: true
color: green
---
あなたはドキュメント生成者です。

指定されたモジュールを隅々まで読み、公開されている関数とクラスについて、用途、引数、戻り値、副作用、使用例を含む API ドキュメントを生成します。

実装の細部ではなく、利用者目線の情報に重点を置いてください。出力は docs/ ディレクトリ配下に Markdown で書き出します。
```

`background: true` でバックグラウンド実行に固定。長時間動くタスクは、ユーザーが他の作業をしているあいだに進めてもらえるのが理想です。

これら三つに共通するのは、フロントマターが Sub-agent の性格を決定的に決めている点です。本文（システムプロンプト）は短くて構いません。フロントマターで責務と権限を絞り、本文では「何をするか」と「何をしないか」だけを伝える、というのが書きやすい構成です。

### 4-5　チームで使うときの設計

ここまで個人で使う前提で書いてきましたが、Claude Code はチームで共有する道具としての側面も強くあります。

`.claude/` ディレクトリは、リポジトリにコミットすべきものです。CLAUDE.md、`.claude/commands/`、`.claude/skills/`、`.claude/agents/`、`.claude/settings.json` のすべてが、チーム全員で共有する資産になります。誰かが新しい Skill を書いて push すれば、翌日には全員のローカルでその Skill が使えます。

個人設定との分離も意識します。`~/.claude/` 配下は個人のグローバル設定で、ここに置いた Skill や Sub-agent はすべてのプロジェクトで使えます。チーム共通の規約や Skill はプロジェクトの `.claude/` に、個人の好みや実験中のものはホームの `~/.claude/` に、と置き場所を分けると混乱しません。

CLAUDE.md には「新しいメンバーが最初に読むもの」としての役割を意識して書きます。チームに新しい人が入ってきたとき、CLAUDE.md だけ読んで Claude Code を起動すれば、その日のうちにビルドが通り、テストが走り、PR が作れる。これが達成されているなら、その CLAUDE.md は良い CLAUDE.md です。逆に、新人が入るたびに口頭で「あ、これも書いておけばよかった」となるなら、その都度書き足していきます。CLAUDE.md は固定の文書ではなく、観察によって改善されるものです。

Sub-agent の memory スコープも、チーム運用では考えどころです。`memory: project` を選ぶと、その Sub-agent の蓄積知識が `.claude/agent-memory/<agent-name>/` 配下に保存されてリポジトリにコミットできます。チーム全員が、その Sub-agent の過去の判断を共有できます。逆に、個人的な作業ログを共有したくない Sub-agent は `memory: local` にして、コミット対象から外します。`memory: user` は、ユーザーの全プロジェクトで共有されるグローバル記憶です。三つを使い分けるという視点を持っておくと、運用が破綻しません。

### 4-6　よくある失敗パターンと対処

実際に運用していると、いくつか同じ失敗を踏みがちです。代表的なものを挙げます。

失敗1: CLAUDE.md が長くなりすぎて Claude が無視するようになる

書きたい気持ちは分かりますが、200 行を超えたら一度立ち止まります。何がドメイン共通で、何が局所的か。局所的なものは `.claude/rules/` に分割します。`paths` フロントマターを使えば、特定ディレクトリで作業しているときだけ読み込ませることもできます。CLAUDE.md は定数ではなく、定期的に剪定するものです。

失敗2: Command を作ったが使わなくなる

Command は自分で `/` を叩かないと使われません。たとえ description が完璧でも、習慣化していなければ起動されません。「これは Command で済むな」と思った瞬間に `/` を叩く癖をつけると、習慣が育ちます。それでも使われない Command があるなら、その Command は本当に必要だったのか、もう一度疑ったほうがいいかもしれません。

失敗3: Sub-agent が途中で止まる

`maxTurns` が低すぎる、`permissionMode` が `default` のままで権限ダイアログ待ちになっている、というのが二大原因です。前者は数値を上げます。後者は、Sub-agent の責務に合わせて `acceptEdits` などに切り替えます。読み取り専用の Sub-agent なら `default` のままでも問題ないことが多く、書き込みを伴う Sub-agent なら `acceptEdits` か、信頼できるなら `bypassPermissions` を検討します。

失敗4: Skill の description があいまいで Claude が呼んでこない

これは本当によくあります。「コードレビューを行う」では条件として弱いです。「Pull Request の差分が話題に出たとき、セキュリティと品質の観点でレビューを行う」のように、いつ呼ぶかが読み取れる文章にします。description の最初の 250 文字に、最も使ってほしい条件を入れることも忘れないでください。

失敗5: Hook が遅くてターンの応答性が落ちる

Hook はターン中に同期的に走ります。重い処理を Hook の中に書くと、Claude の応答待ちが長くなります。重い処理は、Hook が起動するバックグラウンドジョブの方に押し出します。Hook 自身は数百ミリ秒で終わるのが理想です。

### 4-7　日々の運用サイクル

最後に、日々の運用のなかで意識しておくと長く使い続けられる習慣をいくつか挙げます。

Claude Code は頻繁にアップデートされます。アップデートのたびに changelog を読む癖をつけてください。新しい組み込み Sub-agent や Hook イベントが追加されることがあります。気づかずに使い続けると、新しい仕組みのメリットを取り逃します。

「Claude に何度も同じ説明をした」と感じた瞬間が、Skill を作るタイミングです。逆に言えば、二回までは Skill にしないでいいのです。三回目で初めて、その説明には永続化する価値があります。

Command は、一日に同じことを二回やったときが作り時です。一回目は試行錯誤、二回目で「これ毎回やるな」と気づく、三回目には Command 化されている。このリズムが回り始めると、Claude Code が手放せない道具になります。

`/rename` でセッションに名前をつけ、`/resume` で過去のセッションを呼び戻す習慣も、長期戦で効きます。複数のタスクを並行して進めるとき、セッションが整理されていれば、午後に午前の続きをすぐに再開できます。

コミットは一時間に一回を目安にします。Claude Code は自律的に大きな変更を入れることがあるので、こまめにコミットしておかないと、いざというとき差し戻しの単位が大きくなります。Claude にコミットを任せず、自分でコミットの単位を決めるほうが、後で振り返りやすくなります。

そして、Hook で計測した使用状況を月に一度ぐらい見直します。使われていない Skill は description を直すか、削除します。よく使われている Skill には新しい引数や条件を足します。Claude Code の運用は、書いて終わりではなく、観察して改善する小さなループです。このループが回り始めると、自分のプロジェクトに「Claude Code が住んでいる」状態になります。

## あとがき

ここまで読んで、何か一つでも自分のプロジェクトに残っているなら、この記事は完成しています。逆に、読み終えても何も書いていないなら、まだ完成していません。完成させるのは読者の側です。

冒頭で書いた通り、矢印は常にあなたに向けています。筆者の経験は「なぜこの順番で説明するか」の根拠としてだけ下敷きにしました。CLAUDE.md の話を最初に置いたのは、設計思想の手触りがないまま Skill や Sub-agent の話をすると機能の羅列に見えてしまうからです。Hook を後半に置いたのは、Hook は道具立てが揃ってから初めて活きるからです。レシピ集を実物のコードで並べたのは、コピーして手を入れることが、最初の一歩としてはいちばん速いからです。

Claude Code は、Claude を呼び出して使う道具ではなく、Claude が動く環境を設計する道具です。設計の主体はあなたです。CLAUDE.md は環境設定で、Skill は知識の貯水池で、Sub-agent は責務分担で、Hook は信頼の多層化です。これらを少しずつ自分のプロジェクトに合わせて積み上げていくと、ある日「これがないと前みたいには働けない」という感覚に変わります。

そのとき、Claude Code はあなたのプロジェクトに根付いています。
